# design plugin — 요구사항 문서

> plugin-dev + skill-creator가 이 문서를 읽고 구현에 즉시 착수할 수 있도록 작성된 요구사항 및 배경 문서. 컴포넌트 구조 결정(Commands/Skills/Agents 선택 등)과 산출물 형식 결정은 plugin-dev의 역할이므로 이 문서에서 다루지 않는다.

## 1. 이 플러그인이 존재하는 이유

Claude Code 에이전트를 설계할 때, 실제 병목은 구현이 아니라 **설계 의사결정**이다.

- 어떤 발화가 나올지 모른다
- 발화에서 요구사항을 어떻게 도출하는지 모른다
- 요구사항을 Command로 만들어야 하는지 Skill로 만들어야 하는지 모른다
- 설계가 끝났다고 생각했는데 실제 발화를 따라가면 연결이 끊어진다

plugin-dev와 skill-creator는 이 문제를 해결하지 않는다. 두 플러그인은 **"무엇을 만들지 이미 아는 사람"** 을 위한 도구다.

이 플러그인은 **"무엇을 만들어야 하는지 모르는 상태"** 에서 시작하여, plugin-dev가 Discovery 없이 바로 구현에 들어갈 수 있는 설계 산출물까지 도달하는 설계 과정 전체를 지원한다.

---

## 2. 사용자

에스코어 또는 그 고객사에서 **도메인 버티컬 에이전트를 처음 설계하는 사람.**

- AI 개발자가 아니어도 된다
- 도메인 업무는 잘 안다 (경영정보, 계약검토, 코딩 등)
- Claude Code 플러그인 구조(plugin.json, SKILL.md, commands/)를 어느 정도 이해한다
- 혼자 또는 소규모 팀으로 작업한다

이 사람이 이 플러그인 없이 할 수 없는 것:

- 페르소나의 실제 발화를 체계적으로 도출하는 것
- 발화로부터 요구사항을, 요구사항으로부터 구성요소를 연결하는 것
- 설계 후 "작동하는가"가 아닌 "연결이 끊어진 곳이 있는가"를 검증하는 것

## 3. 핵심 산출물 — 설계 산출물 요구사항

이 플러그인의 모든 작업은 하나의 설계 산출물을 완성하기 위한 과정이다. **산출물의 구체적인 형식(표, YAML, JSON 등)은 plugin-dev가 결정한다.** 이 섹션은 산출물에 반드시 담겨야 하는 정보와 그 정보가 필요한 이유를 정의한다.

### 담겨야 하는 정보

|정보 항목|의미|왜 필요한가|
|---|---|---|
|발화 식별자|발화를 고유하게 참조하는 ID|구멍 발견 시 어느 발화에서 왔는지 추적|
|발화 내용|사용자가 실제로 할 말|설계의 모든 판단이 발화에서 출발함|
|요구사항 식별자|요구사항을 고유하게 참조하는 ID|구멍과 요구사항을 1:1로 연결|
|요구사항 내용|이 발화를 처리하기 위해 필요한 것|구성요소 결정의 근거|
|구성요소명|요구사항을 담당하는 컴포넌트 이름|plugin-dev가 파일을 만들 때 직접 사용|
|계위|Command / Skill / references / MCP / Hook / Agent|어느 레이어에 구현할지 결정|
|소속 플러그인|이 구성요소가 속하는 플러그인|플러그인 간 경계와 의존성 파악|
|구멍 설명|연결이 끊어진 지점과 이유|재설계 시 무엇을 수정해야 하는지 파악|
|구멍 심각도|심각 / 중간 / 경미|재설계 루프 진입 여부 판단|

### 구조적 요구사항

다음 세 가지 관계가 산출물 안에서 추적 가능해야 한다.

**1:N 발화-요구사항 관계** 발화 하나가 여러 요구사항을 만들 수 있다. "3월 사업부별 매출 보여줘" 한 발화에서 SQL 변환, 스키마 참조, DML 차단이 동시에 요구사항으로 나온다. 어떤 형식을 쓰더라도 이 관계를 표현할 수 있어야 한다.

**1:N 요구사항-구성요소 관계** 요구사항 하나가 여러 구성요소로 구현될 수 있다. "DB에 읽기 전용으로 접근한다"는 요구사항이 MCP 서버와 Hook으로 나뉠 수 있다.

**발화-구멍 역추적** 구멍이 어느 발화, 어느 요구사항에서 비롯되었는지 역추적 가능해야 한다. 구멍만 보고 무엇을 수정해야 하는지 알 수 있어야 한다.

### 단계적 완성 요구사항

산출물은 작업이 진행되면서 점진적으로 완성된다. 각 작업 단계가 끝날 때마다 특정 정보 항목이 추가되고, 반복(Walk-through 실패 → 작업 D 복귀) 시 기존 항목이 수정된다. 선택하는 형식은 이 점진적 추가와 수정을 자연스럽게 수용할 수 있어야 한다.

> **[설계 결정 2026-03-20] 산출물 형식:** 원문은 "형식은 plugin-dev가 결정한다"고 위임했다. 구현 과정에서 **Markdown 반복 행 구조**로 결정했다. 발화 하나에 여러 요구사항이 대응될 때 발화 ID를 동일한 값으로 반복하는 행 구조로 1:N 관계를 표현한다. 선택 근거: Claude가 텍스트로 생성·수정하기 쉽고, 점진적 열 추가(v1→v2→…)가 자연스러우며, plugin-dev와 사람이 모두 읽을 수 있는 형식이다. 상세 형식은 `skills/design-session/references/artifact-schema.md` 참조.

### plugin-dev 인계 조건

작업 G 완료 시 이 산출물이 plugin-dev에 전달된다. 이 산출물은 plugin-dev의 Phase 1~2가 다루는 정보(목적, 사용자, 구성요소 목록, 계위)를 포함하도록 설계되었다. plugin-dev는 수신한 산출물을 검토하고 어느 Phase부터 시작할지 스스로 판단한다.

## 4. 지원해야 하는 작업

### 작업 A — 페르소나 정의

**목적:** 발화 가설과 요구사항 판단의 기준점을 만든다.

페르소나가 없으면 발화가 추측이 된다. "사용자가 이런 말을 할 것 같다"가 아니라 "이 사람의 업무에서 이 상황이 생기면 이렇게 말한다"가 되어야 한다.

페르소나 정의에서 반드시 드러나야 하는 것:

- 실제 업무 방식 — 어떤 시스템을 어떤 순서로 쓰는가
- 기술·역량 문제가 아닌 구조적 제약
- 진짜 페인포인트 — 표면 니즈가 아닌 "왜" 한 번 더 파고든 결과
- AI 활용 수준 — 현재 어떤 도구를 어떻게 쓰는가
- 실제 발화 스타일 — 모호한 표현, 맥락 의존 표현 포함

산출물: 페르소나 카드 (저장 대상, 이후 작업의 판단 기준으로 유지)

---

### 작업 B — 발화 가설 도출

**목적:** 설계 전 Walk-through에 쓸 초기 발화 세트를 준비한다.

발화가 나쁘면 요구사항도 나쁘다. 발화는 반드시 **업무 시나리오 다양성**으로 선정해야 한다. 구성요소를 먼저 생각하고 발화를 역산하면 안 된다.

커버해야 하는 발화 유형:

|유형|설명|이유|
|---|---|---|
|일상 업무|페르소나가 매일 하는 요청|핵심 경로 커버|
|임기응변|회의 중 즉석 질문, 모호한 표현|실제 사용 패턴 반영|
|크로스 시스템|여러 데이터 소스를 합쳐야 하는 경우|구성요소 간 연결 필요성 도출|
|치명적 경로|잘못 작동했을 때 피해가 가장 큰 경우|안전장치(Hook) 요구사항 도출|

산출물: 설계 산출물 v1 — 발화 정보만 채워진 상태

---

### 작업 C — 발화 → 요구사항 추출 (설계 전 Walk-through)

**목적:** 각 발화를 처리하는 데 필요한 작업들을 빠짐없이 도출한다.

에이전트 시점에서 발화를 단계적으로 따라가면서 각 단계에서 필요한 것을 요구사항으로 변환한다.

이 시점에서 구성요소 계위(Skill인지 MCP인지)는 결정하지 않는다. "무엇이 필요한가"만 도출한다.

Walk-through 진행 방식:

```
[발화]: "3월 사업부별 매출 보여줘"

단계 1. 질문 의도 파악
    필요한 것: 자연어를 구조화된 데이터 요청으로 해석
    → 요구사항: "자연어를 SQL로 변환할 수 있어야 한다"

단계 2. 데이터 위치 파악
    필요한 것: 어느 테이블에 있는지 탐색
    → 요구사항: "DB 스키마를 참조할 수 있어야 한다"

단계 3. 조회 실행
    필요한 것: DB에 쿼리를 날리고 결과를 받아옴
    → 요구사항: "DB에 읽기 전용으로 접근해야 한다"
    → 요구사항: "DML은 항상 차단해야 한다"
```

산출물: 설계 산출물 v2 — 요구사항 정보 추가된 상태

---

### 작업 D — 요구사항 → 산출물 통합

**목적:** plugin-dev의 계위 판단 결과를 받아 설계 산출물에 통합한다.

계위를 판단하는 것은 plugin-dev의 역할이다. 이 작업에서 이 플러그인이 담당하는 것은:

- 요구사항 목록을 plugin-dev가 계위 판단을 수행할 수 있는 형식으로 준비
- plugin-dev가 돌려준 계위 결정(구성요소명, 계위, 소속 플러그인)을 산출물에 기록
- 판단 결과를 저장하고 다음 작업(E, F)의 입력으로 관리

산출물: 설계 산출물 v3 — 구성요소·계위·소속 플러그인 정보 추가된 상태

> **[설계 결정 2026-03-20] ②④⑤ 사후 검증 담당 추가:** 원문은 계위 판단 결과 통합만 다뤘다. 구현 과정에서 구성요소 구현 이후 필요한 ②입출력 불일치, ④도구 미연결, ⑤실패 경로 검증의 담당 구성요소가 명시되지 않았음을 발견했다. requirements-sync의 트리거 B가 plugin-dev 구성요소 구현 완료 시점에 이 검증을 담당하도록 확정했다.

### 작업 E — 발화 목록 보완

**목적:** 구성요소 커버리지를 확인하고 발동 경로가 없는 구성요소를 발견한다.

구성요소가 있는데 그것을 트리거하는 발화가 없으면 사실상 죽은 구성요소다. 이 단계는 커버리지를 높이기 위한 것이 아니라, **"이 발화가 오면 이 구성요소가 실행된다"는 연결 경로를 전수 확인**하기 위한 것이다.

확인 항목:

- Command 진입점을 트리거하는 발화가 있는가
- Skill 체이닝 경로가 발화에 있는가
- 플러그인 간 연결 경로가 발화에 있는가
- 실패·예외 경로가 발화에 있는가

발동 경로 없는 구성요소 발견 시 → 해당 경로를 발동시킬 발화를 산출물에 추가

산출물: 설계 산출물 v4 — 발화 항목 추가된 상태

### 작업 F — 연결 구멍 검증 (설계 후 Walk-through)

**목적:** 완성된 구성 목록으로 각 발화를 처리할 수 있는지 검증하고 구멍을 기록한다.

구멍을 발견하는 것이 목적이다. 발견 즉시 해결하지 않는다.

검증 기준 — 다음 5가지로만 판정한다:

|번호|기준|설명|
|---|---|---|
|①|연결 없음|필요한 구성요소가 산출물에 없다|
|②|입출력 불일치|앞 단계 출력 형식 ≠ 이 단계 입력 형식|
|③|트리거 미정의|이 Skill/Command가 언제 발동되는지 정의 없음|
|④|도구 미연결|이 Skill의 allowed-tools에 필요한 MCP 없음|
|⑤|실패 경로 미처리|이 단계 실패 시 에이전트 행동 미정의|

구멍 심각도 기준:

|심각도|의미|처리|
|---|---|---|
|심각|이 발화가 작동하지 않음|작업 D로 복귀, 관련 항목 재작성|
|중간|작동하지만 문제가 생김|해당 발화만 재추적|
|경미|나중에 개선 가능|Could 항목으로 이관|

완료 기준: 모든 발화 추적이 끝나고, 각 발화가 "사용자에게 반환" 또는 "구멍 기록"으로 끝났을 때.

> **[설계 결정 2026-03-20] 작업 F 범위 수정:** 원문의 구멍 5기준(①~⑤) 중 ②입출력 불일치, ④도구 미연결, ⑤실패 경로 미처리는 설계 시점에 판정 불가능하다(구성요소 구현 명세가 없으므로). 따라서 작업 F는 **①연결 없음, ③트리거 미정의만 적용**하고, ②④⑤는 구현 이후 requirements-sync 트리거 B가 담당한다. 이 변경은 구현 시작 전 사용자 확인을 받았다.

> **[설계 결정 2026-03-20] 재진입 절차 추가:** 원문은 "심각 구멍 발견 시 작업 D로 복귀"만 명시했으나 실제 재진입 방법이 정의되지 않았다. 구현에서는 requirements-sync가 핵심 경로 변경 시 ① artifact-final.md에 변경 반영, ② artifact-v4.md로 복사(gap-analyzer 입력), ③ state.md의 current_step을 F 또는 D로 설정한 후, 사용자가 "설계 이어서 해줘"로 design-session을 재개하는 방식으로 정의했다.

### 작업 H — 설계 상태 및 산출물 관리

**목적:** 설계 과정이 반복되는 상황에서 현재 상태를 추적하고, 산출물을 버전 관리한다.

설계는 선형이 아니다. Walk-through에서 심각 구멍이 발견되면 작업 D로 돌아가고, 그 결과가 산출물에 반영되어야 한다. 상태 추적이 없으면 설계가 어디까지 왔는지, 무엇이 바뀌었는지 알 수 없다.

#### 관리해야 하는 상태

```
현재 작업 단계 (A~G 중 어디인가)
페르소나 카드 (작업 A 완료 시 저장)
설계 산출물 버전 이력
  v1: 발화 정보만
  v2: 요구사항 추가
  v3: 구성요소·계위 추가
  v4: 발화 보완
  v5: 구멍 추가
  확정본
구멍 목록 (심각/중간/경미 분류)
구멍 해소 이력 (어떤 구멍이 어떻게, 언제 해소되었는가)
반복 이력 (몇 번째 루프인가, 어느 작업으로 복귀했는가)
```

#### 산출물 업데이트 정책 — 하이브리드

산출물 업데이트는 **자동 저장 시점**과 **사용자 트리거 시점**으로 구분한다.

**자동 저장 시점 (작업 경계):**

각 작업(A~G)이 완료되는 시점은 상태 전이의 명확한 경계이므로 자동으로 저장한다.

```
작업 A 완료 → 페르소나 카드 저장
작업 B 완료 → 산출물 v1 저장
작업 C 완료 → 산출물 v2 저장
작업 D 완료 → 산출물 v3 저장
작업 E 완료 → 산출물 v4 저장
작업 F 완료 → 산출물 v5 저장
심각 구멍 발견으로 작업 D 복귀 → 복귀 이력 기록 후 v3부터 재작성
작업 G 완료 → 확정본 저장
```

**사용자 트리거 시점 (작업 내부):**

작업 내부 진행 중(예: plugin-dev와 요구사항을 하나씩 검토하는 중)에는 중간 상태가 빈번하므로 자동 저장하지 않는다. 사용자가 "지금까지 내용 저장해줘" 또는 "현재 산출물 보여줘"라고 요청할 때 반영한다.

**이 구분이 필요한 이유:** 작업 내부의 잦은 자동 저장은 산출물에 불완전한 중간 상태를 남긴다. 여러 플러그인이 동시에 작동하는 상황에서 산출물이 계속 바뀌면 다음 플러그인(plugin-dev 등)이 어느 버전을 기준으로 삼아야 하는지 모호해진다. 작업 완료 경계에서만 자동 저장함으로써 산출물의 각 버전이 항상 완결된 상태임을 보장한다.

### 작업 H+ — 상태 수동 복원 (state-reconcile)

> **[설계 결정 2026-03-20] 신규 추가:** 원문의 작업 H는 자동 저장과 상태 추적을 다뤘으나, 여러 세션에 걸쳐 작업하다 state.md와 실제 파일이 불일치하는 경우를 복원하는 수단이 없었다. 사용자 요청(Q3)에 따라 `/agent-designer:sync` 명령으로 호출하는 **state-reconcile** Skill을 추가했다. 이 Skill은 모든 artifact 파일을 읽어 실제 상태를 재도출하고 state.md를 수정한다. 설계 흐름에 영향을 주는 새 작업이 아닌, 작업 H의 안전장치로 분류한다.

## 부록 A — plugin-dev를 위한 개념 사전

plugin-dev가 이 플러그인과 협업하기 위해 알아야 하는 개념.

### 설계 산출물

발화 → 요구사항 → 구성요소가 모두 연결된 단일 문서. 발화 하나가 여러 요구사항을, 요구사항 하나가 여러 구성요소를 가질 수 있다.

이 산출물은 plugin-dev의 Phase 1(Discovery)과 Phase 2(Component Planning)가 다루는 정보를 포함하도록 설계되었다. plugin-dev는 수신한 산출물이 Phase 3(Detailed Design) 진입에 충분한지 확인한 후 시작 지점을 결정한다.

계위 항목의 값 종류:

|값|의미|
|---|---|
|Command|사용자 /slash 진입점. Skill을 오케스트레이션하는 얇은 래퍼|
|Skill|재사용 가능한 전문 기능 단위|
|references|특정 Skill과 항상 함께 쓰이는 보조 참조 파일|
|MCP|외부 시스템 접근 실행 도구|
|Hook|이벤트 기반 자동 실행 (LLM 판단 없이)|
|Agent|멀티스텝 자율 실행 주체|

### Walk-through

발화를 에이전트 시점에서 단계적으로 따라가는 검증 방식.

- **설계 전 Walk-through**: 발화 + 업무 프로세스 → 요구사항 도출
- **설계 후 Walk-through**: 발화 + 구성 목록 → 구멍 발견

### 구멍 5기준

설계 후 Walk-through에서 사용하는 판정 기준:

1. 연결 없음 — 필요한 구성요소가 산출물에 없다
2. 입출력 불일치 — 앞 단계 출력 형식 ≠ 이 단계 입력 형식
3. 트리거 미정의 — 이 Skill/Command가 언제 발동되는지 정의 없음
4. 도구 미연결 — 이 Skill의 allowed-tools에 필요한 MCP 없음
5. 실패 경로 미처리 — 이 단계 실패 시 에이전트 행동 미정의

## 부록 B — skill-creator를 위한 개념 사전

skill-creator가 이 플러그인과 협업하기 위해 알아야 하는 개념.

### 발화 가설 목록 = eval 발화의 소스

이 플러그인이 완성한 설계 산출물의 발화 항목은 skill-creator의 eval 발화 세트 작성에 직접 활용된다.

발화 유형별 eval 분류:

|발화 유형|should_trigger|이유|
|---|---|---|
|일상 업무 발화|true|반드시 해당 Skill이 발동되어야 하는 경로|
|임기응변·모호한 발화|true|명시적 요청 없어도 맥락이면 발동되어야 함|
|치명적 경로 발화|true|Hook 포함 전체 경로 발동 필요|
|범위 밖 발화 (산출물에 없는 것)|false|이 Skill이 발동되면 안 되는 경우|

### 발화-Skill 연결 관계

설계 산출물의 "발화 ↔ 구성요소(Skill)" 연결이 undertrigger 판정의 기준선이다. "이 발화가 왔을 때 이 Skill이 발동되지 않았다"가 곧 undertrigger.

undertrigger는 description이 소극적으로 작성된 경우에 주로 발생한다. description에 구체적인 발화 예시와 "명시적으로 요청하지 않아도 맥락이면 사용한다"는 표현이 포함되어야 발동 정확도가 높아진다.
