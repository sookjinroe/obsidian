# 에이전트 설계 프로세스

> 에이전트를 처음부터 설계할 때 따라가는 순서와 각 단계의 산출물. 구성요소 정의·판단 트리는 agent_component_reference.md 참조.

---

## 전체 루프

```
① 페르소나 정의
   산출물: 페르소나 카드
        ↓
② 발화 가설 작성               ← 업무 시나리오 기준
   산출물: 설계 매트릭스 (발화 컬럼만 채움)
        ↓
③ 설계 전 Walk-through
   산출물: 설계 매트릭스 (요구사항 컬럼 추가)
        ↓
④ 계위 판단 트리 적용
   산출물: 설계 매트릭스 (구성요소·계위·플러그인 컬럼 추가)
        ↓
⑤ 발화 목록 보완               ← 구성요소 커버리지 기준으로 발화 추가
   산출물: 설계 매트릭스 (발화 행 추가)
        ↓
⑥ 설계 후 Walk-through ←────────────────┐
   산출물: 설계 매트릭스 (구멍 컬럼 추가) │
        ↓                                │
   심각 구멍 있음? ──Yes──→ 구성·요구사항 수정 ─┘
        │
       No
        ↓
⑦ 구성 확정
   산출물: 설계 매트릭스 (확정본)
        ↓
⑧ CF 설계서 작성
   산출물: CF 설계서 (SKILL.md, commands/ 등)
```

## 핵심 산출물: 에이전트 설계 매트릭스

발화 → 요구사항 → 구성요소가 모두 연결된 단일 문서. 단계가 진행될수록 컬럼이 채워진다.

|발화ID|발화|요구사항ID|요구사항|구성요소|계위|플러그인|구멍|심각도|
|---|---|---|---|---|---|---|---|---|
|U01|"3월 사업부별 매출 보여줘"|R01|자연어 → SQL 변환|text-to-sql|Skill|경영정보|-|-|
|U01|(동일)|R01|자연어 → SQL 변환|sales-db-mcp|MCP|경영정보|-|-|
|U01|(동일)|R03|DML 항상 차단|DML차단 Hook|Hook|경영정보|-|-|
|U02|"매출이랑 생산량 같이 봐줘"|R02|복수 DB 접근|multi-db 라우팅|-|-|❌ 미정의|심각|

**하나의 발화가 여러 행을 가질 수 있다.** 발화 하나가 여러 요구사항을 만들고, 요구사항 하나가 여러 구성요소를 만들기 때문.

**이 형태의 장점:**

- 발화가 바뀌면 연결된 요구사항·구성요소가 같은 행에 있어 즉시 파악
- 요구사항 추가 시 어떤 발화에서 나왔는지 추적 가능
- 구멍이 어떤 발화·요구사항에서 왔는지 한 줄로 확인
- 별도 문서 간 동기화 문제 없음

## Step 1. 페르소나 정의

### 목적

에이전트가 누구를 위한 것인지 명확히 하고, 발화 가설과 요구사항의 기준점을 만든다.

### 방법

- 인터뷰 또는 도메인 조사로 실제 사용자 파악
- 실제 업무 방식 반영 (AI 활용 수준 포함)
- "못하는 사람도 할 수 있게"가 아닌 "잘하는 사람이 구조적 장벽을 넘게"

### 산출물: 페르소나 카드

```yaml
name: [이름]
role: [직급/팀]
company: [회사 유형 및 규모]

ai_proficiency: >
  현재 AI 활용 수준.
  어떤 도구를 어떻게 쓰는가.

workflow: >
  현재 업무 방식 (As-Is).
  어떤 시스템을 어떤 순서로 쓰는가.

structural_barriers:
  - 기술·역량 문제가 아닌 구조적 제약 목록

pain_points:
  - 진짜 페인포인트 (표면 니즈가 아닌 진짜 니즈)
  - "왜" 한 번 더 파고든 결과

utterance_patterns:
  - 실제 발화 스타일 예시
  - 모호한 표현, 상황 의존 표현 포함
```

## Step 2. 발화 가설 작성

### 목적

설계 전 Walk-through에 쓸 초기 발화 세트를 준비한다. 이 시점에는 구성요소가 아직 없으므로 **업무 시나리오 다양성**으로 선정한다.

### 방법 — 업무 시나리오 기준 (설계 전)

```
필수 커버 항목
├── 자주 쓰이는 경로          페르소나의 일상 업무
├── 임기응변·모호한 입력      회의 중 즉석 질문
├── 크로스 시스템 경로        여러 데이터 소스를 합쳐야 하는 경우
└── 실패하면 치명적인 경로    잘못 작동했을 때 가장 큰 피해
```

> 구성요소 커버리지 기준(MCP 라우팅, Hook 동작 등)은 Step 5에서 적용. 지금은 실제 사용자가 할 법한 말인지가 기준.

## Step 3. 설계 전 Walk-through

### 목적

각 발화를 처리하는 데 필요한 작업들을 도출한다. 계위(Skill인지 MCP인지 Command인지)는 아직 결정하지 않는다.

### 방법

```
[발화 U01]: "3월 사업부별 매출 보여줘"

단계 1. 질문 의도 파악
    필요한 작업: 자연어를 구조화된 데이터 요청으로 해석
    → 요구사항: "자연어를 SQL로 변환할 수 있어야 한다" (R01)

단계 2. 데이터 위치 파악
    필요한 작업: 매출 데이터가 어느 테이블에 있는지 탐색
    → 요구사항: "DB 스키마를 참조할 수 있어야 한다" (R02)

단계 3. 조회 실행
    필요한 작업: DB에 쿼리를 날리고 결과를 받아옴
    → 요구사항: "DB에 읽기 전용으로 접근해야 한다" (R03)
    → 요구사항: "DML은 항상 차단해야 한다" (R04)

단계 4. 결과 반환
    필요한 작업: 출처와 함께 결과 제시
    → 요구사항: "결과에 출처를 표시해야 한다" (R05)
```

|---|---|---|---|---|---|---|---|---|

## Step 4. 계위 판단 트리 적용

### 목적

요구사항 목록의 각 항목에 구성요소·계위·플러그인을 배정한다.

### 방법

agent_component_reference.md의 판단 트리를 각 요구사항에 적용.

**Command 배정 시 추가 확인:**

- 사용자가 /슬래시로 직접 호출하는가?
- 여러 Skill을 순서대로 조합해야 하는가?
- 해당하면 Command로 배정하고, 어떤 Skill을 어떤 순서로 로드하는지 명시

**references/ 배정 시 추가 확인:**

- Skill 기준(독립 트리거, 재사용, 복잡성) 셋 다 해당 없는가?
- 특정 Skill과 항상 함께 쓰이는 보조 지식인가?
- 해당하면 어느 Skill의 references/ 파일인지 명시

### 산출물

설계 매트릭스의 구성요소·계위·플러그인 컬럼을 채운다.

|발화ID|발화|요구사항ID|요구사항|구성요소|계위|플러그인|구멍|심각도|
|---|---|---|---|---|---|---|---|---|
|U01|"3월 사업부별 매출 보여줘"|R01|자연어 → SQL 변환|text-to-sql|Skill|경영정보|||
|U01|(동일)|R02|DB 스키마 참조|text-to-sql/references/schema.md|references|경영정보|||
|U01|(동일)|R03|DB 읽기 전용 접근|sales-db-mcp|MCP|경영정보|||
|U01|(동일)|R04|DML 항상 차단|DML차단|Hook|경영정보|||

## Step 5. 발화 목록 보완

### 목적

Step 4 결과로 나온 구성요소 목록을 보고, 아직 커버되지 않은 구성요소 경로가 있으면 발화를 추가한다.

### 방법

```
구성요소 목록 확인
    ↓
각 구성요소가 어떤 발화에서 발동되는가 확인
    ├── 커버됨    → 유지
    └── 커버 안 됨 → 해당 경로를 발동시킬 발화 추가
```

확인 항목:

```
├── Command 진입점 발화가 있는가?
├── Skill 체이닝 경로가 발화에 있는가?
├── 플러그인 간 연결 경로가 발화에 있는가?
└── 실패·예외 경로가 발화에 있는가?
```

## Step 6. 설계 후 Walk-through

### 목적

구성 목록 초안으로 각 발화를 처리할 수 있는지 검증한다. 구멍을 발견하는 것이 목적. 발견 즉시 해결하지 않는다.

### 추적 형식

```
[발화 U01]: "3월 사업부별 매출 보여줘"

단계 1. text-to-sql Skill 트리거
    구성에 있는가: ✅
    입력: 자연어 발화
    출력: SQL 초안
    references/schema.md 참조 → ✅ (text-to-sql 내부에 포함됨)
    → 다음 단계로

단계 2. DML차단 Hook (PreToolUse)
    구성에 있는가: ✅
    → 다음 단계로

단계 3. sales-db-mcp 실행
    구성에 있는가: ✅
    입력: SQL
    출력: 결과셋
    → 다음 단계로

단계 4. 출처첨부 Hook (PostToolUse)
    구성에 있는가: ✅
    → 사용자에게 반환 ✅
```

### 구멍 판정 5가지 기준

```
① 연결 없음        필요한 구성요소가 매트릭스에 없다
② 입출력 불일치    앞 단계 출력 형식 ≠ 이 단계 입력 형식
③ 트리거 미정의    이 Skill/Command가 언제 발동되는지 정의 없음
④ 도구 미연결      이 Skill의 allowed-tools에 필요한 MCP 없음
⑤ 실패 경로 미처리 이 단계 실패 시 에이전트 행동 미정의
```

### 구멍 심각도 기준

```
심각   작동 안 됨    → 구성 수정 후 Walk-through 재실행
중간   문제 생김     → 구성 수정 후 해당 발화만 재추적
경미   나중에 가능   → Could 항목으로 이관
```

### 완료 기준

모든 발화 추적이 끝나고 각 발화가 "사용자에게 반환" 또는 "구멍 기록"으로 끝났을 때.

---

## Step 7. 구성 확정

### 조건

설계 후 Walk-through에서 심각 구멍이 없을 때.

### 산출물

설계 매트릭스 최종본. 구멍 처리 내역이 해당 행에 반영된 상태. 변경된 행은 비고에 "GXX로 수정/추가" 표시.

## Walk-through 원칙

### 원칙 1 — 두 시점 구분

```
설계 전 Walk-through              설계 후 Walk-through
인풋: 발화 + 업무 프로세스        인풋: 발화 + 구성 목록 초안
아웃풋: 요구사항 (매트릭스 추가)  아웃풋: 구멍 (매트릭스 추가)
목적: 필요한 것 도출              목적: 연결 구멍 발견
```

### 원칙 2 — 전제 조건

```
설계 전: 페르소나 카드 완료
설계 후: 설계 매트릭스 (발화·요구사항·구성요소 컬럼 채워진 상태)
```

### 원칙 3 — 불변 규칙

```
① 발화 단위로 추적한다
② 에이전트 시점으로 따라간다
③ 각 단계에서 필요한 것을 명시한다 (추정 금지)
④ 완벽한 커버리지를 목표로 하지 않는다
⑤ 구멍은 매트릭스에 기록하고 추적을 계속한다 (즉시 해결 금지)
```

### 원칙 4 — 구멍 판정 5가지 기준

```
① 연결 없음        필요한 구성요소가 매트릭스에 없다
② 입출력 불일치    앞 단계 출력 형식 ≠ 이 단계 입력 형식
③ 트리거 미정의    이 Skill/Command가 언제 발동되는지 정의 없음
④ 도구 미연결      이 Skill의 allowed-tools에 필요한 MCP 없음
⑤ 실패 경로 미처리 이 단계 실패 시 에이전트 행동 미정의
```

## 일반 서비스 디자인과 대응

|단계|일반 서비스|에이전트 설계|차이|
|---|---|---|---|
|사용자 정의|페르소나|페르소나 카드|AI 활용 수준·구조적 장벽 추가|
|요구사항|기능 목록 (별도 문서)|설계 매트릭스 (발화와 통합)|발화가 요구사항의 근거|
|흐름 설계|플로우차트|설계 전 Walk-through|에이전트 시점 추가|
|기능 설계|기능 명세서|설계 매트릭스 (계위 포함)|Command/Skill/references 계위 결정이 동작 결정|
|검증|QA·테스트 케이스|설계 후 Walk-through|"작동하는가" → "연결되어 있는가"|
|구현 명세|화면 정의서|CF 설계서 (SKILL.md, commands/)|화면 대신 LLM 맥락이 구현체|
