  발견한 문제 3가지                                                                                                
                                                                                                                   
  ---                                                                                                                
  문제 1 — gap-analyzer의 작동 전제가 틀렸다                                                                       
                                                                                                                   
  나는 "발화 × 5기준 순회는 기계적이므로 Agent에 위임 가능하다"고 했다. 그런데 구멍 기준을 실제로 들여다보면:    
                                                     
  - ①연결 없음, ③트리거 미정의 → 산출물 안에서 확인 가능. 체크리스트 수준.                                           - ②입출력 불일치, ④도구 미연결, ⑤실패 경로 미처리 → 구성요소가 실제로 무엇을 하는지, 어떤 형식으로 출력하는지를
  알아야 판정 가능.                                                                                                
                                                         
  설계 단계(작업 F)에서는 구성요소의 이름과 계위만 산출물에 있다. 구성요소 명세(입력 형식, 출력 형식, 실패 처리      방식)는 plugin-dev가 구현해야 생긴다. 즉 작업 F 시점에 ②④⑤는 판정 불가다.
                                                                                                                   
  이 문제는 gap-analyzer 설계를 수정하면 해결되는 수준이 아니다. 작업 F의 검증 범위 자체를 재정의해야 한다.          요구사항 문서에 이 제약이 기술되지 않은 것은 문서의 공백일 수 있다.
                                                                                                                   
  의사결정 필요: 작업 F는 설계 시점에 판정 가능한 ①③만 다루고, ②④⑤는 plugin-dev 구현 이후 requirements-sync가        다루도록 범위를 나눌 것인지, 아니면 설계 단계에서 구성요소 명세를 더 구체화하여 ②④⑤도 커버할 것인지.             
                                                                                                                   
  ---
  문제 2 — requirements-sync의 범위가 실제 목적에 못 미친다                                                          
  나는 이 Skill을 "요구사항이 바뀌었다는 발화가 오면 산출물을 업데이트한다"는 반응형 도구로 설계했다. 그런데       
  사용자가 명시한 확장 목적은 "플러그인 빌드 과정에서 변경되는 모든 요구조건에 대한 업데이트"다.                                                                   
  plugin-dev 구현 중 실제로 발생하는 변경은:                                                                       
  - 구성요소 하나가 둘로 분리됨                                                                                    
  - 요구사항이 기술적으로 불가능하다고 판명됨            
  - 구현 중 새 요구사항이 발견됨                                                                                   
  - 두 구성요소가 합쳐짐                                 
                                                     
  각 변경은 산출물의 연결 관계 전체에 영향을 줄 수 있고, 경우에 따라 Walk-through 재실행이 필요하다. 내 현재 설계는
   "변경 → 산출물 반영 → 필요시 안내"라는 수동적 흐름이다. 실제로 필요한 것은 변경 → 영향 추적 → 재검증 범위 자동 
  판단 → 실행이다. 이 차이는 Skill 지시문의 수동성 문제가 아니라 설계된 작동 흐름 자체의 문제다.                                                                            
  ---                                                                                                                문제 3 — Skill을 프로세스 실행기로 착각했다            
                                                                                                                   
  작업 A~G를 하나의 Skill에 담으면서 "상태 파일을 읽어 단계를 판단하고 순서대로 진행한다"고 설계했다. 이것은
  Skill이 무엇인지를 잘못 이해한 것이다.                                                                           
                                                                                                                   
  Skill은 Claude에게 지식과 절차를 제공하는 문서다. Claude가 그 지식을 적용해 사용자와 대화하는 것이지, Skill이    
  상태를 추적하며 단계를 실행하는 것이 아니다. "state.md를 읽어라, 단계를 판단하라, 저장하라"는 지시는 Claude에게    전달되는 것이므로 동작은 하지만, 이 지시들이 실제로 매 응답마다 일관되게 실행된다는 보장이 없다.
                                                                                                                   
  특히:                                                                                                              - 사용자가 중간에 다른 질문을 끼워넣을 때          
  - 대화가 길어져 이전 Skill 내용이 컨텍스트 밖으로 밀릴 때                                                          - 단계 경계가 모호할 때 (사용자가 작업 C 도중 작업 A 내용을 수정하려 할 때)
                                                                                                                   
  이런 상황에서 상태 일관성이 깨질 수 있다. 내 설계는 이 문제를 인식하지 않았다.
                                                                                                                   
  ---                                                
  현재 계획의 수준 평가                                                                                            
                                                         
  잘 된 부분                                             
  - 구성요소 유형 선택의 근거(Skill/Agent/Hook)는 1차 회고 이후 실질적으로 개선됨
  - 1:1 매핑에서 벗어나 트리거 조건과 작동 방향을 기준으로 구성요소를 분리한 것은 맞음                               - SessionStart Hook의 역할과 근거는 명확함                                          
                                                                                                                   
  부족한 부분                                                                                                      
  - gap-analyzer의 전제 오류는 설계의 핵심 작업(F)을 무효화할 수 있는 수준의 문제                                  
  - requirements-sync는 범위가 실제 필요보다 협소하게 설계됨                                                       
  - Skill의 상태 일관성 문제는 구현 단계에서 실패로 이어질 수 있는 구조적 취약점
