# 에이전트 설계 원칙

> 경영정보 에이전트 설계 과정에서 도출한 구성요소별 설계 기준. Claude Code 플러그인 구조(SKILL.md, plugin.json) 기반.

---

## 1. 구성요소 개요

|구성요소|한 줄 정의|핵심 질문|
|---|---|---|
|에이전트|목표를 자율적으로 달성하는 주체|스스로 계획하고 판단하는가?|
|플러그인|도메인 지식의 배포 단위|독립적으로 의미 있는 능력 묶음인가?|
|Skill|LLM 판단이 필요한 작업 단위|어떻게 생각하고 행동할지 가르쳐야 하는가?|
|MCP 서버|외부 시스템 접근 도구|실행만 하면 되는가?|
|서브에이전트|독립·병렬 위임 단위|메인 흐름과 분리해서 실행하는 게 나은가?|
|Hooks|이벤트 기반 자동 실행|항상, 판단 없이 실행되어야 하는가?|

---

## 2. 에이전트 (Agent)

### 정의

목표를 받아서 스스로 계획을 세우고, 도구를 실행하고, 결과를 보고 다음 행동을 수정하는 자율적 주체.

### 에이전트로 만드는 기준

```
① 목표 지향성
   단순 명령 실행이 아니라 "목표"를 받는다
   "매출 데이터 보여줘" (명령) vs
   "이번 달 실적 이상한 거 찾아서 원인까지 파악해줘" (목표)

② 멀티스텝 자율성
   여러 단계를 스스로 결정한다
   조회 → 이상치 감지 → 드릴다운 → 리포트
   각 단계를 사용자가 지시하지 않아도 된다

③ 자가 수정
   중간 결과를 보고 다음 행동을 바꾼다
   이상치가 없으면 드릴다운 생략
   데이터가 부족하면 추가 질의
```

### 에이전트 단위 기준

```
하나의 에이전트 = 하나의 페르소나 + 하나의 책임 영역

좋은 예
├── 경영정보 에이전트  (경영 데이터 분석 전담)
├── 계약 검토 에이전트 (법무 문서 검토 전담)
└── 코딩 에이전트      (코드 작성/리팩토링 전담)

나쁜 예
└── 뭐든지 에이전트   (경계 없음 → CF 설계 불가)
```

### 에이전트 vs 서브에이전트 구분

```
메인 에이전트: 사용자와 직접 대화하며 전체 목표 관리
서브에이전트: 메인이 위임하는 전문화된 단일 작업
```

## 3. 플러그인 (Plugin)

### 정의

Skills, 서브에이전트, Hooks, MCP 서버를 하나의 배포 단위로 묶은 컨테이너. 에이전트에 능력을 추가하는 단위.

### 플러그인으로 분리하는 기준 — 3가지 질문

```
Q1. 도메인 의존성
    이 능력이 특정 도메인 지식을 알아야만 실행 가능한가?
    
    Yes → 해당 도메인 플러그인에 포함
    No  → 범용 플러그인으로 분리 후보

Q2. 입출력 범용성
    입력이 특정 도메인 데이터에 고정되어 있는가?
    
    Yes → 해당 도메인 플러그인에 포함
    No  → 범용 플러그인으로 분리 후보

Q3. 단독 완결성
    분리했을 때 그 플러그인이 단독으로 의미 있는가?
    
    Yes → 분리
    No  → 합치기
```

### 예시 — PPT 생성

```
PPT 슬라이드 생성 자체
├── 도메인 의존성: 없음 (어떤 내용이든 PPT로 만들 수 있음)
├── 입출력 범용성: 있음 (어떤 데이터든 입력 가능)
└── 단독 완결성: 있음
    → PPT 플러그인으로 분리 ✅

경영 실적 월간 보고 PPT 템플릿
├── 도메인 의존성: 있음 (경영 지표 구조를 알아야 함)
├── 입출력 범용성: 없음 (경영 데이터에 고정)
└── 단독 완결성: 없음 (PPT 플러그인 없이는 작동 안 함)
    → 경영정보 플러그인 내 Skill로 ✅
      (PPT 플러그인을 호출하는 방식)
```

### 플러그인 구조

```
my-plugin/
├── .claude-plugin/
│   └── plugin.json     # 메타데이터 (이름, 설명, 작성자)
├── skills/             # LLM 판단 작업
│   └── skill-name/
│       └── SKILL.md
├── agents/             # 서브에이전트
│   └── agent-name.md
├── hooks/              # 이벤트 자동 실행
│   └── hooks.json
└── .mcp.json           # MCP 서버 연결
```

## 4. Skill

### 정의

LLM이 특정 작업을 어떻게 생각하고 행동해야 하는지를 가르치는 지침. 에이전트가 맥락에 따라 자동으로 로드하거나, 사용자가 /슬래시커맨드로 직접 호출.

### Skill로 만드는 기준

```
LLM의 판단이 필요한가?
    → 자연어를 SQL로 변환하는 것은 판단 (Skill)
    → SELECT 쿼리를 DB에 실행하는 것은 단순 실행 (MCP)

판단 방식을 우리가 통제해야 하는가?
    → LLM에게 맡기면 불안정한 것
    → 일관된 기준이 필요한 것
```

### 두 가지 호출 방식

```
model-invoked (자동)
    에이전트가 대화 맥락 보고 스스로 로드
    description의 트리거 조건에 맞으면 자동 발동
    → text-to-sql: "데이터 보여줘" 맥락이면 자동

user-invocable (수동)
    사용자가 /skill-name 으로 직접 호출
    → /조회, /분석, /비교

조합 규칙
├── 기본: 둘 다 가능 (자동 + 수동)
├── disable-model-invocation: true
│   → 수동만 가능
│   → 부작용 있거나 타이밍을 사람이 통제해야 할 때
│   → 예: /리포트 전송, /배포, /결재요청
└── user-invocable: false
    → 자동만 가능 (사용자가 직접 호출 불필요)
    → 배경 지식 제공용
    → 예: legacy-context (시스템 구조 설명)
```

### Skill 파일 구조 (SKILL.md)

```yaml
---
name: text-to-sql
description: >
  자연어 질문을 SQL 쿼리로 변환.
  사용자가 데이터 조회, 매출/비용/실적 관련 질문을 할 때 사용.
allowed-tools: mcp__db-mcp
disable-model-invocation: false
user-invocable: true
---

# Text-to-SQL

## Rules
- 수치는 반드시 출처 테이블·컬럼과 함께 제시
- DML(INSERT/UPDATE/DELETE) 절대 생성 금지
- 실행 전 사용자에게 쿼리 확인

## Skills
- 기간 표현("지난달", "저번 분기") 자동 해석
- 모호한 질문은 명확화 질문 후 진행

## Examples
[좋은 예]
입력: "3월 사업부별 매출 보여줘"
출력: SELECT dept, SUM(revenue) FROM sales WHERE month=3 GROUP BY dept

[나쁜 예]
입력: 동일
출력: SELECT * FROM sales  ← 집계 없음, 컬럼 불명확
```

## 5. MCP 서버

### 정의

외부 시스템에 실제로 접근하는 실행 도구. LLM의 판단 없이 입력을 받아 실행하고 결과를 반환.

### MCP로 만드는 기준

```
판단이 필요 없는 순수 실행인가?
    → DB 쿼리 실행: Yes → MCP
    → 어떤 쿼리를 쓸지 판단: No → Skill

외부 시스템에 접근해야 하는가?
    → Yes → MCP
    → No  → Skill 또는 기본 도구
```

### Skill vs MCP 구분 예시

```
Skill (판단)                    MCP (실행)
────────────────────────       ──────────────────────────
자연어 → SQL 변환               SELECT문 DB에 실행
이상치인지 판단                  ERP API 호출
어떤 테이블 쓸지 결정            파일 저장
기간 표현 해석                   메신저 메시지 전송
결과 해석·요약                   차트 이미지 생성
```

### .mcp.json 구조

```json
{
  "mcpServers": {
    "db-mcp": {
      "command": "node",
      "args": ["./mcp-servers/db/index.js"],
      "env": { "DB_CONNECTION": "..." }
    },
    "erp-mcp": {
      "command": "node",
      "args": ["./mcp-servers/erp/index.js"]
    }
  }
}
```

## 6. 서브에이전트 (Sub-agent)

### 정의

메인 에이전트가 특정 작업을 위임하는 독립적 실행 단위. 메인 흐름과 분리되어 실행되고 결과를 메인에게 반환.

### 서브에이전트로 만드는 기준

```
① 메인 흐름과 독립적으로 실행 가능한가?
   → SQL 검증은 메인 대화 흐름 밖에서 실행 가능

② 병렬 처리가 효율적인가?
   → 시각화 + 리포트 구성을 동시에 실행

③ 전문화된 단일 책임인가?
   → "SQL만 검증", "차트만 생성"

④ 메인 에이전트가 결과만 받으면 되는가?
   → 중간 과정을 메인이 알 필요 없음
```

### Skill vs 서브에이전트 구분

```
Skill                           서브에이전트
────────────────────────       ──────────────────────────
메인 흐름 안에서 실행           메인 흐름 밖에서 독립 실행
동기적 처리                     비동기·병렬 처리 가능
경량 판단 작업                  무거운 전문 작업
맥락 공유                       격리된 실행 환경
```

## 7. Hooks

### 정의

특정 이벤트 발생 시 LLM 판단 없이 자동으로 실행되는 코드. "항상 이래야 한다"는 규칙을 코드 레벨에서 강제.

### Hooks로 만드는 기준

```
LLM 판단에 맡기면 안 되는 것인가?
    → CF는 확률적. 100% 보장이 필요한 것은 Hook

항상 실행되어야 하는 것인가?
    → 예외 없이 모든 경우에 적용

부작용 방어가 목적인가?
    → 잘못된 실행을 막는 안전장치
```

### 이벤트 유형

```
PreToolUse  ← 도구 실행 전
    → 허용/거부/수정 결정
    → 예: DML 차단, 위험 경로 접근 차단

PostToolUse ← 도구 실행 후
    → 결과 가공·로깅
    → 예: 출처 첨부, 결과 포맷 변환

UserPromptSubmit ← 사용자 입력 시
    → 입력 전처리
    → 예: 개인정보 마스킹

Stop ← 에이전트 종료 시
    → 정리 작업
    → 예: 세션 로그 저장
```

### hooks.json 구조

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "db-mcp",
        "hooks": [{
          "type": "prompt",
          "prompt": "SQL에 INSERT/UPDATE/DELETE가 포함되면 deny. SELECT만 허용."
        }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "db-mcp",
        "hooks": [{
          "type": "prompt",
          "prompt": "결과에 반드시 출처 테이블명을 포함시켜라."
        }]
      }
    ]
  }
}
```

## 8. 판단 트리 — 작업 계위 결정

새로운 작업이 생겼을 때 이 순서로 결정.

```
새 작업 등장
      ↓
외부 시스템 접근이 필요한가?
      ├── Yes → MCP 서버
      └── No  ↓

항상·자동으로 실행되어야 하는가? (판단 불필요)
      ├── Yes → Hooks (PreToolUse / PostToolUse)
      └── No  ↓

목표를 받아 멀티스텝으로 자율 실행하는가?
      ├── Yes → 에이전트 (또는 서브에이전트)
      └── No  ↓

LLM 판단이 필요한가?
      ├── No  → 기본 도구 사용으로 해소
      └── Yes ↓

메인 흐름 밖에서 독립·병렬 실행이 나은가?
      ├── Yes → 서브에이전트
      └── No  → Skill

플러그인 분리 여부
      ↓
도메인 의존성 있는가? + 입출력 범용성 있는가? + 단독 완결성 있는가?
      ├── 도메인 종속 → 해당 도메인 플러그인
      └── 범용        → 별도 플러그인으로 분리
```

## 9. 경영정보 에이전트 적용 예시

### 9-1. 전체 플러그인 구조

```
경영정보 에이전트
├── 경영정보 플러그인     (도메인 종속)
├── PPT 플러그인          (범용)
├── 그룹웨어 플러그인     (범용)
├── 스케줄러 플러그인     (범용)
└── Excel 플러그인        (범용, 조건부)
```

플러그인 분리 기준 재확인:

|플러그인|도메인 의존성|입출력 범용성|단독 완결성|판정|
|---|---|---|---|---|
|경영정보|있음|없음|없음|도메인 종속|
|PPT|없음|있음|있음|범용 분리|
|그룹웨어|없음|있음|있음|범용 분리|
|스케줄러|없음|있음|있음|범용 분리|
|Excel|없음|있음|있음|범용 분리|

---

### 9-2. 경영정보 플러그인 (도메인 종속)

경영 데이터 도메인 지식이 없으면 실행 불가능한 것들만 포함.

```
경영정보 플러그인
├── Skills
│   ├── text-to-sql          [자동+수동]
│   │   자연어 → SQL 변환
│   │   경영 KPI 용어 해석 포함 (매출 = revenue 컬럼 등)
│   │
│   ├── schema-lookup        [자동]
│   │   질문 의도에 맞는 테이블·컬럼 탐색
│   │   고객사 스키마 Knowledge 참조
│   │
│   ├── anomaly-detect       [자동]
│   │   경영 지표 기준 이상치 판단
│   │   단순 수치 이상이 아닌 업종 기준 적용
│   │
│   ├── period-compare       [자동]
│   │   "지난달", "전분기", "YoY" 등 기간 표현 해석
│   │   회계 기간(사업연도) 기준 적용
│   │
│   ├── drill-down           [자동]
│   │   이상치 발견 시 원인 데이터 자동 탐색
│   │   경영 계층 구조 이해 (사업부 > 팀 > 제품군)
│   │
│   └── report-template      [자동]   ← PPT 플러그인 호출
│       경영 실적 데이터를 월간 보고 형식으로 구성
│       PPT 플러그인에 구조화된 데이터로 넘김
│
├── 서브에이전트
│   ├── sql-validator
│   │   생성된 SQL 오류 검증·수정
│   │   집계 함수, 조인 조건, 날짜 형식 체크
│   │
│   ├── insight-writer
│   │   데이터 결과를 경영진 언어로 서술
│   │   "C사업부 매출 40% 감소, 주요 원인은..."
│   │
│   └── report-assembler
│       여러 조회 결과를 하나의 보고서로 취합
│       섹션 구성, 순서, 요약 작성
│
├── Hooks
│   ├── PreToolUse [db-mcp, erp-mcp]
│   │   ├── DML 차단: SELECT 외 쿼리 코드 레벨 차단
│   │   └── 쿼리 확인: 실행 전 사용자 승인 요청
│   │
│   └── PostToolUse [db-mcp, erp-mcp]
│       ├── 출처 첨부: 결과에 테이블명·조회 시점 표시
│       └── 수치 포맷: 단위(억원, %) 자동 적용
│
└── MCP 서버
    ├── db-mcp       PostgreSQL·MySQL·Oracle 쿼리 실행
    ├── erp-mcp      ERP(더존·SAP) 데이터 조회
    └── viz-mcp      차트·Excel 파일 출력

```

PPT 플러그인

│   │
│   ├── layout-selector       [자동]
│   │   콘텐츠 유형에 맞는 레이아웃 선택
│   │   텍스트 중심 / 차트 중심 / 비교표 / 타이틀

│   │   bar / line / pie 유형 판단
│   │
│   └── /ppt-create           [수동만, disable-model-invocation: true]
│       사용자가 명시적으로 PPT 생성 시작
│       에이전트가 알아서 PPT를 만들면 안 되므로 수동 전용

├── 서브에이전트
│   └── slide-renderer
│       슬라이드 계획 → 실제 파일 생성
│       python-pptx 기반 렌더링 실행
│
├── Hooks
│   └── PostToolUse [slide-renderer]
│       생성 완료 후 파일 경로·슬라이드 수 확인 메시지
│
└── MCP 서버
    └── pptx-mcp     .pptx 파일 생성·저장

    report-template (PPT용 구조화)

    slide-planner → layout-selector
    chart-embedder
    slide-renderer → pptx-mcp (파일 생성)
         ↓ 파일 경로 전달
그룹웨어 플러그인
    recipient-resolver ("팀장님" → 실제 수신자)
    message-composer
    /send (사용자 최종 확인 후)
    kakaowork-mcp (전송)
