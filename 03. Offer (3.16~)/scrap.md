1단계: 구축 수익
   AI Agent 도입 컨설팅
   → CF 설계 + Agent 구축 납품
   → 고객사 특화 CF가 산출물에 포함

2단계: 재사용 수익
   동일 버티컬 2번째 고객부터
   → 1단계에서 축적한 CF 자산 재활용
   → 원가 절감 = 수익률 향상 OR 가격 경쟁력

3단계: 유지보수 수익
   CF 업데이트 (규제 변경, 프레임워크 버전업)
   RAG 지식베이스 갱신
   → 연간 계약 형태


**Agent 배포·관리 플랫폼**이 핵심 아이템입니다. CF를 작성해도 조직 전체에 배포하고, 버전을 관리하고, 누가 어떤 Agent를 쓰는지 추적하는 체계가 없으면 확산이 안 됩니다. 이 플랫폼 자체가 상품이 됩니다.

| 구분      | 내용                                                |
| ------- | ------------------------------------------------- |
| 역할      | CF 버전 관리, 에이전트 등록/배포, 사용 로그, 권한 관리, CF 업데이트 중앙 배포 |
| 형태      | 웹 기반 관리 콘솔. 관리자 화면과 사용자 화면 분리                     |
| 관리자 사용자 | IT 담당자, 프로젝트 리더. CF 등록·수정·배포                      |
| 일반 사용자  | 현업 직원. 배포된 에이전트를 채팅 UI로 사용                        |
**Skill Evaluator(CF 품질 측정 도구)** 도 중요합니다. "이 CF를 적용했더니 코드 리뷰 시간이 40% 줄었다"는 수치가 없으면 다음 프로젝트 제안이 어렵습니다. Skill 적용 전후 지표를 자동으로 수집하는 도구는 영업 도구이기도 합니다.
**코딩 에이전트의 경우**: 생산성 향상 → 개발 기간 단축 → 같은 인력으로 더 많은 프로젝트 수주 가능 → 매출 증대. 또는 동일 프로젝트를 더 적은 인력으로 → 수익률 향상. 생산성은 이 사슬의 첫 번째 고리일 뿐입니다.
	**생산성 외의 가치**:
	- 품질: 버그 발생률 감소, 보안 취약점 사전 차단 → 운영 리스크 감소
	- 지식 보존: 핵심 개발자 퇴직 시 CF에 지식이 남아있음 → 조직 연속성
	- 온보딩: 신규 입사자가 CF를 통해 빠르게 사내 표준 습득
	**코딩 외 버티컬의 가치 측정**: 생산성 지표가 아니라 리스크 지표로 접근해야 합니다. 계약 검토는 "검토 누락으로 인한 분쟁 발생 건수", 컴플라이언스는 "법적 위반 적발 건수"가 측정 지표입니다. 이쪽이 오히려 비용이 명확하게 잡힙니다. 분쟁 한 건이 수억이므로 에이전트 구축 비용과 ROI 비교가 쉽습니다.

**MCP 커넥터 라이브러리**도 자산이 됩니다. ERP, ITSM, DMS 등 국내 기업 환경에 맞는 MCP 래퍼를 표준화해두면 구축 속도가 빨라지고, Tools 교체 비용이 줄어듭니다. 예를 들어 더존 ERP, 영림원 ERP, 그룹웨어(카카오워크, 네이버웍스), 전자결재 시스템 등입니다. 글로벌 솔루션은 SAP, Salesforce MCP는 이미 있지만 국내 특화 시스템은 없습니다.

|형태|성격|예시|차이|
|---|---|---|---|
|**Rules**|정적 제약. 판단 없음. 항상 적용|"클래스명은 반드시 PascalCase" / "하도급법 3조 필수 포함"|위반 불가. if-then 없음|
|**Skills**|조건부 절차. 판단 포함|"커버리지 60% 미만이면 테스트 먼저 작성 후 리팩토링 제안"|상황에 따라 다른 행동|
|**Knowledge**|참조 문서. 검색하여 사용|Spring 가이드 / 판례 DB 벡터화|필요할 때만 꺼내 씀|
|**Examples**|입출력 패턴. few-shot|좋은/나쁜 코드 쌍 / 위험/안전 조항 비교|직접 규칙화 어려운 것을 사례로|
|**Memory**|대화 내 상태 유지|"오늘 오전 UserDao 수정됨"|세션 한정|

첫째, **CF 소유권 문제**입니다. 고객사 납품 시 CF는 누구 것인가가 계약서에 명확하지 않으면, 고객사가 CF를 가져가서 경쟁사에 넘기거나 직접 운영할 수 있습니다. 재사용 자산이 외부로 유출되는 구조를 막는 법적·기술적 장치가 필요합니다.

둘째, **LLM 의존성 리스크**입니다. Claude/GPT API 비용, 버전 변경, 서비스 중단 시 전체 서비스가 멈춥니다. 고객사 입장에서도 "우리 계약서가 외부 API로 나가는 것"에 대한 보안 거부감이 큽니다. On-premise LLM 옵션 없이는 공공기관이나 금융권 수주가 어렵습니다.

셋째, **에이전트 오답의 책임 소재**입니다. 계약 검토 에이전트가 리스크 조항을 놓쳤을 때, 혹은 코딩 에이전트가 보안 취약점을 포함한 코드를 생성했을 때 책임은 누구에게 있는가. 이것이 정리되지 않으면 고객사 의사결정권자가 도입 승인을 내리지 않습니다. 상품 설계 시 "에이전트는 보조 도구이며 최종 판단은 사람이 한다"는 구조를 명확히 해야 합니다.
