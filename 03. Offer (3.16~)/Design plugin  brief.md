# Design Plugin — 설계 명세 (plugin-dev Discovery 대체)

> 이 문서는 plugin-dev의 Phase 1 Discovery를 대체한다. Phase 2 Component Planning부터 시작하면 된다.

---

## 플러그인 목적

에이전트를 설계하는 사람이 발화 가설 수립부터 구성 확정까지 구조적으로 진행할 수 있도록 안내하는 플러그인.

설계 결과물은 **에이전트 설계 매트릭스** — 발화·요구사항·구성요소가 하나의 문서에 연결된 형태. 이 매트릭스가 완성되면 plugin-dev로 넘어간다.

---

## 대상 사용자

에스코어 컨설턴트 또는 Claude Code 기반 에이전트를 설계하는 사람. Claude Code와 플러그인 구조에 익숙하다고 가정한다. AI 사용에 능숙하나 에이전트 설계 방법론은 외부에서 학습 필요.

---

## 해결할 작업

단계별 상세 정의는 `agent_design_process.md` Steps 1~7 참조. 아래는 각 단계의 기능적 설명이다.

|단계|작업|입력|출력|
|---|---|---|---|
|1|페르소나 정의|사용자가 제공하는 도메인·역할 정보|페르소나 카드 (YAML)|
|2|발화 가설 작성|페르소나 카드|설계 매트릭스 (발화 컬럼)|
|3|설계 전 Walk-through|설계 매트릭스 (발화)|설계 매트릭스 (요구사항 컬럼 추가)|
|4|계위 판단|설계 매트릭스 (요구사항)|설계 매트릭스 (구성요소·계위·플러그인 컬럼 추가)|
|5|발화 보완|설계 매트릭스 (구성요소)|설계 매트릭스 (발화 행 추가)|
|6|설계 후 Walk-through|설계 매트릭스 (구성요소)|설계 매트릭스 (구멍·심각도 컬럼 추가)|
|7|구성 확정|설계 매트릭스 (구멍 처리 완료)|설계 매트릭스 최종본|

**이 플러그인의 범위: Steps 1~7만.** Step 8 (CF 설계서 작성) → plugin-dev 담당. Step 9 (Skill eval 루프) → skill-creator 담당.

---

## 핵심 설계 제약

### 매트릭스 단일 문서 원칙

발화·요구사항·구성요소는 별도 문서로 분리하지 않는다. 단계가 진행될수록 같은 매트릭스에 컬럼이 추가되는 구조.

### 설계 전/후 Walk-through 구분

Walk-through는 두 번 실행하며 목적이 다르다.

- 설계 전: 인풋 = 발화 + 업무 프로세스. 아웃풋 = 요구사항.
- 설계 후: 인풋 = 발화 + 구성요소 목록. 아웃풋 = 구멍. 구멍 발견 즉시 해결하지 않는다. 매트릭스에 기록하고 추적을 계속한다.

### 구멍 판정 5기준

설계 후 Walk-through에서 이 기준으로만 구멍을 판정한다. ① 연결 없음 ② 입출력 불일치 ③ 트리거 미정의 ④ 도구 미연결 ⑤ 실패 경로 미처리 상세 정의는 `agent_design_process.md` Step 6 참조.

---

## 다른 플러그인과의 협업

### Step 4에서 plugin-dev 협업

계위 판단 단계에서 plugin-dev의 plugin-structure Skill이 보유한 판단 트리 지식을 활용한다. design plugin이 담당하는 것: 요구사항 행 단위 매핑, 매트릭스 컬럼 채우기. plugin-dev가 담당하는 것: 계위 판단 로직 자체.

### Step 7 완료 후 plugin-dev 핸드오프

구성 확정된 매트릭스를 plugin-dev에 전달하면 plugin-dev가 실제 파일 구조와 SKILL.md를 작성한다.

### Step 8~9는 이 플러그인이 관여하지 않음

plugin-dev (skill-development Skill) → SKILL.md 작성 skill-creator → eval 루프 및 description 최적화

## 각 단계별 Skill 동작 명세

> plugin-dev Phase 5에서 각 Skill의 SKILL.md를 작성할 때 참조. 각 Skill이 어떻게 동작해야 하는지의 규칙과 제약이다.

### Step 1 — 페르소나 정의

산출물 형식:

```yaml
name / role / company / ai_proficiency / workflow / structural_barriers / pain_points / utterance_patterns
```

규칙:

- "못하는 사람도 할 수 있게"가 아닌 "잘하는 사람이 구조적 장벽을 넘게" 관점으로 작성
- pain_points는 표면 니즈가 아닌 "왜"를 한 번 더 파고든 결과
- utterance_patterns에는 모호한 표현, 상황 의존 표현 포함

### Step 2 — 발화 가설 작성

선정 기준 (4가지 커버 필수):

```
자주 쓰이는 경로 / 임기응변·모호한 입력 / 크로스 시스템 경로 / 실패하면 치명적인 경로
```

규칙:

- 실제 사용자가 할 법한 말인지가 기준. 구성요소 커버리지는 Step 5에서 적용.
- 산출물: 설계 매트릭스의 발화 컬럼만 채운 상태

### Step 3 — 설계 전 Walk-through

규칙:

- 발화 하나를 에이전트 시점으로 단계별로 따라가며 필요한 작업을 도출
- 각 단계에서 "이 작업을 처리하려면 무엇이 필요한가"를 명시 (추정 금지)
- 계위(Skill인지 MCP인지)는 아직 결정하지 않음
- 하나의 발화가 여러 요구사항을 만들 수 있음 → 매트릭스에 여러 행으로 기록
- 산출물: 설계 매트릭스의 요구사항 컬럼 추가

### Step 4 — 계위 판단

규칙:

- agent_component_reference.md의 판단 트리를 요구사항 행 단위로 적용
- Command 배정 시: 사용자가 /슬래시로 직접 호출하는가 + 여러 Skill 조합이 필요한가 확인
- references/ 배정 시: Skill 기준 셋 다 해당 없는가 + 특정 Skill과 항상 함께 쓰이는가 확인
- 산출물: 설계 매트릭스의 구성요소·계위·플러그인 컬럼 추가

### Step 5 — 발화 보완

확인 항목 (4가지):

```
Command 진입점 발화가 있는가
Skill 체이닝 경로가 발화에 있는가
플러그인 간 연결 경로가 발화에 있는가
실패·예외 경로가 발화에 있는가
```

규칙:

- 구성요소 목록을 보고 커버 안 된 경로에 해당하는 발화만 추가
- 산출물: 설계 매트릭스에 발화 행 추가

### Step 6 — 설계 후 Walk-through

구멍 판정 5기준 (이 기준으로만 판정):

```
① 연결 없음        필요한 구성요소가 매트릭스에 없다
② 입출력 불일치    앞 단계 출력 형식 ≠ 이 단계 입력 형식
③ 트리거 미정의    이 Skill/Command가 언제 발동되는지 정의 없음
④ 도구 미연결      이 Skill의 allowed-tools에 필요한 MCP 없음
⑤ 실패 경로 미처리 이 단계 실패 시 에이전트 행동 미정의
```

심각도 기준:

```
심각  → 작동 안 됨. 구성 수정 후 Walk-through 재실행
중간  → 문제 생김. 해당 발화만 재추적
경미  → 나중에 가능. Could 항목으로 이관
```

규칙:

- 구멍 발견 즉시 해결하지 않음. 매트릭스에 기록하고 추적 계속.
- 완료 기준: 모든 발화가 "사용자에게 반환" 또는 "구멍 기록"으로 끝난 상태
- 심각 구멍이 있으면 Step 4로 돌아가 구성 수정 후 재실행
- 산출물: 설계 매트릭스의 구멍·심각도 컬럼 추가

### Step 7 — 구성 확정

조건: 설계 후 Walk-through에서 심각 구멍이 없을 것 규칙:

- 구멍 처리 내역이 해당 행에 반영된 상태
- 변경된 행은 비고에 "GXX로 수정/추가" 표시
- 산출물: 설계 매트릭스 최종본 (이후 plugin-dev로 핸드오프)
