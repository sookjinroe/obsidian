## 1. 핵심 개념

### AI Agent

LLM을 두뇌로 삼아 목표가 주어지면 스스로 계획을 세우고 외부 도구를 실행하며 결과를 보고 다음 행동을 수정하는 자율 시스템.

**ReAct 루프** (작동 원리):

```
Thought(추론) → Action(도구 실행) → Observation(결과 확인) → 반복
```

### Context Framework (CF)

"LLM이 어떻게 생각하고 행동해야 하는가"를 사전에 정의한 체계. 단순 프롬프트가 아니라, LLM 활용의 일관성과 재현성을 보장하는 맥락 관리 체계.

**CF의 5가지 형태:**

|형태|성격|API 입력 위치|
|---|---|---|
|Rules|절대 제약. 판단 없음. 항상 적용|system (상단)|
|Skills|조건부 절차. 상황 판단 포함|system (하단)|
|Knowledge|RAG용 참조 문서. 필요 시 검색|documents[]|
|Examples|few-shot 패턴. 좋은/나쁜 사례 쌍|messages[]|
|Memory|세션 내 상태 유지|messages[]|

> Rules vs Skills: Rules는 "무조건 이렇게 해라", Skills는 "이런 상황이면 이렇게 판단해서 행동해라"

**CF 표준화 방향**: Anthropic Agent Skills, OpenClaw SKILL.md 등 업계 표준이 수렴 중. 에스코어 CF는 런타임 독립적 내부 표준 포맷으로 작성하고, 런타임별 어댑터를 통해 변환.

### MCP (Model Context Protocol)

LLM에게 외부 시스템을 "호출 가능한 함수" 형태로 제공하는 오픈 표준. Anthropic, OpenAI, Google, Microsoft 모두 채택. 사실상 업계 표준.

**에스코어의 MCP 전략:**

- 이미 MCP 있는 시스템(더존, MS, 카카오워크, Slack 등) → **가져다 씀**
- MCP 없는 시스템(공공 레거시, 금융 내부 등) → **파트너십으로 공동 개발**
- MCP 커넥터 자체는 점점 commodity화 중. 커넥터보다 **CF + 에이전트**가 자산.

### 에이전트 런타임

에이전트가 실행되는 엔진. 에스코어는 런타임과 경쟁하지 않음.

|런타임|용도|
|---|---|
|Cline / Continue|코딩 특화, IDE 환경|
|OpenClaw|범용, 메시징 앱 환경, MIT 라이선스|
|LangGraph|직접 구축 시 기반 프레임워크|

> **전략**: OpenClaw 기반으로 시작, 더 좋은 것 나오면 교체. CF가 런타임 독립적으로 설계되어야 교체 가능.

## 2. 버티컬 선정 기준

Anthropic의 초기 플러그인 선정 기준을 참조:

```
① 고비용 전문직의 반복 작업 자동화
   주니어 뱅커, 법무 담당자의 저부가가치 문서 작업

② SaaS 협력 모델
   기존 툴 대체가 아닌 데이터 벤더 파트너십

③ 검토 중심 워크플로우
   AI 초안(80%) + 인간 전문가 검토(20%) 구조
```

**에스코어 추가 기준:**

- 에스코어가 이미 아는 도메인인가 (금융사 경험, SI 구축 경험)
- 글로벌 벤더(Anthropic, MS)가 커버하지 못하는 국내 특화 영역인가
- 외부 매출이 발생하는 고객이 있는가
- CF 재사용성이 높은가 (불변 레이어 비율)

### 버티컬 우선순위

|버티컬|Anthropic 커버|에스코어 강점|국내 특화|우선순위|
|---|---|---|---|---|
|코딩|✅|판매 중|eGovFrame|**현재**|
|재무/데이터 분석|✅ 글로벌|금융사 경험|국내 회계기준|**1순위**|
|국내 금융 컴플라이언스|❌|금융사 경험|금융규제 특화|**1순위**|
|국내 공공/eGovFrame|❌|SI 경험|전자정부 특화|**1순위**|
|법무/계약|✅ 글로벌|없음|하도급법 등|2순위|
|HR|✅ 글로벌|없음|국내 노동법|2순위|
|영업|✅ 글로벌|없음|-|검토|

> **핵심**: Anthropic이 글로벌 표준을 커버하는 영역에서는 국내 규제·레거시 환경이 공백. 에스코어는 이 공백을 파고드는 포지션.

## 3. 자산 — 에스코어가 쌓아가는 것

### 3-1. CF 자산 (핵심 경쟁력)

**현재 보유:**

- 코딩 CF: eGovFrame 표준, Java/Spring 규칙, 보안 취약점 패턴, 단위 테스트 생성
- 금융 CF: 금융사 납품 경험 기반 컴플라이언스 체크리스트 (일부)

**구축 예정 (버티컬별):**

- 재무/데이터 분석 CF: KPI 정의 표준, 이상치 판단 기준, 재무 분석 패턴
- 국내 금융 컴플라이언스 CF: 금융규제 조문, 내부통제 기준
- 국내 공공 CF: eGovFrame 아키텍처 패턴, 전자정부 연동 규칙

**CF 레이어 구조 (모든 버티컬 공통):**

```
불변 레이어 (모든 고객사 공통) → 에스코어 핵심 자산
└── 규제·표준·업종 공통 기준

가변 레이어 (고객사마다 교체) → 커스터마이징 비용
└── 고객사 고유 기준, 내부 프로세스, 시스템 연동
```

### 3-2. 엔터프라이즈 플랫폼

## 4. 오퍼 — 왜 에스코어인가

> 고객이 다른 선택지를 고를 때 부담해야 하는 것 vs 에스코어를 선택할 때

### 오퍼 1. 국내 규제·레거시 환경을 이미 알고 있습니다

Anthropic, MS, AWS는 글로벌 표준을 다루지만 국내 특화 환경을 모릅니다. 에스코어는 eGovFrame, 국내 금융규제, 국내 레거시 시스템을 이미 SI 경험으로 알고 있습니다. → **글로벌 벤더가 커버하지 않는 국내 환경의 CF와 연동이 처음부터 가능합니다.**

### 오퍼 2. 도메인 CF가 이미 구조화되어 있습니다

AI 에이전트가 전문가처럼 작동하려면 도메인 지식이 CF로 녹아있어야 합니다. 처음부터 만들면 도메인 전문가 인터뷰 + CF 설계 + 검증에 수개월이 걸립니다. → **에스코어의 불변 레이어는 이미 있습니다. 고객사 고유 기준만 추가하면 됩니다.**

### 오퍼 3. 개인 도구가 아닌 조직 인프라를 제공합니다

Claude.ai, ChatGPT는 개인이 씁니다. 조직 전체가 안전하게 쓰려면 별도 인프라가 필요합니다. → **CF 중앙 관리, RBAC, 감사 로그, On-premise 배포를 플랫폼으로 제공합니다.** → **규제 변경 시 전사 에이전트 동시 업데이트. 개인 도구로는 불가능합니다.**

### 오퍼 4. AI로 개발하기 때문에 더 빠릅니다

→ **일반 SI 대비 개발 속도 30~50% 단축. 더 빠른 ROI.**

## 5. 경쟁 구도

|경쟁자|그들이 잘하는 것|우리의 포지션|
|---|---|---|
|Anthropic/OpenAI|글로벌 표준 CF, 범용 에이전트|국내 특화 CF, 기업 인프라|
|MS Copilot|M365 내 통합 에이전트|크로스 시스템 오케스트레이션|
|더존 ONE AI|더존 ERP 내 에이전트|더존 + 타 시스템 연결 에이전트|
|LG CNS / SK C&C|각자 계열사 기반|에스코어: 삼성 계열 기반 + 외부 확장|

> **핵심 포지션**: 단일 SaaS 안에서 작동하는 에이전트는 각 벤더가 한다. 여러 시스템을 넘나드는 크로스 시스템 에이전트는 에스코어가 한다.

## 6. 수익 구조

```
1단계: 구축 수익 (현재)
   버티컬 에이전트 구축 납품
   → 코딩 에이전트 (진행 중)
   → 재무/금융/공공 버티컬 확장

2단계: 재사용 수익
   동일 버티컬 2번째 고객부터
   → 불변 레이어 CF 재활용 → 원가 절감 → 수익률 향상

3단계: 유지보수 구독
   CF 업데이트 (규제 변경, 시스템 버전업)
   → 연간 계약
```
