# 커버리지 확인 및 발화 보완 (artifact-v4)

---

## 커버리지 점검

모든 구성요소를 순회하며 기존 발화(U01~U10)에 의해 발동 경로가 확보되어 있는지 확인한다.

| 구성요소명 | 계위 | 발동 경로 확보 여부 | 발동 발화 |
|---------|-----|----------------|---------|
| nl-to-sql | Skill | ✅ | U01, U02, U03, U04, U07, U08 |
| date-resolver | Skill | ✅ | U02, U03 |
| conversation-context | Skill | ✅ | U04 |
| schema-explorer | Skill | ✅ | U05, U06 |
| schema-reference | references | ✅ (nl-to-sql/schema-explorer 경유) | U01~U08 |
| redshift-connector | MCP | ✅ | U01~U08 |
| dml-guard | Hook | ✅ | U09, U10 — DML 거부 경로 확보 |

모든 구성요소에 발동 경로가 존재한다. 단, 아래 두 가지 경계 케이스에서 경로가 약하거나 테스트되지 않은 상황이 발견된다.

---

## 보완 발화 추가

### 보완 이유 1: dml-guard — 암묵적 DML 유도 케이스 미포함
U09·U10은 명시적 삭제/수정 요청이지만, "잘못된 데이터를 고쳐달라"는 간접 표현이나 UPDATE 구문을 SQL로 직접 요청하는 경우가 실무에서 발생할 수 있다. dml-guard의 경계값 테스트 발화가 필요.

### 보완 이유 2: redshift-connector — 연결 실패 또는 빈 결과 반환 케이스 미포함
쿼리는 맞지만 결과가 없거나 연결 오류 시 에이전트 동작 경로가 명시되지 않음.

### 보완 이유 3: schema-reference — 스키마에 없는 테이블명 발화 케이스 미포함
사용자가 실제로 존재하지 않는 테이블명을 입력했을 때의 경로가 없음.
