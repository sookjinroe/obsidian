# 설계 진행 절차

## 충분성 판단 루프

**목적:** design-drafter가 초안을 생성할 수 있을 만큼의 최소 맥락을 확보한다.

### 충분성 기준

아래 세 가지가 모두 파악되면 drafter를 호출한다.

| 항목 | 충분 조건 예시 |
|---|---|
| ① 사용 주체 | "재무팀 데이터 분석가", "영업팀 매니저", "법무 담당자" |
| ② 핵심 목적과 불편함 | "자연어로 SQL 조회", "계약서 검토 자동화", "수동 집계 대체" |
| ③ 핵심 제약 | "Redshift 연결 필요", "DML 차단 필수", "사내망만 접근 가능" |

세부 시스템 명칭, 테이블 구조, 내부 용어는 필수가 아니다. drafter가 ⚠️로 마킹하고 초안 확인 시 보완한다.

### 진행 방식

1. 사용자의 첫 발화에서 ①②③ 충족 여부를 판단한다
2. 충족되지 않은 항목만 질문한다. 한 번에 하나씩.
3. 매 답변 후 다시 충족 여부를 판단한다
4. 세 가지 모두 충족 → drafter 호출

### 표면 답변에서 파고드는 방법

답변이 피상적일 때 아래 패턴으로 한 단계 더 들어간다. 가장 중요한 하나만 골라 파고든다.

**"시간이 걸린다" 류 답변 →** 어느 단계에서 막히나요? 시스템 문제인가요, 정보를 찾아야 해서인가요, 판단을 내려야 해서인가요?

**"복잡하다" 류 답변 →** 복잡함이 어디서 오나요? 여러 곳에서 정보를 모아야 하나요, 아니면 규칙이 많아서인가요?

**"AI 쓰고 있어요" 류 답변 →** 어떤 작업에 쓰나요? 불편한 점이 있나요? *(목적: 에이전트가 기존 도구를 대체하는지 보완하는지 파악해 역할 중복 방지)*

**③이 불명확할 때 →** 이 에이전트가 절대 해서는 안 되는 것이 있나요? 연결해야 하는 시스템이 있나요?

### 예시 흐름

**막연한 경우:**
```
사용자: "업무 자동화를 하고 싶어요"
→ ①②③ 모두 부족
→ "어떤 분야의 업무인가요? 누가 주로 쓸지도 알려주시면 좋겠습니다"

사용자: "영업팀 매니저들이 고객 데이터를 분석하는 업무예요"
→ ① 충족 (영업팀 매니저), ② 부분 충족
→ "지금은 어떻게 하고 계신가요? 어느 부분이 가장 불편하신가요?"

사용자: "매번 SQL 직접 짜거나 개발팀에 요청해야 해요. CRM DB에 바로 접근하고 싶어요"
→ ② 충족 (SQL 직접 작성 불편), ③ 충족 (CRM DB 접근)
→ 충분성 충족 → drafter 호출
```

**구체적인 경우:**
```
사용자: "재무팀 분석가들이 Redshift에서 자연어로 조회하고 싶어. DML은 막아야 해."
→ ①②③ 모두 즉시 충족
→ 바로 drafter 호출
```

## design-drafter 호출 및 초안 확인

### 호출 절차

1. "지금까지 파악한 내용으로 설계 초안을 생성하겠습니다"라고 안내
2. 수집된 정보를 아래 형식으로 요약해 drafter에 전달:

```
사용 주체: [역할/부서/환경]
핵심 목적: [무엇을 해결하는가]
주요 불편함: [현재의 문제]
핵심 제약: [반드시 지켜야 할 것]
추가 컨텍스트: [기타 수집된 정보]
```

3. design-drafter가 `persona.md`, `artifact-v1~4.md`를 자율 생성한다

### 초안 확인 절차

drafter 완료 후:

1. 생성된 구성요소 목록과 발화 수를 요약 제시
2. "큰 방향이 맞나요? 맞지 않는 부분이 있으면 말씀해주세요"
3. ⚠️ 항목 목록을 제시하며 추가 정보 요청 (선택)
4. ⚠️ 항목 중 계위가 불확실한 것 → plugin-dev `plugin-structure` 스킬로 함께 검토
5. 수정사항을 `artifact-v4.md`에 반영

초안 확인 완료 → `state.md`의 `current_step`을 `F`로 업데이트

## gap 검증

**목적:** 완성된 구성 목록으로 각 발화를 처리할 수 있는지 검증하고 gap을 기록한다.

### 진행 방식

`gap-analyzer` 에이전트를 호출해 자율 검증을 실행한다.

```
gap-analyzer 입력: artifact-v4.md
gap-analyzer 출력: gap-report.md
```

gap-report.md를 읽어 결과를 사용자에게 보고한다.

**심각 gap이 있는 경우:**
- 해당 발화와 요구사항을 짚어 사용자에게 설명
- `state.md`의 `loop_history`에 기록
- `iteration` +1
- `artifact-v4.md`에서 해당 항목 수정 후 gap-analyzer 재실행

**심각 gap이 없는 경우:**
- 중간/경미 gap은 산출물에 기록 (경미는 Could 항목으로 이관)
- `artifact-v5.md` 저장
- 구성 확정으로 이동

### 완료 조건

모든 발화 추적이 끝나고, 각 발화가 "처리 가능" 또는 "gap 기록"으로 끝났을 때.

## 구성 확정

**목적:** 심각 gap이 없는 상태의 설계 산출물을 최종본으로 확정하고 plugin-dev에 인계한다.

### 진행 방식

1. `artifact-v5.md`를 기반으로 `artifact-final.md`를 생성한다
2. 확정본에 포함된 정보를 사용자에게 요약 제시:
   - 에이전트 목적 및 사용자
   - 구성요소 목록 (이름, 계위, 소속 플러그인)
   - 발화-요구사항-구성요소 연결 관계
   - 미해소 gap(중간/경미) 목록
3. 사용자 최종 확인
4. `state.md`의 `current_step`을 `finalized`로 업데이트

### plugin-dev 인계

"plugin-dev:create-plugin에 이 산출물을 전달해 구현을 시작할 수 있습니다. plugin-dev는 산출물을 검토하고 시작 지점을 스스로 결정합니다."

### plugin-dev Phase 5 이후: skill-creator 연결 안내

구성 확정 완료 시 사용자에게 전체 사이클을 안내한다:

```
다음 단계 안내:
1. plugin-dev:create-plugin으로 구현 시작
2. 구성요소 구현 완료될 때마다 requirements-sync가 입출력·도구·실패 경로 사후 검증
3. plugin-dev Phase 5(Skill 구현) 완료 후 → requirements-sync가 skill-creator eval 세트 준비
4. skill-creator로 각 Skill의 트리거 품질 검증
```

### 완료 조건

사용자가 확정본을 승인했을 때. `artifact-final.md` 저장, `state.md` 업데이트.
