# 작업 A~G 상세 절차

## 작업 A — 페르소나 정의

**목적:** 발화 가설과 요구사항 판단의 기준점을 만든다.

### 진행 방식

1. 사용자에게 아래 항목을 순서대로 질문한다. 한 번에 모두 묻지 않고 자연스러운 대화로 끌어낸다.

   - 이 에이전트를 주로 누가 사용하는가? (역할, 부서, 직급)
   - 그 사람의 실제 업무 방식은 어떤가? (어떤 시스템을, 어떤 순서로, 얼마나 자주 쓰는가)
   - 현재 가장 불편한 것은 무엇인가? ("시간이 걸린다"보다 한 단계 더 파고든 구조적 이유
   - AI나 자동화 도구를 현재 어떻게 활용하는가?
   - 그 사람이 실제로 쓰는 말투와 표현은 어떤가? (모호한 표현, 약어, 맥락 의존 표현 포함)

2. 답변을 종합해 페르소나 카드 초안을 작성한다.

3. 사용자에게 확인받는다.

### 페르소나 카드 형식

```markdown
# 페르소나: [이름 또는 역할명]

## 역할 및 환경
- 직책/부서:
- 주요 사용 시스템:
- 업무 루틴:

## 구조적 제약
- (기술적 문제가 아닌 업무 구조상의 제약)

## 진짜 페인포인트
- (표면 니즈 아래의 근본 이유)

## AI 활용 현황
-

## 발화 스타일 특징
- (모호한 표현, 맥락 의존, 자주 쓰는 약어 등)
```

### 완료 조건

페르소나 카드를 사용자가 확인했을 때. `persona.md`로 저장하고 작업 B로 이동.

---

## 작업 B — 발화 가설 도출

**목적:** 설계 전 Walk-through에 쓸 초기 발화 세트를 준비한다.

### 발화 유형 (4가지 필수 커버)

| 유형 | 설명 | 목적 |
|---|---|---|
| 일상 업무 | 페르소나가 매일 하는 요청 | 핵심 경로 커버 |
| 임기응변 | 회의 중 즉석 질문, 모호한 표현 포함 | 실제 사용 패턴 반영 |
| 크로스 시스템 | 여러 데이터 소스를 합쳐야 하는 경우 | 구성요소 간 연결 필요성 도출 |
| 치명적 경로 | 잘못 작동했을 때 피해가 가장 큰 경우 | 안전장치(Hook) 요구사항 도출 |

### 진행 방식

1. 페르소나 카드를 기반으로 4가지 유형별 발화 초안을 각 2~3개씩 생성한다
2. 구성요소를 먼저 생각하고 발화를 역산하지 않는다. 페르소나의 업무에서 자연스럽게 나오는 발화를 만든다
3. 사용자에게 검토 요청: "이 발화들이 실제로 그 사람이 할 말처럼 보이나요? 빠진 상황이 있나요?"
4. 수정·추가 반영 후 확정

### 발화 목록 형식 (artifact-v1.md)

```markdown
| 발화 ID | 유형 | 발화 내용 |
|--------|-----|---------|
| U01 | 일상 업무 | "3월 사업부별 매출 보여줘" |
| U02 | 임기응변 | "지난번 그거 다시 뽑아줘" |
| U03 | 크로스 시스템 | "올해 채용 인원이랑 매출 증가율 같이 보고 싶어" |
| U04 | 치명적 경로 | "이 쿼리 그냥 실행해줘" |
```

### 완료 조건

모든 유형이 커버된 발화 목록을 사용자가 확인했을 때. `artifact-v1.md`로 저장하고 작업 C로 이동.

---

## 작업 C — 설계 전 Walk-through (발화 → 요구사항)

**목적:** 각 발화를 처리하는 데 필요한 것들을 빠짐없이 도출한다.

### 진행 방식

발화를 하나씩 에이전트 시점에서 단계적으로 따라간다. 이 시점에서 구성요소 계위(Skill인지 MCP인지)는 결정하지 않는다. "무엇이 필요한가"만 도출한다.

```
[발화]: "3월 사업부별 매출 보여줘"

단계 1. 질문 의도 파악
    필요한 것: 자연어를 구조화된 데이터 요청으로 해석
    → 요구사항 R01: "자연어를 SQL로 변환할 수 있어야 한다"

단계 2. 데이터 위치 파악
    필요한 것: 어느 테이블에 있는지 탐색
    → 요구사항 R02: "DB 스키마를 참조할 수 있어야 한다"

단계 3. 조회 실행
    필요한 것: DB에 쿼리를 날리고 결과를 받아옴
    → 요구사항 R03: "DB에 읽기 전용으로 접근해야 한다"
    → 요구사항 R04: "DML은 항상 차단해야 한다"
```

각 발화 완료 후 사용자에게: "이 발화에서 이런 요구사항이 나왔습니다. 빠진 단계가 있나요?"

### 완료 조건

모든 발화의 Walk-through가 완료되고 요구사항이 도출됐을 때. `artifact-v2.md`로 저장하고 작업 D로 이동.

---

## 작업 D — 계위 판단 연동 (요구사항 → 구성요소)

**목적:** 각 요구사항을 어떤 계위의 구성요소로 구현할지 결정하고 산출물에 기록한다.

### 계위 종류

| 계위 | 의미 | 선택 기준 |
|---|---|---|
| Command | 사용자 /slash 진입점 | 사용자가 명시적으로 실행하는 워크플로우 |
| Skill | 재사용 가능한 전문 기능 | Claude가 맥락에 따라 자동 활용하는 능력 |
| references | 특정 Skill과 항상 함께 쓰이는 보조 파일 | 스키마, 정책, 문서 등 참조 자료 |
| MCP | 외부 시스템 접근 도구 | DB, API 등 외부 실행이 필요한 경우 |
| Hook | 이벤트 기반 자동 실행 | LLM 판단 없이 항상 실행되어야 하는 규칙 |
| Agent | 멀티스텝 자율 실행 주체 | 복잡한 자율 판단이 필요한 작업 |

### 진행 방식

1. plugin-dev의 `plugin-structure` 스킬을 활용해 각 요구사항에 적합한 계위를 판단한다
2. 판단 결과를 사용자에게 제시하고 근거를 설명한다
3. 구성요소명, 계위, 소속 플러그인을 산출물에 기록한다
4. 사용자 조정 의견을 반영한다

### 완료 조건

모든 요구사항에 구성요소가 매핑되고 사용자가 확인했을 때. `artifact-v3.md`로 저장하고 작업 E로 이동.

---

## 작업 E — 발화 목록 보완

**목적:** 구성요소 커버리지를 확인하고 발동 경로가 없는 구성요소를 발견한다.

### 확인 항목

- 각 Command/Skill을 발동시키는 발화가 존재하는가?
- Skill 체이닝 경로가 발화에 반영되어 있는가?
- 플러그인 간 연결 경로가 발화에 있는가?
- 실패·예외 경로를 다루는 발화가 있는가?

### 진행 방식

구성요소 목록을 순회하며 각 구성요소에 대해: "이 구성요소를 발동시키는 발화가 artifact에 있는가?"를 확인한다.

발동 경로가 없는 구성요소 발견 시 → 해당 경로를 발동시킬 발화를 artifact에 추가하고 사용자에게 알린다.

### 완료 조건

모든 구성요소에 발동 경로가 있을 때. `artifact-v4.md`로 저장하고 작업 F로 이동.

---

## 작업 F — 설계 후 Walk-through (구멍 검증)

**목적:** 완성된 구성 목록으로 각 발화를 처리할 수 있는지 검증하고 구멍을 기록한다.

### 진행 방식

`gap-analyzer` 에이전트를 호출해 자율 검증을 실행한다.

```
gap-analyzer 입력: artifact-v4.md
gap-analyzer 출력: gap-report.md
```

gap-report.md를 읽어 결과를 사용자에게 보고한다.

**심각 구멍이 있는 경우:**
- 해당 발화와 요구사항을 짚어 사용자에게 설명
- `state.md`의 `loop_history`에 기록
- `iteration` +1
- 작업 D로 복귀해 관련 항목 재작성

**심각 구멍이 없는 경우:**
- 중간/경미 구멍은 산출물에 기록 (경미는 Could 항목으로 이관)
- `artifact-v5.md` 저장
- 작업 G로 이동

### 완료 조건

모든 발화 추적이 끝나고, 각 발화가 "처리 가능" 또는 "구멍 기록"으로 끝났을 때.

---

## 작업 G — 구성 확정

**목적:** 심각 구멍이 없는 상태의 설계 산출물을 최종본으로 확정하고 plugin-dev에 인계한다.

### 진행 방식

1. artifact-v5.md를 기반으로 `artifact-final.md`를 생성한다
2. 확정본에 포함된 정보를 사용자에게 요약 제시:
   - 에이전트 목적 및 사용자
   - 구성요소 목록 (이름, 계위, 소속 플러그인)
   - 발화-요구사항-구성요소 연결 관계
   - 미해소 구멍(중간/경미) 목록
3. 사용자 최종 확인
4. `state.md`의 `current_step`을 `finalized`로 업데이트

### plugin-dev 인계

"plugin-dev:create-plugin에 이 산출물을 전달해 구현을 시작할 수 있습니다. plugin-dev는 산출물을 검토하고 시작 지점을 스스로 결정합니다."

### plugin-dev Phase 5 이후: skill-creator 연결 안내

작업 G 완료 시 사용자에게 전체 사이클을 안내한다:

```
다음 단계 안내:
1. plugin-dev:create-plugin으로 구현 시작
2. 구성요소 구현 완료될 때마다 "requirements-sync"가 ②④⑤ 사후 검증
3. plugin-dev Phase 5(Skill 구현) 완료 후 → requirements-sync가 skill-creator eval 세트 준비
4. skill-creator로 각 Skill의 트리거 품질 검증
```

이 흐름을 안내해 사용자가 전체 빌드 사이클에서 이 플러그인의 역할을 알 수 있게 한다.

### 완료 조건

사용자가 확정본을 승인했을 때. `artifact-final.md` 저장, `state.md` 업데이트.
