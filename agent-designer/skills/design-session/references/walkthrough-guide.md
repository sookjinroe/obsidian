# Walk-through 진행 방식

## 두 종류의 Walk-through

| 구분 | 시점 | 목적 | 입력 | 출력 | 담당 |
|---|---|---|---|---|---|
| 설계 전 Walk-through | 초안 생성 | 발화 → 요구사항 도출 | 발화 + 페르소나 | 요구사항 목록 | design-drafter |
| 설계 후 Walk-through | gap 검증 | gap 발견 | artifact.md (구성요소 컬럼 포함) | gap 보고서 | gap-analyzer |

---

## 설계 전 Walk-through (design-drafter 수행)

에이전트 시점에서 발화를 받았다고 가정하고, 그것을 처리하기 위해 필요한 것을 단계별로 나열한다.

### 진행 원칙

- **에이전트 시점으로 전환한다.** "이 에이전트가 이 발화를 받으면 무엇을 해야 하는가?"
- **단계를 쪼갠다.** 하나의 발화에서 여러 단계가 나올 수 있다. 각 단계마다 필요한 것을 요구사항으로 변환한다.
- **구성요소 계위를 결정하지 않는다.** "Skill이 필요하다"가 아니라 "자연어를 SQL로 변환할 수 있어야 한다"로 표현한다. 계위 결정은 plugin-dev:plugin-structure가 담당한다.
- **치명적 경로 발화는 실패 시나리오도 함께 다룬다.** "이 쿼리 실행해줘"에서 "쿼리가 DML일 때 어떻게 되는가"까지 추적한다.

### 단계별 질문 패턴

각 단계에서 스스로 묻는 질문:

1. **의도 파악**: 이 발화에서 사용자가 원하는 것이 정확히 무엇인가?
2. **데이터/정보 위치**: 필요한 정보가 어디에 있는가? 어떻게 접근하는가?
3. **처리**: 정보를 가져왔으면 무엇을 해야 하는가? (계산, 변환, 필터 등)
4. **안전**: 이 처리에서 잘못될 수 있는 것은 무엇인가? 막아야 하는 것은?
5. **반환**: 결과를 어떤 형식으로 돌려줘야 하는가?

### 예시

```
[발화 U03]: "올해 채용 인원이랑 매출 증가율 같이 보고 싶어"

단계 1. 의도 파악
  필요한 것: 두 개의 다른 지표를 함께 보여달라는 복합 요청 처리
  → R07: "복수 지표 요청을 파싱할 수 있어야 한다"

단계 2. 데이터 위치 파악 — 지표 1
  필요한 것: 채용 인원 데이터는 HR 시스템에 있음
  → R08: "HR 시스템 데이터에 접근할 수 있어야 한다"

단계 3. 데이터 위치 파악 — 지표 2
  필요한 것: 매출 데이터는 ERP 시스템에 있음
  → R09: "ERP 매출 데이터에 접근할 수 있어야 한다"

단계 4. 통합
  필요한 것: 두 시스템의 데이터를 같은 시간축(월별)으로 맞춰 표시
  → R10: "서로 다른 시스템 데이터를 동일 기간 단위로 정렬할 수 있어야 한다"

단계 5. 안전
  필요한 것: HR 시스템 접근 시 개인정보 항목은 집계 수준만 노출
  → R11: "개인 식별 정보는 집계 결과만 반환해야 한다"
```

## 설계 후 Walk-through (gap-analyzer 수행)

plugin-dev:plugin-structure가 구성요소 컬럼을 추가한 후, 각 발화가 실제로 구성요소 목록으로 처리 가능한지 검증한다. 이 Walk-through는 `gap-analyzer` 에이전트가 자율 실행한다.

### gap-analyzer가 적용하는 기준

**① 연결 없음**: 발화의 처리 경로에서 요구사항에 매핑된 구성요소가 artifact.md에 존재하지 않는 경우

**③ 트리거 미정의**: Skill 또는 Command가 artifact.md에 존재하지만, 그것을 발동시키는 발화 경로가 정의되지 않은 경우

> **참고:** ②입출력 불일치, ④도구 미연결, ⑤실패 경로 미처리는 구성요소 구현 이후 `requirements-sync`가 담당한다.

### 설계 후 Walk-through 단계

```
1. artifact.md에서 발화 목록과 구성요소 목록을 읽는다
2. 각 발화에 대해:
   a. 처리 경로를 재구성 (artifact.md의 요구사항 체인 기반)
   b. 각 요구사항에 매핑된 구성요소가 artifact.md에 있는지 확인 → ① 기준
   c. 구성요소 목록의 각 Skill/Command에 이 발화가 발동 경로를 제공하는지 확인 → ③ 기준
   d. 위반 발견 시 gap으로 기록 + 심각도 분류
3. 모든 발화 완료 후 보고서 생성
```

### 심각도 판정 기준

상세 기준은 `references/gap-criteria.md`를 참조한다.
