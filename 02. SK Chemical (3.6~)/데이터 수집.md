

## Jira에서 꺼낼 수 있는 것

**수집 방법:** Jira REST API (`/rest/agile/1.0`, `/rest/api/3`) + DevLake Jira 커넥터 (폴링 or webhook)

**꺼낼 수 있는 raw data:**
- 이슈 생성/완료/상태변경 타임스탬프
- 스프린트 계획 vs 실제 완료 이슈 수
- 이슈별 assignee, story point, 우선순위
- epic/story/task 계층 구조
- 이슈 체류 시간 (상태별)
- 댓글, 첨부파일 이력

**여기서 계산 가능한 것:**
```
- WBS 대비 진척률: 완료 이슈 / 전체 이슈 × 100
- 스프린트 완료율: 스프린트 내 done 이슈 / 계획 이슈
- 이슈 체류 시간: "In Progress" 평균 일수 → 병목 감지
- 미완료 이슈 예측 납기: 현재 속도(velocity) 기준 남은 이슈 소화 예상일
- 이슈 재오픈율: 완료 → 재오픈된 이슈 비율 (품질 간접 지표)
```

**SK케미칼 표준 임계치 적용 지점:**
진척률 기준 (예: 납품 D-30에 70% 이상)을 표준에서 가져와서 pass/fail 판단. Jira 자체는 이 기준을 모름.


## Bitbucket에서 꺼낼 수 있는 것

**수집 방법:** Bitbucket REST API (`/rest/api/1.0`) + DevLake Bitbucket 커넥터

**꺼낼 수 있는 raw data:**
- PR 생성/승인/머지/거절 타임스탬프 + reviewer 목록
- PR당 코멘트 수, 코멘트 해결 여부
- 커밋 수, 커밋 메시지, 변경 파일 수/라인 수
- 브랜치명, 브랜치 생성/삭제 이력
- PR → 머지까지 소요 시간

**여기서 계산 가능한 것:**
```
- 코드리뷰 완료율: approve 받은 PR / 전체 PR
- 리뷰 없이 머지된 PR 수: approve 없이 머지 → 표준 위반 감지
- PR 평균 리뷰 대기 시간: 생성 → 첫 코멘트까지
- 브랜치 전략 준수율: 브랜치명 패턴 체크 (feature/, hotfix/ 등 표준 패턴 매칭)
- 코드 변경 집중도: 특정 파일/모듈에 변경 집중 여부
- PR 크기 분포: 라인 수 기준 → 너무 큰 PR은 리뷰 품질 저하 신호
```

**SK케미칼 표준 임계치 적용 지점:**
"모든 PR은 1인 이상 리뷰 후 머지" 같은 규칙이 표준에 있으면, 위반 PR을 자동으로 잡아낼 수 있음. 지금은 아무도 보지 않음.


## Sparrow에서 꺼낼 수 있는 것

**수집 방법:** Sparrow REST API 또는 분석 결과 JSON/XML export → 파싱 파이프라인

**꺼낼 수 있는 raw data:**
- 취약점 목록 (파일경로, 라인, 규칙ID, 심각도: 고/중/저)
- 분석 실행 이력 (언제 돌렸는지)
- 취약점 상태 (신규/재발/해결)
- 카테고리 (SQL Injection, XSS, 버퍼오버플로우 등)

**여기서 계산 가능한 것:**
```
- 심각도별 취약점 건수 추이 (스프린트/주 단위)
- 취약점 해소율: 이전 분석 대비 감소 건수
- 재발율: 해결 후 재등장한 취약점
- 분석 미실행 감지: N일 이상 Sparrow 미실행 → 프로세스 미준수
- KISA 기준 항목별 달성률 (카테고리 매핑 가능)
```

**SK케미칼 표준 임계치 적용 지점:**
"고위험 0건, 중위험 5건 이하" 같은 기준이 1차 표준에 있으면, 현재 건수와 비교해서 pass/fail + 잔여 해소 필요 건수를 바로 계산 가능.

## Confluence에서 꺼낼 수 있는 것

**수집 방법:** Confluence REST API (`/rest/api/content`) + space/label 구조 활용 + webhook (페이지 생성/수정 이벤트)

**꺼낼 수 있는 raw data:**
- 페이지 생성/수정 타임스탬프 + 작성자
- 페이지 제목, 라벨, 공간(space) 위치
- 첨부파일 존재 여부 및 파일명

**여기서 계산 가능한 것:**
```
- 산출물 제출 여부: 표준에서 정의한 산출물 목록 × 프로젝트별
  → "요구사항 정의서" 라벨 페이지가 있는가 없는가
- 제출 기한 준수율: 페이지 생성일 vs WBS 상 제출 예정일 비교
- 산출물 내용 검수 (AI): 페이지 본문 추출 → LLM으로 표준 체크리스트 대조
  → "요구사항 정의서에 비기능 요구사항 항목이 누락됨" 수준까지 가능
```

**산출물 AI 검수 구체적 방법:**
```
1. Confluence 페이지 본문 추출 (API)
2. 해당 산출물 유형의 표준 체크리스트 로드 (1차 과제 산출물)
3. LLM 프롬프트:
   "다음은 [요구사항 정의서] 표준 체크리스트입니다.
    제출된 문서와 대조하여 각 항목의 충족 여부를 판단하세요."
4. 결과: 항목별 pass/fail/불명확 + 근거 문장
```


| 현재 PM이 하는 것            | 포탈이 하는 것                          | 기술 방법                          |
| ---------------------- | --------------------------------- | ------------------------------ |
| Sparrow 접속해서 취약점 확인    | 고위험 N건, 기준 대비 +N건 자동 표시           | Sparrow API → DevLake → 임계치 비교 |
| Bitbucket에서 PR 목록 훑기   | 리뷰 없이 머지된 PR X건 자동 플래그            | Bitbucket API → 브랜치 규칙 대조      |
| Jira에서 진척률 수동 계산       | WBS 대비 진척률 + 현재 속도 기준 예상 납기 자동 계산 | Jira API → velocity 계산         |
| Confluence 들어가서 산출물 확인 | 미제출 산출물 목록 + 기한 초과 여부 자동 표시       | Confluence API + WBS 일정 대조     |
| 제출된 산출물 직접 읽고 판단       | AI가 체크리스트 대조 후 항목별 결과 제공          | LLM + 1차 표준 체크리스트              |
| 주간 보고용 현황 수동 정리        | 트리거 시 자동 리포트 생성 (ambient 에이전트)    | 집계 DB → LLM 요약                 |
