
**제품 소개** Pull/Push/Action은 사용자가 시스템과 상호작용하는 세 가지 방식을 구분한 것입니다. Pull은 사용자가 능동적으로 조회하는 것, Push는 시스템이 먼저 전달하는 것, Action은 에이전트에게 작업을 요청하는 것입니다. 세 가지 모두 동일한 데이터 파이프라인 위에서 작동하며, 같은 데이터를 다른 방식으로 소비하는 구조입니다. 예를 들어 취약점 급증이라는 이벤트는 Pull에서는 코드 품질 화면에 수치로 표시되고, Push에서는 알림으로 전달되고, Action에서는 "이번 스프린트 리스크 요약해줘"라는 질의의 응답 재료가 됩니다.

---

**How It Works** Rule-based와 LLM 두 엔진을 분리한 이유는 역할이 다르기 때문입니다. 취약점 건수나 지연 일수처럼 정답이 있는 계산은 Rule-based가 처리하고, 문서 내용 해석이나 자연어 응답처럼 맥락 판단이 필요한 것은 LLM이 처리합니다. LLM을 모든 판단에 쓰지 않는 건 비용과 응답 속도 때문입니다. 실시간성이 중요한 이벤트(취약점 발생, PR 머지)는 Webhook으로 수신하고, 정기 집계가 필요한 데이터(커버리지, 진척률)는 스케줄 폴링으로 수집할 수 있습니다.

---

**Pull: 통합현황** 4개 영역의 상태는 각 영역에서 정의된 임계치를 초과했는지 여부로 자동 판단됩니다. 임계치는 도입 시 설정하며 프로젝트마다 다르게 지정할 수 있습니다. AI 요약은 단일 영역 이상이 아니라 영역 간 패턴을 보는 것이 목적입니다. 예시에서 프로세스 위반과 취약점 증가 시점이 일치한다고 짚은 것이 그 예입니다. 이 요약은 매 조회 시 실시간으로 생성되는 것이 아니라, 데이터 변경이 감지될 때 갱신되는 방식으로 LLM 호출 비용을 줄입니다.

---

**Pull: 코드 품질** Sparrow는 정적 분석 도구로, 스캔 결과를 API로 수집합니다. 커버리지는 CI/CD 파이프라인에서 테스트 실행 시 자동 수집됩니다. 두 데이터 모두 개발자가 별도로 입력하지 않아도 수집됩니다. 고위험·중위험 구분은 Sparrow의 심각도 분류를 그대로 사용합니다. 취약점 추이 차트는 스프린트 단위로 집계되며, 특정 스프린트에서 수치가 증가했는지 감소했는지 방향성을 보는 데 주로 활용됩니다. OSS 취약점은 오픈소스 라이브러리의 알려진 CVE를 체크하는 항목으로, Sparrow의 별도 스캔 결과에서 수집됩니다.

---

**Pull: 팀 프로세스** 리뷰어 수와 브랜치명 패턴은 Bitbucket API로 확인할 수 있는 메타데이터입니다. PR 본문이나 코드 내용을 분석하는 것이 아니라, 머지 전 리뷰가 몇 명에게 승인됐는지, 브랜치명이 규칙을 따르는지만 체크합니다. 리뷰어 수 기준과 브랜치 네이밍 규칙은 도입 시 설정합니다. 표에 표시되는 PR 목록은 위반 또는 미달로 분류된 건만 보여주며, 정상 처리된 PR은 집계 수치에만 반영됩니다. PR 머지 시점은 Bitbucket Webhook으로 실시간 감지됩니다.

---

**Pull: 일정·진척** 진척률은 Jira에서 완료 처리된 이슈 수를 전체 이슈 수로 나눈 값입니다. 이슈의 난이도나 크기는 반영되지 않으며, 건수 기준으로 계산합니다. WBS 대비 계획 진척률은 Jira의 마일스톤 또는 픽스 버전 설정을 기준으로 합니다. 예상 지연 일수는 최근 N개 스프린트의 평균 완료율을 현재 잔여 이슈에 적용해 계산하며, 현재 속도가 유지된다는 가정 하의 수치입니다. 병목 이슈는 In Progress 상태로 설정된 기간이 임계치를 초과한 이슈를 자동으로 감지합니다. 체류 기간 임계치는 설정에서 조정할 수 있습니다.

---

**Push: 알림** 알림 트리거는 두 종류입니다. 이벤트 트리거는 취약점 신규 발생, PR 머지처럼 실시간으로 감지되는 것이고, 일정 트리거는 납기 D-7처럼 날짜 기반으로 발송되는 것입니다. 알림 수신 대상과 채널(Slack, 이메일 등)은 프로젝트별로 설정합니다. 동일한 이벤트가 반복 발생할 경우 중복 알림을 방지하는 디바운스 설정이 있습니다. 예시의 알림 4건은 유형별로 색상을 구분했는데, 긴급·주의·안내 세 단계로 나뉩니다. 수신자는 단계별로 다르게 설정할 수 있습니다.

---

**Push: 자동 리포트** 리포트는 트리거 시점의 스냅샷입니다. 사람이 작성하는 것이 아니라 그 시점의 Pull 데이터를 LLM이 문장으로 변환하는 구조입니다. 권고 사항도 데이터에서 자동 도출됩니다. 리포트 수신 대상은 알림과 별도로 설정할 수 있으며, 일반적으로 알림보다 넓은 범위로 설정합니다. 리포트 형식은 HTML 이메일과 PDF 두 가지로 제공됩니다. 트리거 조건(D-7, 스프린트 종료 등)은 도입 시 프로젝트별로 설정하며 이후 변경 가능합니다.

---

**Push: 교차검증** 보고서가 시스템에 업로드되면 자동으로 트리거됩니다. 별도 요청 없이 작동하고, 불일치가 없으면 아무것도 전달되지 않습니다. 비교 대상 항목은 설정에서 커스터마이징할 수 있습니다. 현재 비교 가능한 항목은 진척률, 완료 이슈 수, 취약점 해소 건수처럼 수치로 표현되는 항목입니다. 보고서에서 수치를 파싱하는 방식은 LLM이 문서를 읽고 항목을 추출하는 방식으로 작동하기 때문에, 정해진 양식이 아니어도 대부분의 형식을 처리할 수 있습니다.

---

**Action: 보고서 초안 생성** 에이전트가 자동으로 채울 수 있는 항목과 사람이 직접 입력해야 하는 항목을 구분해서 보여줍니다. 자동 입력 항목은 데이터 소스가 명시되어 있어 수치의 출처를 확인할 수 있습니다. 초안이 생성된 이후에는 일반 문서처럼 편집할 수 있으며, 수정 이력은 별도 저장됩니다. 현재 지원하는 보고서 형식은 주간보고서이며, 추가 형식은 템플릿 등록으로 확장할 수 있습니다.

---

**Action: 보고서 교차검증** Push의 교차검증과 기능은 동일하지만 트리거 방식이 다릅니다. Push는 보고서 제출 시 자동 실행이고, Action은 사용자가 원하는 시점에 직접 요청하는 것입니다. 제출 전 사전 확인 용도로 활용하거나, 특정 항목만 선택해서 검증하는 것도 가능합니다. 두 기능이 같은 엔진 위에서 작동하기 때문에 결과 형식도 동일합니다.

---

**Action: 자연어 질의** 에이전트가 응답할 때 참조한 데이터 소스를 함께 반환합니다. 응답 내용의 근거를 확인할 수 있습니다. 현재 연결된 소스 범위 내에서만 응답하며, 범위를 벗어난 질의에는 답하지 않습니다. 질의 이력은 저장되며, 동일한 질의를 반복할 경우 캐싱된 결과를 반환할 수 있습니다. 응답 언어는 질의 언어를 따릅니다.

---

**마무리** On-Premise 배포는 고객사 내부 네트워크 안에서 모든 데이터가 처리되는 방식입니다. LLM도 내부 서버에 설치된 모델을 사용하거나, 외부 API를 사용하되 데이터가 학습에 활용되지 않는 계약 조건을 적용할 수 있습니다. 연동 도구 중 일부만 사용하는 경우에도 도입 가능하며, 연동되지 않은 영역의 기능은 비활성화됩니다. 도입 기간은 연동 환경과 커스터마이징 범위에 따라 달라지며, 초기 협의 시 범위를 구체화합니다.
