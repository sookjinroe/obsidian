## 제안 프레임

**1차 프로젝트가 만든 것:**
표준 문서들 (프로세스, 산출물 양식, 개발 규칙 등)

**2차 프로젝트가 하는 것:**
> 그 표준들을 **Context Framework에 적재**해서, 에이전트들이 SK케미칼 표준대로 자동으로 움직이게 만드는 것

---

## 제안 구조 (안)

**Layer 1. 표준의 디지털화**
SK케미칼 1차 표준 → Context Framework 적재
- 개발 프로세스 룰셋
- 산출물 양식/기준
- 네이밍/아키텍처 컨벤션
- 협력업체 평가 기준

**Layer 2. 에이전트별 역할**

| 시점 | 에이전트 | SK케미칼 페인포인트 해소 |
|---|---|---|
| 프로젝트 시작 | Plan Agent (Analyst→Architect→Test Designer) | 산출물 초안 자동 생성 → 기간 내 불가능 문제 해소 |
| 개발 중 | Code Agent (표준 컨텍스트 참조) | 협력업체가 표준대로 코딩하도록 실시간 가이드 |
| 커밋/PR 시 | Test Agent | 표준 준수 자동 검토 → PM 리소스 문제 해소 |
| 납품 시 | Code Agent (역방향 역할) | 코드 → 설계문서 역생성, 정합성 확인 |
**Plan Agent** → 순방향 (기획/설계 → 산출물)
- RFP나 요구사항이 들어오면 SK케미칼 표준 양식대로 산출물 초안 자동 생성
- 산출물 분량 문제를 정면으로 해결하는 지점

**Code Agent + Test Agent** → 개발 중 (코드 작성 → 표준 준수 체크)
- 협력업체 개발자가 코드 커밋할 때마다 SK케미칼 표준 자동 검토
- PM이 하나하나 확인 안 해도 됨

**역방향 문서화** (코드 → 설계/요구사항 재구성)
- 납품된 코드가 설계서와 실제로 맞는지 확인
- 협력업체 교체 시 인수인계 문제도 여기서 해소

**Layer 3. 운영 지속성**

- 표준이 바뀌면 Framework만 업데이트 → 모든 에이전트에 즉시 반영
- 즉, **표준의 개선도 AI 운영 안에서 흡수 가능**
