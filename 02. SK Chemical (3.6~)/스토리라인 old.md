## 배경

### 문제 서술

1. **SW 개발 프로젝트의 구조적 복잡성** SK케미칼의 개발 프로젝트는 수행사가 개발을 담당하고 PM이 관리하는 구조로 운영된다. PM 1명이 다수의 프로젝트를 병행하는 환경에서 품질을 일관되게 관리하려면 명확한 기준과 체계가 필요하다.
    
2. **1차: SDLC 표준 수립** 단계별 체크리스트, 산출물 패키지, 품질 임계치를 정의했다. 착수 → 설계 → 개발 → 테스트/배포 → 운영 각 단계에 게이트가 생겼다.
    
3. **표준 운영의 현실적 과제** 표준이 정의되면서 각 단계에서 확인하고 관리해야 할 항목이 구체화됐다. 이를 수동으로 운영하면 PM의 관리 부담이 커지고, 표준이 실제로 지켜지는지 확인하기 어렵다.

> **2차 과제 목표:** 1차에서 정의한 표준을 시스템 안에 내재화한다. 사람이 직접 챙기지 않아도 표준이 작동하게 한다.

## 제안 요약

**품질 포탈** Jira·Bitbucket·Sparrow·Confluence에 분산된 데이터를 한 곳에서 수집하고, SK케미칼 표준 기준으로 해석해서 PM에게 전달한다.

---

### PULL — 대시보드 (PM이 보고 싶을 때)

게이트 리뷰, 주간 보고, 이슈 판단 시 — 여러 도구를 열지 않고 한 화면에서 전체 프로젝트 현황과 기준 대비 상태를 확인한다.

포함 영역:

- 산출물 품질
- 코드 품질
- 프로세스 품질
- 일정 품질

---

### PUSH — 에이전트 (시스템이 먼저 알릴 때)

임계치 초과, 산출물 미제출, 납기 임박 — 이벤트가 발생하면 PM이 능동적으로 들어가지 않아도 시스템이 먼저 리포트를 전달한다.

포함 기능:

- 이벤트 트리거
- 일정 트리거
- 산출물 AI 검수 결과

---

### 데이터 흐름

```
Jira · Bitbucket · Sparrow · Confluence
        ↓ API
      DevLake
        ↓
      품질 포탈
        ↓
        PM
```

## Layer 1 — 통합

흩어진 도구를 하나의 기준으로 연결한다.

|도구|수집 데이터|
|---|---|
|Jira|이슈 상태, 스프린트 진척, 체류 시간|
|Bitbucket|PR 승인 이력, 리뷰어, 커밋 내역|
|Sparrow|취약점 목록, 심각도, 분석 이력|
|Confluence|산출물 페이지, 작성일, 첨부파일|

> 각 도구는 데이터를 가지고 있지만 기준이 없다. SK케미칼 표준 임계치가 포탈 안에 있기 때문에 같은 데이터가 **판단 가능한 지표**가 된다.

## Layer 3 — AI 보조

숫자로 표현할 수 없는 것 — 산출물 내용 검수

---

### 처리 흐름

1. **제출 감지** 수행사가 Confluence에 산출물 업로드 → 비기능 요구사항 정의서 — 방금 업로드됨
    
2. **AI 체크리스트 대조** 1차에서 정의한 표준 체크리스트와 문서 내용 자동 대조 → 9개 항목 검토 중…
    
3. **결과 생성** 항목별 충족/미충족/불명확 판단 + 근거 문장 첨부 → 충족 5 · 미충족 3 · 불명확 1
    
4. **PM 컨펌** AI 판단 초안을 PM이 검토 후 승인 또는 반려 → 2분 소요 (직접 읽을 경우 평균 25분)
