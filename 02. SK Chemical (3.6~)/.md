# SK케미칼 AI 개발 지원 솔루션 제안 — 논리 기반 종합 정리

---

## 1. 배경 및 문제 구조

### 고객사 기본 구조

SK케미칼은 사내 소프트웨어 개발 수요가 높지만 내부 개발 인력이 없다. 모든 개발은 외부 협력업체를 통해 수행되며, SK케미칼 PM이 이를 관리하는 구조다. 협력업체는 프로젝트마다 달라질 수 있고, SK케미칼 PM과 협력업체 PM이 각각 존재하며 R&R은 프로젝트마다 다르다.

이 구조에서 1차 프로젝트(IT/DT 기술문서 표준화 컨설팅)가 완료됐다. 개발 프로세스 표준화, 단계별 산출물 표준화가 완성된 시점이다.

### 문제의 4개 층위

**층위 1 — 구조적 문제: 정보 비대칭**

개발을 수행하는 주체(협력업체)와 기준을 만든 주체(SK케미칼)가 분리돼 있다. 표준은 결국 협력업체에게 요구하는 기준인데, 협력업체 입장에서는 프로젝트마다 낯선 표준을 습득하고 따르는 비용이 발생한다. SK케미칼 입장에서는 잘 따르고 있는지 확인할 수단이 마땅치 않다. 표준 문서가 아무리 잘 만들어져 있어도 이 구조 자체가 해결되지 않으면 표준은 유명무실해진다.

**층위 2 — 실행 문제: 산출물 분량 vs 현실**

워터폴 방식은 산출물이 많다. 요구사항정의서, 화면설계서, API설계서, DB설계서, 테스트 시나리오, 결과보고서 등 모두 표준대로 갖춰야 하는데, 실제 프로젝트는 기간도 짧고 인원도 적다. 현장 피드백으로 나온 "기간 내에 불가능에 가깝다"는 말이 이 gap을 직접 표현한다. 표준을 지키려면 프로젝트를 거절해야 하고, 프로젝트를 하려면 표준을 어길 수밖에 없는 딜레마 구조다.

**층위 3 — 운영 문제: PM 리소스**

표준이 잘 적용되고 있는지 확인하려면 누군가 챙겨야 한다. PM은 단일 프로젝트만 관리하는 게 아니라 여러 협력업체, 여러 프로젝트를 동시에 관리한다. 각 프로젝트의 산출물을 하나하나 검토하는 건 현실적으로 불가능하다. 이는 개발 지식의 부재가 아니라 순수한 리소스(시간) 문제다.

**층위 4 — 지속성 문제: 표준의 노화**

표준은 한 번 만들면 끝이 아니다. 실제 프로젝트에 적용하면서 맞지 않는 부분이 발견되고 개선이 필요하다. 그런데 표준을 개선하면 재전파, 재교육, 기존 산출물 보완이 따라온다. 이 루프가 무거우면 표준은 초기 버전에서 멈추고 시간이 지날수록 현실과 괴리가 커진다.

### 기존 접근(컨설팅)의 본질적 한계

1차 프로젝트는 "무엇을 해야 하는지"를 정의했다. 필수적이고 잘 완성된 작업이다. 그러나 컨설팅의 본질적 한계는, 표준의 실행이 전적으로 사람의 의지, 역량, 시간에 달려 있다는 것이다. 위 4개 층위의 문제가 모두 "사람이 해야 한다"는 전제 위에 있다. 이것은 의지의 문제가 아니라 구조의 문제다.

## 2. 제안의 핵심 전환

**표준의 실행 주체를 사람에서 시스템으로 바꾸는 것**

이 전환을 가능하게 하는 두 가지 메커니즘이 있다.

---

## 3. 메커니즘 1 — Context Framework: 표준의 코드화

### 개념

Context Framework는 에이전트가 작업을 수행할 때 참조하는 모든 것의 집합이다. Rule, Skill, Prompt, MCP, 워크플로우 등이 포함된다. Tabnine의 Enterprise Context Engine과 같은 역할을 한다.

1차 프로젝트에서 만들어진 표준 산출물들이 이 Framework에 적재된다. 구체적인 변환 내용은 다음과 같다.

| 1차 표준 산출물 | Context Framework 내 역할 |
|---|---|
| 개발 프로세스 표준 | 에이전트 워크플로우 순서 및 조건 정의 |
| 산출물 양식/템플릿 | Plan Agent, Tech Writer의 생성 포맷 기준 |
| 네이밍/아키텍처 규칙 | Instruction/Rule셋 — 코드 생성 시 실시간 참조 |
| 검수 기준 | Code Reviewer 체크리스트 |
| 협력업체 평가 기준 | 리뷰 및 리포팅 기준 |

### 이 변환이 단순 입력 작업이 아닌 이유

표준 문서에는 의도와 맥락이 있다. 특정 규칙이 왜 만들어졌는지를 모르면 에이전트에 잘못 적재된다. 예를 들어 특정 네이밍 규칙이 레거시 시스템 연동 때문에 생긴 거라면, 그 맥락이 Rule에 반영돼야 에이전트가 예외 상황을 올바르게 처리한다.

**우리가 1차 프로젝트를 수행했다는 것이 결정적 차별점이다.** 다른 AI 솔루션 벤더는 표준 문서를 받아서 파악부터 시작해야 한다. 우리는 그 표준을 만든 팀이다. 표준의 의도와 현장 맥락을 이미 알고 있다. Context Framework 초기 구성의 품질이 근본적으로 다를 수밖에 없다.

### Framework 업데이트 구조

표준이 개선되면 Framework만 업데이트하면 된다. 이 업데이트는 연결된 모든 에이전트에 즉시 반영된다. 별도 전파, 재교육, 공지 작업이 없다. 표준의 노화 문제(층위 4)가 구조적으로 해소된다.

## 4. 메커니즘 2 — 개발 주기 전 구간 에이전트 배치

표준이 잘 지켜지려면 사후 검토가 아니라 각 작업 시점에 개입해야 한다. 에이전트들이 SDLC의 각 단계에 배치되는 구조다. R&R은 프로젝트마다 다르기 때문에 특정 역할자에 종속되지 않고, 개발 주기의 시점 기준으로 에이전트가 동작한다.

### [분석/기획 단계] — Plan Agent: Analyst

- 입력: RFP, 고객 요구사항, 시스템 환경 구성
- 수행: 요구사항 분류, 작업 분해, 인력/공수 산정
- 출력(초안): WBS, 요구사항정의서, 유스케이스 시나리오

### [설계 단계] — Plan Agent: Architect + Test Designer

- 입력: 요구사항정의서, 유스케이스 시나리오
- Architect 수행: 화면 설계, 데이터 설계, 인터페이스 설계, 개발표준 정의, Backlog 생성
- Architect 출력(초안): 화면설계서, API설계서, DB설계서, Backlog, 개발표준 정의서, 아키텍처 다이어그램
- Test Designer 수행: 분석/설계 산출물 기반 테스트 문서 생성
- Test Designer 출력(초안): 테스트 시나리오, API 테스트 설계서

**산출물 분량 문제에 대한 답:** "표준을 낮추자"도 아니고 "프로젝트를 거절하자"도 아니다. 산출물 생성 비용 자체를 낮추는 것이다. Plan Agent가 표준 양식 기반의 초안을 자동 생성하고, 담당자는 초안을 검토하고 보완한다. "처음부터 만들기"가 "초안 보완하기"로 바뀐다.

### [개발 단계] — Code Agent

- 협력업체 개발자의 개발 환경(IDE 플러그인, CLI 등)에 맞춰 제공
- SK케미칼 Context Framework를 참조하면서 코딩 지원
- 네이밍 규칙, 아키텍처 패턴, 보안 정책, 코딩 컨벤션이 코딩 순간에 자동 개입
- 표준 준수가 "노력해야 하는 것"에서 "기본값"이 된다
- 협력업체 개발자가 프로젝트마다 표준을 새로 숙지해야 하는 온보딩 비용이 구조적으로 감소

### [커밋/PR 단계] — Test Agent ①

**트리거:** Git 서버 코드 변경 감지 (API/Webhook)

**내부 동작 흐름:**
1. Listener: 코드, 프로젝트 정보 수신
2. Context Analyzer: 관련 코드 및 테스트코드 유무 파악
3. Coder: 테스트 코드 생성/수정 → Tester: 테스트 수행 → 결과 피드백 루프
4. Code Reviewer: 코드 리뷰 + 정적 분석 (표준 준수 여부, 보안 취약점 등)
5. Reporter: GitLab 코멘트, Merge Request 생성

**출력:** 테스트코드 Merge Request / 코드 분석 결과 및 수정안 GitLab 코멘트

PM이 확인해야 하는 것이 "전체 코드"가 아니라 "에이전트가 플래그한 이슈"로 압축된다. PM의 리소스 문제(층위 3)가 구조적으로 해소된다. 문제가 납품 시점이 아니라 개발 중에 발견된다.

### [빌드/테스트 단계] — Test Agent ②

**트리거:** Scheduled Build/Test 실패 로그

**내부 동작 흐름:**
1. Listener: 로그 수신
2. Log Analyzer: 빌드/테스트 결과 로그 분석
3. Coder: 에러 수정 방안 도출
4. Code Reviewer: 정적 분석 병행
5. Reporter: GitLab/Jira Issue 생성 (테스트 결과 분석 및 수정안 포함)

개발자가 로그를 수동으로 분석하는 시간이 줄어든다. Jira 이슈가 자동 생성되어 추적 가능한 형태로 관리된다.

### [납품/검수 단계] — Tech Writer (역방향)

- 납품된 코드 → API 명세서, ERD, 시퀀스 다이어그램 역생성
- 초기 합의된 설계서 대비 실제 구현 gap을 시각적으로 파악 가능
- PM이 코드를 직접 읽지 않아도 "설계 대비 구현 현황" 확인 가능

**이 기능이 해결하는 두 가지 문제:**

첫째, 납품 검수 문제. 설계서가 있고 코드가 있는데 둘이 실제로 일치하는지는 코드를 읽어봐야 알았다. Tech Writer 역방향으로 이 gap이 문서로 드러난다.

둘째, 협력업체 교체 시 인수인계 문제. 코드는 있는데 최신 설계문서가 없거나 코드와 맞지 않는 경우, 새 협력업체가 파악하는 데 비용이 발생했다. 역생성 산출물이 자동으로 인수인계 문서가 된다.

## 5. 성과 측정 — 품질 포탈 연동

별도의 성과 지표를 만들 필요가 없다. 기존에 운영 중인 품질 포탈의 수치가 AI 도입 전후를 비교하는 성과 측정 도구가 된다.

| 품질 포탈 연동 항목 | AI 도입 전 | AI 도입 후 |
|---|---|---|
| Sparrow (정적분석) | 납품 후 문제 발견 | Test Agent ①이 커밋 시점에 사전 차단 → 지표 개선 |
| Unit Test / 코드 커버리지 | 개발자 수동 작성에 의존 | Test Agent ①이 자동 생성 및 보완 → 커버리지 수치 향상 |
| Jira 이슈 | 수동 등록, 분석 지연 | Test Agent ②가 자동 생성 및 수정안 첨부 → 해결 속도 단축 |
| Bitbucket/GitLab | PR 리뷰 사이클 수동 | 에이전트 1차 리뷰 → 리뷰 사이클 단축 |
| OSS 취약점 | 별도 점검 | Code Reviewer에 보안 룰셋 포함 → 개발 중 탐지 |

## 6. 표준이 진화하는 3단계 메커니즘

**1단계 — 구축 및 초기 적재**

1차 표준 산출물을 Context Framework에 적재한다. 단순 입력이 아니라 표준의 의도와 맥락을 반영한 구조화 작업이다. 우리가 1차를 수행했기 때문에 이 품질을 보장할 수 있다.

**2단계 — 유지보수 기간 (우리가 관리)**

실제 프로젝트에 에이전트가 투입되면서 표준과 현실의 gap이 데이터로 쌓인다. Test Agent가 반복적으로 플래그하는 패턴, Plan Agent 초안에서 자주 수정되는 부분, Code Reviewer가 자주 지적하는 항목이 표준 개선의 근거가 된다. 이를 기반으로 Context Framework를 업데이트한다. 업데이트는 Framework에만 반영하면 모든 에이전트에 즉시 적용된다.

**3단계 — 자립 운영 (가이드 및 교육)**

SK케미칼 내부에서 Framework를 직접 관리할 수 있도록 가이드와 교육을 제공한다. 새로운 규칙 추가 방법, 에이전트 동작에 미치는 영향 등을 전수한다.

이 구조의 의미는 표준이 "프로젝트 종료 후 서랍에 들어가는 문서"가 아니라 "실제 운영 데이터를 기반으로 계속 정교해지는 시스템"이 된다는 것이다.

## 7. 배포 및 보안 구조

SK케미칼은 금융권처럼 강제된 보안 요건이 없다. 데이터 민감도, 사용 규모, 사용 빈도에 따라 최적 구성을 선택할 수 있다. 이를 "유연한 배포 아키텍처"로 포지셔닝한다.

| 구성 방식 | 적합한 상황 |
|---|---|
| 고객사 온프레미스 | 내부 코드의 외부 노출을 원천 차단해야 하는 경우 |
| Private VPC | 클라우드 활용하되 데이터 격리가 필요한 경우 |
| 퍼블릭 LLM 기업용 API | 보안 요건이 낮고 빠른 도입이 우선인 경우 |

구체적인 배포 구성은 진단 협의를 통해 결정한다. 협력업체 개발자가 SK케미칼 Context Framework를 참조하는 구조에서, 내부 표준 및 코드가 외부로 유출되지 않는 설계는 배포 방식과 무관하게 보장된다.

---

## 8. 지속 운영의 자연스러운 동기

도입 저항에 대한 별도 전략은 제안 범위 밖이지만, 경험적으로 확인된 중요한 사실이 있다. 에이전트를 한 번 사용해본 개발자는 사용 종료 시 역체감이 매우 크다. 에이전트 없이 코딩하던 방식으로 돌아가기 어려워지는 구조다. 이것이 장기적 운영의 자연스러운 동기가 된다. 별도의 의무화 없이도 지속 사용이 유지된다.

## 9. 핵심 차별점 요약

일반적인 AI 코딩 솔루션(Tabnine 등)과의 차별점은 단순히 기능의 차이가 아니다.

**차별점 1 — 표준이 이미 있다**
범용 솔루션은 기업 표준을 새로 입력하고 구조화해야 한다. 우리는 1차 프로젝트에서 SK케미칼 표준을 직접 만들었다. Context Framework 초기 구성의 품질과 속도가 다르다.

**차별점 2 — 표준의 의도를 알고 있다**
표준 문서에는 "왜 이 규칙을 만들었는가"라는 맥락이 있다. 이 맥락 없이 Framework에 적재하면 에이전트가 예외 상황을 잘못 처리한다. 우리는 그 맥락의 당사자다.

**차별점 3 — 표준화에서 운영까지 연속성**
1차(표준화 컨설팅) → 2차(AI 운영화)가 단절 없이 이어진다. 고객 입장에서 맥락을 새로 설명할 필요가 없다.

**차별점 4 — 품질 포탈로 성과가 보인다**
도입 효과가 추상적인 주장이 아니라 기존 포탈 지표의 수치 변화로 측정된다.
