**배경** 
1차 과제에서 SK케미칼 SW 개발 표준(SDLC 프로세스, 체크리스트, 산출물 패키지)을 S-Core가 직접 정의했다. 2차 과제는 그 표준이 실제 프로젝트에서 지켜지는지 확인하는 체계를 만드는 것.

**SK케미칼 구조**
- 내부 개발자 없음. 개발은 전부 협력업체(수행사)가 담당
- PM 1명이 최대 10개 프로젝트 병행
- 사용 도구: Jira, Sparrow, Bitbucket, Unit test, 코드 커버리지, oss 취약점 등

**1차에서 정의된 SDLC 구조**
```
착수 → 설계(게이트) → 개발(증빙) → 테스트/배포(게이트) → 운영
```
각 단계에 체크리스트, 산출물 패키지, 게이트가 정의되어 있음. 단, 아직 실운영은 시작 전.
- 각 품질 요소별 임계치가 수치로 정의되어 있음
- 체크리스트는 문서로 존재하고, 단계별로 별도 정의되어 있음
- 단계별 산출물 패키지 목록이 정의되어 있음

**아이데이션**
지표를 관찰에서 → 판단으로 
수행사가 보낸 데이터(Input)를 발주사의 기준(Rule)으로 필터링하여, PM이 의사결정할 수 있는 지표(Output)로 가공
PM이 여러 도구를 뒤지지 않고 프로젝트 품질을 한눈에 파악하고 리포팅·검수를 용이하게

**구성 예**(확정 아님)

품질포탈
여러 도구에 흩어진 데이터를 한 곳에서 보여주고, SK케미칼 표준 임계치 기준으로 해석해주는 대시보드. 구체적인 지표 선정은 미정.
├── 대시보드
│   ├── 산출물 품질 → 제출 여부 + 내용 검수
│   ├── 코드 품질 → 시큐어코딩, unit test, 코드 커버리지, OSS취약점 등
│   ├── 프로세스 품질 → 코드 리뷰, CI/CD 파이프라인
│   └── 일정 품질 → 진척률, 병목, 이슈트래킹, 일정관리
│
└── 에이전트
    ├── 이벤트 트리거 (제출, 임계치 초과 등)
    ├── 일정 트리거 (납품 D-7 등)
    └── 산출물 작성 / 검수 보조

데이터 수집
Jira·Bitbucket·Sparrow·Confluence API를 통해 수집하고 DevLake로 통합하는 구조가 기술적으로 가능해 보임. 실제 어떤 지표를 수집할지는 별도 논의 필요.

AI 활용
- 산출물 검수: AI가 체크리스트 초안을 작성하고 PM이 컨펌하는 방식이 논의됨. 구체적인 UX나 구현 방식은 미정.
- 지표 해석: 숫자를 그대로 보여주는 게 아니라 맥락 포함 요약이 가능하지 않을까?

Ambient 에이전트
대시보드는 PM이 능동적으로 들어가야 본다는 한계가 있음. 임계치 초과·산출물 제출·납기 임박 등 특정 트리거 시 자동으로 리포트를 푸시하는 방식이 페인포인트를 해소할 수 있지 않을까? 필수 기능인지는 미정.

메모
- 시스템 적으로 강제할 수 있는 것 → 문제 발생을 예방
- 직접 검수/관리해야하는 것 → 모니터링하며 조치
- 체크리스트 내용 확인 필요
