**프로젝트명:** Escor AI 에이전트 설득 프레임워크 문서 작성
**목적:** Escor의 AI 에이전트 영업 및 내부 정렬을 위한 설득 논리 베이스 문서

에이전트 시장에서 Escor가 왜 적합한 파트너인가를 설명하는 스토리라인. 컨설팅팀과 기술팀이 고객 앞에서 같은 언어로 이야기할 수 있도록 하는 공통 기반이며, 독자(고객/내부)에 따라 가공해서 쓰는 원본 재료다.

**세 가지 역할을 동시에 수행하는 문서:**

1. 내부 정렬 — 컨설팅팀과 기술팀의 공통 언어
2. 고객 설득 논리의 베이스 — 독자에 따라 가공해서 사용
3. 각 메이커의 방향 제시

**현재 단계:** 외부 사례로 뼈대 완성. 향후 Escor 실제 사례로 교체 예정.

---

### Escor 맥락

- **회사 구조:** 컨설팅 사업부 + AI 기술 역량 + 풀스택 인프라 구축 노하우
- **방법론 이름:** S-Core
- **핵심 포지셔닝:** 도메인 이해 + 설계 역량 + 기술 실행의 세 가지를 동시에 갖춘 구조. 시장에서 이 조합은 드물다.
- **주요 타겟:** 금융, 공공, 제조 산업의 의사결정자

### 문서 구조 (설득 frame ver1)

```
도입: 같은 시장에서 성공과 실패가 공존한다. 차이는 어디서 오는가.

1장: 왜 대부분은 실패하는가
→ Copilot 사례. 모델 문제가 아니라 설계 부재.
→ BCG 70%는 사람과 프로세스. RAND 문제 정의 오류 1위.

2장: 성공한 곳은 무엇을 다르게 했는가
→ 같은 모델, 설계만 달리해도 10%p 차이 (SWE-bench).
→ Harvey 멀티모델 라우팅, Abridge Linked Evidence.
→ 기술이 아니라 도메인 이해가 먼저였다.

3장: 도메인 이해가 왜 어려운가
→ 3수준 구조: 규칙/용어/판단 전략.
→ 대부분 2수준에서 멈춘다. 조건이 바뀌면 무너진다.

4장: 도메인 이해를 설계로 치환하는 과정
→ JPMorgan, Palantir 911, Harvey.
→ 초기 가설 틀림 → 현장 발견 → 설계 변경 패턴.

5장: 누구랑 해야 하는가
→ 컨설팅/기술/플랫폼 각각의 구조적 한계.
→ 도메인 이해 + 설계 역량 + 기술 실행의 조합이 필요하다.
→ 이 조합은 드물다. [ Escor 자리 ]

6장: 그 조건을 갖춘 구조는 어떻게 작동하는가
→ 4단계 방법론: 엘리시테이션 → 아키텍처 치환 → 스캐폴딩 구축 → 검증
→ 이터레이션: 실패 지점이 1단계로 돌아가는 복리 구조

7장: 그래서 무엇을 기대할 수 있는가
→ "개발자 몇 명 줄어요?"가 잘못된 질문인 이유.
→ 세 가지 구조: 규모 불가능→가능 / 속도가 전략 / 전문가 시간 잠식.

어펜딕스 A: 방법론 상세 (4단계 행동 가이드, 산출물, 이터레이션)
어펜딕스 B: 산업별 설계 가이드 (금융/공공/제조)
```

### 핵심 설계 원칙

- 문서가 "우리가 최고"라고 말하지 않고 독자가 스스로 "이 방법론을 가진 곳"을 찾게 만드는 구조
- 사례는 "초기 가설 → 현장 발견 → 설계 변경" 패턴으로 서술 (독자가 자기 조직에 대입 가능)
- 효과는 "효율 개선"이 아니라 "가능성의 확장" 언어 사용
- 방법론은 어펜딕스로 분리. 본문은 스토리 흐름 유지
