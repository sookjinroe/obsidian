### 경계 잡기
**해석 A — 생태계 레이어 관점** (개발자)
LangChain, CrewAI처럼 "agent를 만들기 위해 그 위에 올라타는 구조체". 경계는 **제어권** — 프레임워크가 agent의 실행 흐름을 가져감. 라이브러리(단순 도구 모음)와 구분됨.

**해석 B — 런타임 컨텍스트 관점** (기획자, 전략)
agent가 작업을 수행할 때 **맥락으로서 주입되는 모든 것의 집합** — 도구, 지식, 지침, 스킬, 메모리 등. 경계는 **agent 내부에서 참조 가능한가** 여부. "framework"라기보다 "context bundle"에 가까운 개념.


**해석 A — 생태계 레이어 관점** (개발자)
**Core (오픈소스 Cline 등)** — 프레임워크 본체, 실행 흐름/인터페이스/라이프사이클
**Context Framework (이 문서)** — Cline의 Extension Point를 SDLC 도메인으로 미리 채워넣은 것. 이게 상품의 핵심 차별점이자 IP.
**오퍼링 (판매 단위)** — Context Framework + 인프라 구성 + 시스템 연동 컨설팅

**"Context"** — agent가 작동할 때 필요한 컨텍스트(도구, 지식, 지침, 스킬)를 SDLC에 맞게 정의했다는 의미
**"Framework"** — 단순 기능 묶음이 아니라, Cline 위에서 일관된 구조로 확장을 구성했다는 의미

**외부 고객에게 전달될 때** "프레임워크"라는 단어가 개발자에게는 "내가 직접 커스터마이징할 수 있는 것"으로 읽힐 수 있음
고객이 추가 확장을 할 수 있는 구조인지, 아니면 닫힌 완성품인지? (에 따라 네이밍을 조정해야할 수도)

**레이어별 제공 형태**

**Rules/Instructions 레이어** — 가장 가벼운 형태. YAML/Markdown 파일로 제공하면 개발자가 직접 편집 가능. Cline의 `.clinerules` 같은 방식.

**Tool/MCP 레이어** — MCP 서버를 모듈 단위로 패키징해서 제공. 개발자가 필요한 것만 골라 연결하거나 자체 MCP 서버를 추가할 수 있음.

**지식/RAG 레이어** — 벡터DB나 문서 저장소의 연결 설정을 열어주면, 개발자가 사내 문서나 도메인 지식을 직접 주입 가능.

---

**근데 여기서 진짜 질문은**

커스터마이징을 허용할수록 **지원 비용과 품질 보증 범위**가 복잡해져요.

그래서 보통 이런 제품은 두 가지 티어로 구분해요. 완성품으로 닫아서 공급하는 **Managed 티어**와, 확장 가이드와 함께 열어주는 **Enterprise/Self-hosted 티어**.

**해석 A 검토를 계속하면**

방금 대화에서 나온 걸 보면, 실제 제품 구조가 "개발자가 직접 Extension Point를 건드린다"기보다 "우리가 미리 채워서 완성품으로 공급한다"에 가까워요.

그렇다면 **해석 A의 조건을 충족하는 건 고객이 아니라 우리(공급자)** 인 거예요. 우리가 Cline을 프레임워크로 사용해서 Context Framework를 만든 것.

맞아요, 그러면 이 문서 자체가 **Context Framework의 목차이자 구성 요소 전체**인 거네요.

그렇다면 해석 B를 정의할 때 핵심은 **"집합"이 아니라 "체계"** 로 가야 할 것 같아요.

단순히 "이런 것들이 다 들어있어요"가 아니라, **왜 이것들이 함께 묶였는지, 어떤 원리로 구성되었는지**가 있어야 "Framework"라는 이름이 정당화되거든요.

---

지금 문서의 구성 원리를 보면 **SDLC 단계 × 기능 카테고리** 의 매트릭스 구조예요.

이게 해석 B의 Framework로서 의미를 가지려면 — **"agent가 SDLC 각 단계에서 올바르게 작동하려면 어떤 컨텍스트가 필요한가"를 체계적으로 정의한 것** 이라고 볼 수 있어요.
**Context Framework 정의 (초안)**

"특정 도메인에서 AI agent가 효과적으로 작동하기 위해 필요한 컨텍스트(도구, 지식, 지침, 스킬)를 체계적으로 정의한 구조"

**이 문서는 그 중 SDLC 도메인에 적용한 인스턴스**


**Context Framework 경계 정의**

**포함** — agent가 작동할 때 직접 참조하거나 실행하는 것들. 도구(Tool), 지식(RAG), 지침(Rules/Instructions), 스킬(Prompt/Rule 기반 기능)

**제외** — agent 외부에서 작동을 가능하게 해주는 것들. 인프라(사내망, 서버, 배포 환경), 컨설팅(시스템 연동, 도입 지원)

---

경계 기준은 **"agent가 런타임에 직접 쓰는가"** 예요.

인프라와 컨설팅은 agent가 잘 작동하기 위한 **전제 조건**이지, agent가 작업 중에 참조하는 컨텍스트가 아니니까요.

이렇게 되면 오퍼링 구조도 자연스럽게 정리돼요. **Context Framework = 제품 코어**, 인프라 + 컨설팅 = **딜리버리 레이어**. 이 구분으로 팀 내 커뮤니케이션도 훨씬 깔끔해질 것 같아요.


아, 반대군요. 정리하면:

**개발자** 입장 — Cline 포함한 전체가 Context Framework (기술 스택 전체를 하나의 구조로 봄)

**기획/전략** 입장 — Context Framework는 우리가 만든 레이어만 (Cline은 그냥 오픈소스 베이스)

---

그리고 제품 구조를 다시 보면:

**파는 것** = AI 코딩 어시스턴트 (Cline + Context Framework가 결합된 완성품)

**Context Framework** = 그 완성품 안에 있는 우리의 IP 레이어

**오퍼링** = 완성품 + 도입에 수반되는 모든 것

---

그렇다면 커뮤니케이션 미스의 핵심은 **"Context Framework = 제품"으로 오해하는 것**이네요. 기획/전략이 Context Framework를 제품명처럼 쓰기 시작하면, Cline이 우리 거인지 오픈소스인지 경계가 흐려지고 — 고객에게도 잘못된 메시지가 전달될 수 있어요.

이 미스를 막으려면 내부에서 **제품명과 Framework 이름을 명확히 분리**하는 게 필요할 것 같아요. 제품명이 따로 있나요?



이해했어요. 각 정의가 읽히는 사람마다 어떻게 다양하게 해석될 수 있는지를 펼쳐놓는 거군요. 그 해석들을 테이블에 올려놓고 "우리가 의도하는 건 어느 것인가"를 함께 좁혀가는 용도로.

## Context Framework 정의 논의 자료

### 목적

Context Framework의 정의를 하나로 통일하기 위해, 각 정의 후보가 어떻게 다양하게 읽힐 수 있는지를 먼저 펼쳐놓고 우리가 의도하는 해석을 함께 확정한다.

---

### 안 A

**"오픈소스 에이전트의 Extension Point를 특정 도메인에 맞게 채운 구조"**

이 정의는 읽는 사람에 따라 다음과 같이 해석될 수 있다.

**해석 A-1.** Cline이라는 특정 오픈소스를 기반으로 한 기술 구현체. 즉 Cline + 우리 레이어의 결합물 전체를 가리킨다.

**해석 A-2.** Cline은 예시일 뿐이고, 어떤 오픈소스 에이전트든 그 위에 올라타는 도메인 특화 레이어를 가리킨다. 베이스 에이전트가 바뀌어도 Framework는 유지된다.

**해석 A-3.** "채운 구조"를 완성된 패키지로 읽어, 한번 정의되면 고정된 산출물로 본다.

**해석 A-4.** "채운 구조"를 살아있는 레이어로 읽어, 도메인이 확장되거나 기능이 추가되면 지속적으로 변화하는 것으로 본다.

---

### 안 B

**"특정 도메인에서 AI agent가 효과적으로 작동하기 위해 필요한 컨텍스트를 체계적으로 정의한 것"**

**해석 B-1.** 컨텍스트의 목록, 즉 도구/지식/지침/스킬 등의 구성요소 리스트를 가리킨다. SDLC 문서가 곧 Framework 그 자체다.

**해석 B-2.** 목록이 아니라 그것들을 구성하는 원리와 기준을 가리킨다. 문서는 그 원리를 적용한 결과물이다.

**해석 B-3.** "특정 도메인"을 SDLC로 한정해서 읽어, 코딩 어시스턴트에만 해당하는 개념으로 본다.

**해석 B-4.** "특정 도메인"을 변수로 읽어, SDLC는 첫 번째 인스턴스이고 HR, 리서치 등에도 동일하게 적용 가능한 개념으로 본다.

---

### 논의 질문

각 해석 중 우리가 의도하는 것은 무엇인가?


두 가지를 추가하면 좋을 것 같아요.

- Cline 자체는 Context Framework인가?
- 인프라 설정(폐쇄망 런타임)은 Context Framework인가?
