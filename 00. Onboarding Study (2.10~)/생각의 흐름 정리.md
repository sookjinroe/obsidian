### 1. 왜 하는가
**보안 규제로 인한 기술 고립**
금융·공공기관은 보안 규제로 인해 외부 **클라우드**의 **SOTA AI**를 사용하지 못함. 이로 인해 최신 AI 기술로부터 고립되어 내부 개발자들의 생산성이 정체되는 문제를 겪는다. 이것은 단순히 불편함을 넘어 인력 유출과 조직 경쟁력의 약화로 이어짐

### 2. 어떻게 구성하는가
<small> 통제된 환경에서도 - SOTA 성능의 에이전트를 지속 공급하고 - 투자 회수 가치를 증명 </small>
**On-Premise AI 스택 구축**
이 격차를 해결하려면 외부와 단절된 고객사 내부망 안에서만 독립적으로 작동하는 하드웨어와 소프트웨어가 결합된 전용의 AI 환경을 물리적으로 구축해야 함.
> [!info]- AI 스택의 구성 예시
>- **Top (소프트웨어):** AI 에이전트, 사용자 인터페이스
>- **Middle (환경):** 파이썬, CUDA, AI 모델 라이브러리, 데이터베이스
>- **Bottom (인프라):** 운영체제(Linux 등), 서버 하드웨어(GPU/CPU/RAM)

**지속 가능한 성능**
설치형 AI는 업데이트가 멈추는 순간 도태된다. 오픈소스 리눅스를 기업용으로 검증하고 관리해 성공한 레드햇(Red Hat)의 방식처럼, 매달 쏟아지는 최신 오픈소스 AI 모델들을 기업 환경에 맞게 검증하고 패키징하여 정기적으로 공급하는 구독형 운영 체계 필요
> [!info]- Red Hat → 기업용 운영체제인 Red Hat Enterprise Linux (RHEL)
>- 검증된 오픈소스: 
>  오픈소스 프로젝트(Fedora)에서 검증된 기술을 가져와, 기업이 안심하고 쓸 수 있도록 안정성과 보안을 강화하여 패키징
>- 구독 기반: 
>  소프트웨어 자체 값을 받는 것이 아니라, 정기적인 **기술 지원, 보안 업데이트, 인증(Certification)** 서비스에 대한 연간 사용료 모델
>- 지속성: 
>  모듈형 표준 제품을 제공하되, 각 기업의 특수한 환경(네트워크, 보안 등)은 Configuration으로 처리, Ansible(앤서블)이라는 도구로 표준 제품+설정을 자동 배포
>- 생태계 주도:
>  오픈소스 커뮤니티에 적극적으로 기여하여 기술의 표준을 주도, 본인들이 만든 기술이 표준이 되면 이를 제품화하여 시장을 선점하기도 함 
>기업 환경에서는 무료 오픈소스에 돈을 내는 이유로 다음의 가치가 있음
>- 서버에 문제가 생겼을 때 24시간 기술 지원을 받을 수 있는 운영 안정성
>- 한 번 설치하면 10년 이상 보안 패치를 보장, 잦은 시스템 변경이 어려운 금융·공공 기관에 적합
>- 주요 하드웨어(서버) 및 소프트웨어(DB 등) 제조사들이 "RHEL에서 완벽히 작동한다"는 것을 공식 인증

**증명 가능한 규모 - 비용 최적화**
클라우드보다 비싼 인프라는 경쟁력이 없다. 고객사의 조직 규모와 실제 사용량에 맞춰 과투자 없이 최적의 성능을 낼 수 있도록 하드웨어와 소프트웨어 스택을 셋팅하여 투자금을 회수할 수 있는 가성비를 숫자로 증명해야 함
<small>*클라우드는 쓰는만큼 비용이 우상향하지만, 온프레미스는 인프라 감가상각 + 인건비 세이브로 투자금 회수 가능</small>

### 3. 무엇이 다른가
<small>제한된 로컬 인프라로 클라우드급 성능을 내도록 지식과 자원을 어떻게 엮고(Context), 어떻게 연산할 것인가(Compute)를 정의</small>
<small>→ 자체 AI가 아니어도 의미 있는 이유</small>
**Context Engineering**
범용 AI가 기업 고유의 업무 맥락을 이해할 수 있도록 Knowledge와 Tool을 설계, 사내 지식(소스 코드 의존성, DB 스키마, 코딩 컨벤션 등)을 지식 그래프 등으로 구조화하고, 금융 가드레일 점검 등 에이전트가 수행해야 할 특화 작업을 전용 Tool로 개발하여 지능과 시스템을 정밀하게 결합할 수 있다.
> [!info]- 금융권 Context Framework 예시
> - **Knowledge:** 코드 베이스(AST), 데이터 사전, 보안 규정 인터프리터, API 명세서
> - **Tools:** 가속 심의 자동 체크, 취약점 스캐너, 계정계 DB 쿼리 샌드박스
> - **Environment:** 폐쇄망 전용 런타임, 사내 표준 빌드 시스템 연동

**Low Latency Engineering**
인프라와 런타임 최적화로 Latency를 최소화하고, 확보된 물리적 성능을 에이전트의 연산량으로 전환한다. 확보된 연산 자원을 에이전트의 Self-Reflection이나 기업의 Context 반영 프로세스에 재투자하여 출력 품질을 끌어올림

### 4. 어떻게 체감되는가
**내 작업 환경에 통합된 + Actionable Agent**
사용자의 작업 도구(IDE), 터미널, 파일 시스템과 완전히 동기화되어 실제 수정을 수행, 격리된 실행 환경(Sandbox)과 복구 기능(Rollback)을 통해 에이전트의 실수가 시스템 장애로 이어지지 않는 안전한 실행 환경을 제공

**추적·통제 가능한 자동화**
AI의 추론 과정을 투명하게 공개하여 블랙박스 리스크를 해소한다. 사용자는 에이전트의 사고 단계를 실시간으로 모니터링하고, 필요 시 기업 정책에 맞게 방향을 교정할 수 있는 제어권을 가짐으로써 안전한 자동화를 구현

### 5. Vertical Agent 생태계로 확장
<small>SW 개발 에이전트 외의 다른 업무 영역으로 어떻게 확장할 것인가?</small>
**워크플로우의 비표준성**
SW 개발은 '코딩'이라는 행위가 전 세계적으로 표준화되어 있어 확장성이 높음. 반면, 타 직무(금융, 행정 등)는 기업마다 고유한 워크플로우(작업 순서 및 승인 절차)를 가지고 있음. 타 업무 확장은 제품 판매가 아닌 '업무 프로세스 재설계'에 가까운 커스텀 엔지니어링을 요구함

**실행 환경의 파편화** 
개발 업무는 CLI, Git, 파일 시스템 등 텍스트 기반의 구조화된 인터페이스 안에서 완결됨. 반면, 타 직무는 비구조화된 웹 UI, 레거시 GUI, 엑셀, 내부 ERP 등으로 파편화 되어있음

**오픈소스 생태계 분화**
AI 에이전트 시장은 '코딩'을 시작으로 리서치, 데이터 분석, 행정 자동화 등 직무별로 특화된 Vertical Open Source Software들이 등장(실험)하는 단계에 진입함. 최신 에이전트들을 소싱하여 On-Premise로 공급하는 것으로 출발할 수 있음
> [!info]+ 오픈소스 버티컬 에이전트 예
> - **리서치 특화 - [GPT-Researcher](https://github.com/assafelovic/gpt-researcher):** 20개 이상의 소스에서 자료를 긁어와서 스스로 개요를 짜고 논문급의 심층 보고서 작성    
>- **데이터 분석 특화 - [Vanna.ai](https://vanna.ai/):** 자연어로 질문하면, 스스로 SQL을 짜서 DB에 접근하고 시각화 차트까지 그려줌
>- **웹 서핑/운영 특화 - [Skyvern](https://github.com/Skyvern-AI/skyvern):** 복잡한 웹 UI를 사람처럼 눈으로 보고 판단하며 브라우저를 제어
