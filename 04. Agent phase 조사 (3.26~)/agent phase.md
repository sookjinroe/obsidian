## Phase 1 — configuration이 지시다

configuration이 대화가 끝나면 사라집니다. 프롬프트와 temperature가 전부였던 이 시기에, 에이전트에게 전달하는 것은 지금 이 순간을 위한 말뿐이었습니다. 다음 대화에서 에이전트는 이전에 무슨 말을 했는지 모릅니다. 매번 처음부터 다시 설명해야 했습니다.

에이전트 외부에는 아무것도 없었습니다. 접근할 수 있는 파일도, 도구도, 기억도 없었습니다. 에이전트는 학습한 것과 지금 이 순간 사용자가 말해주는 것만 처리했습니다.

**configuration이 존재하는 방식:** 일회성. 사용되면 사라집니다. 에이전트 외부에 아무것도 남지 않습니다.

---

## Phase 2 — configuration이 능력 선언이다

처음으로 에이전트 외부에 지속적인 무언가가 생겼습니다. 도구 스키마입니다. "이 에이전트는 웹 검색을 할 수 있고, 이 에이전트는 계산기를 쓸 수 있다"를 JSON으로 정의해서 에이전트에게 알려줬습니다.

그런데 도구 스키마는 에이전트에게 도구의 존재를 알려줄 뿐입니다. 에이전트는 "search_web을 query='서울 날씨'로 호출하겠습니다"라는 구조화된 신호를 출력합니다. 그러면 그 신호를 받은 개발자의 코드가 실제 API를 호출하고 결과를 에이전트에게 돌려줍니다. **실행은 에이전트 외부의 코드가 담당합니다.** 에이전트는 무엇을 호출할지 결정하지만, 실제로 호출하지는 않습니다.

LangChain의 체이닝도 같은 성격입니다. "검색 → 요약 → 번역"을 연결하는 것은 능력들의 순서를 정의한 것입니다. 흐름이 생겼지만 고정된 순서이고, 실행 로직은 여전히 코드 안에 있습니다.

**configuration이 존재하는 방식:** 에이전트가 무엇을 할 수 있는지를 기술합니다. 실행 자체는 configuration 밖에 있습니다.

---

## Phase 3 — configuration이 로직이다

LangGraph의 그래프가 등장했습니다. 노드는 에이전트나 함수이고, 엣지는 조건부 전환입니다. 그리고 상태(state)가 생겼습니다.

이전까지는 "A 결과가 X이면 B로 가고, Y이면 C로 간다"는 분기 로직을 개발자가 파이썬 코드로 작성했습니다. LangGraph에서는 이 분기 자체가 그래프 정의입니다. 노드를 만들고 conditional_edge로 연결하면 그것이 실행 로직입니다. configuration을 정의하는 것이 프로그래밍하는 것과 같아졌습니다.

상태가 생긴 것도 중요합니다. 각 단계의 결과가 다음 단계에 전달됩니다. "A 에이전트가 분석한 결과를 B 에이전트가 이어받아 처리한다"가 가능해졌습니다. 이전의 체인은 결과를 단순히 다음으로 넘겼지만, 그래프는 전체 상태를 관리하면서 어느 방향으로 흐를지를 동적으로 결정합니다.

Phase 2에서 실행 로직이 코드 안에 있었다면, Phase 3에서는 실행 로직이 configuration 안에 들어왔습니다.

**configuration이 존재하는 방식:** 어떤 조건에서 무엇을 어떤 순서로 실행하는지를 직접 정의합니다. configuration이 곧 프로그램입니다.

---

## Phase 4 — configuration이 경계다

에이전트가 실제 세계에 직접 닿기 시작했습니다. Phase 2에서는 에이전트가 "이 도구를 호출하겠다"는 신호를 냈고 코드가 실제로 실행했습니다. Phase 4에서는 에이전트가 직접 파일을 읽고, 터미널을 실행하고, 데이터베이스에 접근합니다. 중간에 코드가 없습니다.

MCP가 이 단계의 핵심입니다. 이전까지는 각 에이전트가 외부 시스템을 자기 방식으로 연결했습니다. 파일 시스템에 접근하는 코드, API를 호출하는 코드가 각각 달랐습니다. MCP 이후에는 "이 에이전트는 이 MCP 서버에 연결된다"고 선언하는 것만으로 그 서버가 제공하는 모든 것에 에이전트가 직접 닿을 수 있게 됐습니다. 연결이 configuration의 선언으로 성립합니다.

이때 configuration의 역할이 Phase 2와 근본적으로 달라집니다. Phase 2의 도구 스키마는 에이전트에게 도구의 존재를 알려줬습니다. Phase 4의 MCP configuration은 에이전트에게 실제 접근권을 부여합니다. 그리고 동시에 어디까지 허용하고 어디서 막을지를 정의합니다. 이 MCP 서버에는 연결되고, 저 시스템에는 연결되지 않는다 — 이것이 에이전트가 세계와 맞닿는 경계입니다.

에이전트가 세계에 직접 닿을 수 있게 됐기 때문에, 그 경계를 어떻게 설정하느냐가 어플리케이션의 안전성과 신뢰성을 결정하게 됐습니다.

**configuration이 존재하는 방식:** 에이전트가 실제로 접근할 수 있는 것과 없는 것의 경계를 정의합니다. 선언이 곧 접근권의 실제 부여입니다.

## Phase 5 — configuration이 정체성이다

Phase 4까지는 configuration이 에이전트에게 무언가를 추가했습니다. 도구를, 로직을, 연결을. Phase 5에서는 다른 일이 일어납니다. configuration의 집합이 에이전트가 무엇인지를 정의합니다.

실행 엔진은 범용입니다. Claude Code는 어디서든 같습니다. 그런데 그 위에 어떤 configuration을 올리느냐에 따라 완전히 다른 에이전트가 됩니다. TDD를 강제하는 스킬과 훅을 올리면 TDD 방식으로만 일하는 에이전트가 됩니다. 특정 조직의 코드 리뷰 기준이 담긴 스킬을 올리면 그 조직의 기준으로 검토하는 에이전트가 됩니다. 같은 엔진인데 configuration이 다르면 전혀 다른 에이전트입니다.

이것이 소프트웨어 개발의 의미를 바꿉니다. 이전에는 어떻게 구현할 것인가가 핵심이었습니다. 코드를 작성하는 것이 어플리케이션을 만드는 것이었습니다. Phase 5에서는 무엇을 정의할 것인가가 핵심입니다. 스킬을 어떻게 구성할지, 훅이 어느 시점에 무엇을 막을지, 에이전트들이 어떻게 협력할지 — 이 configuration의 집합이 어플리케이션입니다. 코드와 어플리케이션의 관계가 바뀌었습니다.

Phase 2에서 configuration은 에이전트에게 "이런 것들을 할 수 있다"고 알려줬습니다. Phase 5에서 configuration은 "이 에이전트는 이렇게 존재한다"를 정의합니다. 이름, 역할, 일하는 방식, 무엇을 허용하고 무엇을 막는지가 모두 configuration으로 표현됩니다.

**configuration이 존재하는 방식:** 에이전트의 정체성 그 자체입니다. configuration을 보면 이 에이전트가 무엇인지 알 수 있습니다.

## Phase 2와 Phase 4의 차이 — 직접 비교

같은 "웹 검색"을 예로 들면 차이가 명확합니다.

**Phase 2에서 웹 검색:** 에이전트는 search_web 도구가 있다는 것을 알고, "query='서울 날씨'로 search_web을 호출하겠습니다"라는 신호를 출력합니다. 그 신호를 받은 코드가 실제 검색 API를 호출하고 결과를 에이전트에게 돌려줍니다. 에이전트는 검색을 직접 하지 않았습니다.

**Phase 4에서 웹 검색:** 에이전트가 웹 검색 MCP 서버에 직접 연결되어 있습니다. 에이전트가 검색을 결정하면 MCP 서버에 직접 요청을 보내고 결과를 받습니다. 중간에 아무도 없습니다.

Phase 2는 에이전트가 무엇을 원하는지 표현하면 다른 무언가가 실행합니다. Phase 4는 에이전트가 직접 실행합니다. 이것이 두 Phase의 본질적 차이입니다.
