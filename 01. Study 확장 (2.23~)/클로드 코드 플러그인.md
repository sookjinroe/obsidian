## 2. 에이전트 호환성 및 확장 기술
클로드 코드를 위해 작성된 스킬과 플러그인은 다른 AI 에이전트에서도 사용 가능한 범용성을 지향합니다.
* **OpenCode**: `.claude/skills/` 경로를 직접 읽어 별도 변환 없이 네이티브로 지원합니다.
* **타 에이전트 (Cursor, Windsurf, Aider 등)**: `openskills` CLI 도구를 통해 설치하며, `AGENTS.md`에 삽입되거나 `.clinerules` 형식으로 변환되어 작동합니다.
* **동기화**: `openskills sync` 커맨드를 통해 외부 소스 레포지토리와 로컬 스킬 간의 업데이트를 유지할 수 있습니다.

## 3. 플러그인 아키텍처 및 구성 요소
클로드 코드 플러그인은 코드가 아닌 **마크다운(.md)과 JSON 파일**로 구성된 파일 기반 구조입니다.

| 구성 요소 | 파일/경로 | 상세 역할 |
| --- | --- | --- |
| **메타데이터** | `plugin.json` | 플러그인 이름, 설명, 버전을 명시하는 신분증 역할. |
| **커넥터** | `.mcp.json` | **MCP(Model Context Protocol)** 서버 설정. 외부 DB나 SaaS(HubSpot, Slack, Snowflake 등)와 연결. |
| **커맨드 (Commands)** | `commands/*.md` | 사용자가 `/call-prep`처럼 **명시적으로 호출**할 때 실행되는 지시문. 입력과 실행 절차 정의. |
| **스킬 (Skills)** | `skills/*/SKILL.md` | YAML 프론트매터의 `description`을 기반으로 클로드가 **맥락에 따라 자동으로 판단**하여 호출하는 기술. |
| **에이전트 (Agents)** | `agents/*.md` | 특정 작업에 특화된 별도의 컨텍스트에서 작동하는 **서브 에이전트** 정의. |
| **개인화 설정** | `settings.local.json` | 사용자 개인의 회사명, 제품명, 할당량 등 실제 데이터를 저장. |

---

## 4. 데이터 연동 방식 (MCP)
플러그인 자체는 '무엇을 할 것인가'에 대한 논리만 담고 있으며, 실제 데이터 접근은 MCP가 담당합니다.
* **재무**: ERP(SAP 등)나 데이터 웨어하우스(Snowflake 등)에서 전표 및 쿼리 데이터를 호출.
* **세일즈**: CRM(HubSpot, Salesforce)과 연동하여 딜 히스토리나 담당자 정보를 호출.
* **유연성**: 특정 툴에 종속되지 않고 카데고리 기반으로 설계되어, `.mcp.json` 수정만으로 연동 서비스 교체가 가능합니다.

---

## 5. 개발 및 선정 전략
Anthropic이 초기 플러그인 분야를 선정한 배경은 다음과 같은 실무적 기준에 근거합니다.
1. **고비용 전문직의 반복 작업**: 주니어 뱅커나 법무 담당자가 수행하던 저부가가치 문서 작업을 자동화하여 효율 극대화.
2. **SaaS 협력 모델**: 기존 소프트웨어를 대체하는 대신, 주요 데이터 벤더(LSEG, FactSet 등)와 파트너십을 맺어 데이터 인프라 내부로 침투.
3. **검토 중심 워크플로우**: AI가 초안(80%)을 작성하고 인간 전문가가 최종 검토(20%)하는 구조가 자연스러운 문서 중심 산업 공략.

| **OpenCode**          | ✅ 네이티브 지원   | `.claude/skills/` 경로를 그대로 읽음. 별도 변환 불필요 |
| **Cursor / Windsurf** | ✅ 지원        | openskills 툴로 AGENTS.md에 삽입, 별도 가공 없음   |
| **Cline**             | ✅ 지원        | `.clinerules` 방식으로 변환 가능                |
| **GitHub Copilot**    | ✅ 지원        | AGENTS.md 기반                            |
| **Codex**             | ✅ 지원        | `~/.codex/skills/`                      |
| **Devin**             | ⚠️ 미확인      | 공식 호환 언급 없음                             |

---

### 핵심 포인트 3가지
**1. OpenCode는 변환 없이 그냥 됩니다**
OpenCode가 `.claude/skills/*/SKILL.md` 경로를 직접 스캔합니다. Claude Code용 스킬을 그대로 두면 OpenCode도 자동으로 읽어갑니다. 파일 하나도 건드릴 필요가 없어요.
**2. 다른 에이전트들도 커맨드 하나로 됩니다**
`openskills`라는 CLI 툴이 있는데, "Anthropic의 스킬 시스템을 모든 AI 에이전트에 가져다 주는" 범용 인스톨러입니다. Claude Code, Cursor, Windsurf, Aider, Codex 등에 동일하게 설치되며, 개별 플러그인 수정은 불필요합니다.
**3. 플러그인 전체(Commands, Hooks 등) 변환도 커맨드 하나로 됩니다**
`compound-engineering-plugin`처럼 Claude Code 플러그인 전체를 OpenCode, Codex, Gemini CLI, GitHub Copilot, Kiro CLI 등으로 변환하는 CLI도 이미 존재합니다. 스킬은 심링크로 연결되어 원본이 업데이트되면 자동 반영됩니다.

---

### 업데이트 문제도 해결됨
외부 스킬들은 소스 레포와 자동으로 동기화 유지됩니다. `openskills sync` 커맨드 하나로 전체 갱신 가능하고, "Write once. Run everywhere." 구조입니다.

---

**결론**: 세세한 가공 없이 쓸 수 있는 구조가 맞습니다. OpenCode는 아예 변환도 필요 없고, 다른 에이전트들도 커맨드 하나(`openskills sync`)로 동기화됩니다. Devin만 공식 확인이 안 되는 상황이에요.



### 로컬에 없는 데이터 연동은?
플러그인의 스킬/커맨드는 "무엇을 해야 하는가"만 정의하고, 실제 데이터는 MCP가 가져오는 구조입니다.
### 플러그인별 실제 연동 방식
**재무(Finance) 플러그인 예시**
`.mcp.json`에 MCP 서버를 추가하는 방식입니다. ERP(NetSuite, SAP)는 시산표와 분개 데이터를 자동으로 가져오고, 데이터 웨어하우스(Snowflake, BigQuery)는 재무 데이터 쿼리에 사용됩니다. 이 연결 없이도 데이터를 붙여넣거나 파일을 업로드해서 분석할 수 있습니다. [GitHub](https://github.com/anthropics/knowledge-work-plugins/tree/main/finance)
**데이터(Data) 플러그인 예시**
데이터 웨어하우스로 Snowflake, Databricks, BigQuery 또는 SQL 호환 DB를, 분석/BI로 Amplitude, Looker, Tableau 등을 `.mcp.json`이나 Claude Code 설정에 MCP 서버로 추가하면 직접 데이터 접근이 활성화됩니다. [GitHub](https://github.com/anthropics/knowledge-work-plugins/tree/main/data)
**세일즈(Sales) 플러그인 예시**
CRM(HubSpot, Close), 데이터 인리치먼트(Clay, ZoomInfo) 등을 MCP 서버로 연결하고, `sales/.claude/settings.local.json`에 회사명, 담당자 이름, 할당량 같은 개인화 정보를 JSON으로 입력합니다. [GitHub](https://github.com/anthropics/knowledge-work-plugins/tree/main/sales)


## 선정 기준 분석: 왜 이 플러그인들을 우선 개발했나

조사 결과와 Anthropic의 발언들을 종합하면 선정 기준이 꽤 명확합니다.

**1. 반복적이고 고비용 화이트칼라 노동부터**
"투자은행 플러그인이 처리하는 일은 주니어 뱅커들이 주말 내내 하던 작업"이라는 평가처럼, 고급 인력이 저부가가치 작업에 시간을 낭비하는 영역을 우선 타겟했습니다. 오퍼레이터 플러그인의 오퍼레터·온보딩 플랜처럼 "누구나 매주 쓰지만 전략 로드맵엔 없는" 작업들입니다. [Implicator](https://www.implicator.ai/anthropic-launches-10-claude-cowork-plugins-for-investment-banking-hr-and-design/)

**2. Claude Code 성공 방정식의 복제**
엔지니어들에게 Claude Code가 "없으면 못 사는 도구"가 된 방식을 그대로 일반 지식 노동자에게 적용한다는 전략입니다. 개발자 → 금융인 → 법무 → HR 순으로 파급 효과가 퍼지는 구조입니다. [CNBC](https://www.cnbc.com/2026/02/24/anthropic-claude-cowork-office-worker.html)

**3. 엔터프라이즈 매출의 80%를 지키면서 확장**
Anthropic은 "SaaS를 대체하는 게 아니라 기존 소프트웨어와 함께 작동하는 플랫폼"이라는 포지셔닝을 명확히 했습니다. FactSet, S&P, LSEG와 공동 개발한 금융 플러그인처럼, 기존 SaaS 벤더를 파트너로 끌어들여 주가 하락 충격을 완화하는 전략적 선택도 포함됩니다. [CNN](https://edition.cnn.com/2026/02/24/tech/anthropic-claude-plugins-office-jobs)

**4. 문서 생산량이 많고 검증이 어려운 영역**
법률 계약 리뷰, 재무 모델, 에쿼티 리서치 노트처럼 LLM이 초안을 내고 전문가가 검토하는 구조가 자연스러운 영역들입니다. "AI가 완전 자율 처리"보다 "AI가 80% 처리 + 인간 검토"가 당장 현실에서 받아들여지는 영역을 공략한 것입니다.

**5. 금융 섹터 집중은 전략적 파트너십 포석**
Thomson Reuters, NYSE, FactSet, MSCI, S&P Global, LSEG가 모두 직접 파트너로 참여했습니다. 이들과의 관계를 경쟁이 아닌 협력으로 전환함으로써 규제 산업의 신뢰 장벽을 낮추는 동시에 엔터프라이즈 데이터 인프라 안으로 들어가는 경로를 확보했습니다. [Constellation Research](https://www.constellationr.com/insights/news/anthropic-expands-cowork-plugins-across-enterprise-functions)


## 클로드 코드 사용자가 플러그인 직접 빌드/커스텀
## 플러그인 아키텍처 전체 구조

플러그인은 코드가 전혀 없는 **순수 파일 기반 구조**입니다. 전부 마크다운(.md)과 JSON 파일로만 이루어져 있어요.

```
sales/                         ← 플러그인 루트
├── .claude-plugin/
│   └── plugin.json            ← 1. 메타데이터 (플러그인 이름, 설명, 버전)
├── .mcp.json                  ← 2. 커넥터 (어떤 외부 툴과 연결할지)
├── commands/                  ← 3. 커맨드 (사용자가 직접 호출)
│   ├── call-summary.md
│   ├── forecast.md
│   └── pipeline-review.md
├── skills/                    ← 4. 스킬 (Claude가 자동 판단해서 사용)
│   ├── call-prep/SKILL.md
│   ├── account-research/SKILL.md
│   ├── competitive-intelligence/SKILL.md
│   ├── draft-outreach/SKILL.md
│   └── daily-briefing/SKILL.md
├── CONNECTORS.md              ← 5. 커넥터 가이드
└── README.md
```

---

## 구성 요소 4가지 상세 설명

**① plugin.json — 플러그인 신분증**

```json
{
  "name": "sales",
  "description": "Research prospects, prep for calls...",
  "version": "1.0.0"
}
```

Claude가 이 플러그인이 무엇인지 인식하게 해주는 최소한의 메타데이터. 수정할 일이 거의 없어요.

---

**② .mcp.json — 커넥터 (어떤 DB/툴과 연결할지)**

```json
{
  "mcpServers": {
    "crm": {
      "command": "npx",
      "args": ["@hubspot/mcp-server"],
      "env": { "HUBSPOT_API_KEY": "${HUBSPOT_API_KEY}" }
    },
    "slack": {
      "command": "npx", 
      "args": ["@slack/mcp-server"]
    }
  }
}
```

`.mcp.json`에 사전 구성된 MCP 서버가 있지만, 같은 카테고리의 어떤 MCP 서버든 작동합니다. 플러그인은 툴 종속이 아닌 카테고리 기반으로 설계되어 있어요.

→ **커스텀 포인트**: HubSpot 대신 Salesforce, Slack 대신 Teams 등으로 교체.

---
**③ commands/ — 커맨드 (사용자가 `/sales:call-prep` 처럼 직접 호출)**
```markdown
## Call Prep

Research a prospect company and person before a sales call.

## Input
- Company name or URL
- Contact name and title (optional)

## Steps
1. Search for recent company news, funding rounds, hiring signals
2. Pull existing deal history from ~~crm if connected
3. Review past meeting notes from ~~transcripts if connected
4. Generate talking points, likely objections, and open questions

## Output
A structured briefing with: company overview, key contacts,
recent news, your prior interactions, and suggested agenda.
```

단순 마크다운 파일. "이런 입력을 받아서 이런 절차로 이런 결과를 내라"는 지시문입니다.

→ **커스텀 포인트**: 우리 회사 영업 프로세스에 맞게 절차(Steps) 수정, 출력 형식 변경.

---

**④ skills/ — 스킬 (Claude가 맥락 보고 자동 판단)**
스킬은 YAML 프론트매터로 시작하며, `description` 필드가 핵심 트리거 메커니즘입니다. "언제 사용할지"와 "무엇을 하는지"를 모두 여기에 담아야 합니다.

```markdown
---
name: competitive-intelligence
description: Builds competitive battlecards and analysis.
  Use when user mentions competitors, asks how we compare
  to another product, needs win/loss analysis, or is 
  preparing for a competitive deal situation.
---

## When a competitive situation comes up:

1. Identify the competitor being referenced
2. Pull any stored battlecard from ~~knowledge_base
3. Search web for recent competitor news and product updates
4. Structure as: [Competitor strengths] [Our advantages]
   [Common objections + responses] [Win themes]

## Format
Keep it scannable. Use tables for direct comparisons.
```

커맨드는 "명시적 호출"이고, 스킬은 "암묵적 자동 실행"입니다. 대화 맥락에서 경쟁사 얘기가 나오면 Claude가 알아서 이 스킬을 불러옵니다.

→ **커스텀 포인트**: 우리 회사 경쟁사 이름 추가, 배틀카드 포맷 수정, 트리거 조건 조정.

---

## 커스텀 vs 신규 생성, 정확히 뭘 바꾸는가

Anthropic이 직접 안내하는 커스텀 3가지는 다음과 같습니다. 커넥터 교체(.mcp.json 수정), 회사 컨텍스트 추가(스킬 파일에 용어/조직구조/프로세스 삽입), 워크플로우 수정(스킬 지시문을 우리 팀 방식으로 변경)입니다.

정리하면:

|커스텀 대상|수정 파일|예시|
|---|---|---|
|**어떤 툴 연결?**|`.mcp.json`|Salesforce → HubSpot으로 교체|
|**어떻게 작동?**|`skills/*/SKILL.md`|우리 영업 5단계 프로세스로 절차 수정|
|**어떤 명령?**|`commands/*.md`|`/call-prep` 출력 형식 변경|
|**개인/회사 정보**|`settings.local.json`|회사명, 제품명, 경쟁사, 할당량 입력|

---

## "Claude와 함께 만든다"는 게 구체적으로 뭔가

`cowork-plugin-management` 플러그인이 별도로 있고, 이를 쓰면 Claude가 질문을 통해 스킬, 커맨드, 커넥터를 함께 작성해줍니다. 기술적으로 보면 Claude가 대화로 요구사항을 파악해서 위에 설명한 SKILL.md, commands/*.md, .mcp.json 파일들을 직접 생성해주는 구조입니다.

결국 **"Claude와 함께 만든다"는 건 Claude가 파일을 대신 써주는 것**이고, **"커스텀한다"는 건 기존 파일의 내용을 우리 회사 것으로 교체하는 것**입니다. 둘 다 결국 텍스트 파일 편집이고, 코드는 없습니다.

**GPTs랑 거의 동일한 경험**이에요. 대화로 요구사항 말하면 Claude가 파일 생성해주는 구조.

차이라면 GPTs는 결과물이 ChatGPT 안에만 갇혀있는데, 클로드 플러그인은 마크다운 파일로 저장되니까 GitHub에 올려서 팀 공유하거나 OpenCode 같은 다른 에이전트에서도 쓸 수 있어요.

---

## 관리 화면 탭 → 파일 매핑

|관리화면 탭|실제 파일|한 줄 설명|
|---|---|---|
|**Commands**|`commands/*.md`|사용자가 `/call-prep` 처럼 **직접 호출**하는 것|
|**Skills**|`skills/*/SKILL.md`|Claude가 맥락 보고 **자동으로 가져다 쓰는** 것|
|**Agents**|`agents/*.md`|특정 작업에 특화된 **별도 컨텍스트로 돌아가는 서브에이전트**|

---

Commands vs Skills 헷갈리기 쉬운데, 간단하게 구분하면:

- **Commands** = "내가 필요할 때 부른다" → `/call-prep 삼성전자` 이렇게 명시적 호출
- **Skills** = "Claude가 알아서 꺼낸다" → 경쟁사 얘기 나오면 자동으로 competitive-intelligence 스킬 활성화

**Agents**는 조금 다른 레이어인데, 메인 Claude 스레드 밖에서 별도 컨텍스트 창으로 돌아가는 서브에이전트입니다. 예를 들어 재무 플러그인에서 `/dcf` 커맨드를 실행하면 → 메인 Claude가 dcf-analyst 에이전트를 별도로 띄워서 → 그 에이전트가 모델링 작업만 집중해서 처리하고 → 결과를 메인에 돌려주는 식이에요. 무거운 작업을 분리해서 처리할 때 씁니다.
