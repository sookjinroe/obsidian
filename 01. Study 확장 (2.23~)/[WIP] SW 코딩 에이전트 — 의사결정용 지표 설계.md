## 1. 현재 지표의 문제점

### 1-1. 구조적 문제 — 용도 혼동
현재 AI 개발 생산성 지표는 도입 후 효과/운영 모니터링에 적합함(시계열 분석 필요)
그러나 실제 영업 현장에서는 도입 결심을 유도하는 지표로도 사용됨
 *'26 경영전략회의 가트너 4대 지표/기 개발된 대시보드 참고함

| 모니터링 지표             | 의사결정 지표       |
| ------------------- | ------------- |
| 도입 후 효과 추적이 목적      | 도입 결심 유도가 목적  |
| 도입 이후 시간이 흘러야 측정 가능 | 도입 이전에도 존재해야함 |
| 개발팀/CTO             | 의사결정권자        |

### 1-2. 해석 문제
대시보드나 파일럿 결과(생산성 지표, 직원 후기)는 다음과 같은 질문을 받게 됨
> "속도가 20% 빨라졌다 
> → 그래서 개발자 몇 명 줄일 수 있는지?" 
> → "개발자가 실제 개발하는 시간이 20%밖에 안 되는데, 그 20%를 줄여서 무슨 효과?"

**→ 지표 자체가 오히려 인건비 절감 프레임으로 유도하는 형태**

### 1-3. 인과관계 문제
솔루션 도입 후 생산성이 높아져도, 그 원인이 솔루션인지 다른 변수(인원 충원, 프로젝트 난이도, 시즌성 등)인지 분리 불가능
"도입 이후 수치가 좋아졌다"는 상관관계는 확인 가능 
→ "이 솔루션 때문에 좋아졌다"는 인과관계는 증명이 어려워 반박의 여지가 있음

### 1-4. 비즈니스 결과 연결 문제
코딩 속도, AI 채택률 등 개발 내부 지표는 매출, 비용 절감, 출시 속도 등 비즈니스 결과와 직접 연결되지 않음
담당자 입장에서는 수치가 좋아도 KPI와 연동이 안되면 내부 보고 시 근거가 부족함

## 2. 문제 분석을 통한 인사이트
### 2-1. ROI의 한계
ROI가 보여주지 못하는 것, 보여줘야 하는 것
- **두려움** — 안 하면 뒤처진다, 경쟁사가 하고 있다, 핵심 인력이 나가면 어떡하나
- **확신** — 우리 조직의 이 문제, 이걸로 풀리겠다
- **명분** — 위에 보고할 수 있는 그럴듯한 근거가 생겼다

### 2-2. 포지셔닝 불분명
"S/W 개발 어시스턴트" 소구 시 고객은 단순 코딩 툴과 비교 
→ 퍼블릭 구독 서비스 대비 가격이 비싸 보이고 "몇 명 줄여요?"로 이어질 가능성

실제 오퍼는 코딩·정보 탐색·문서 파악·커뮤니케이션 등 사내 개발자의 전체 업무를 커버,
→ 코드 지식이 문서화/최신성 유지되고, 사내 맥락을 AI가 이해하고, 회사 컨벤션이 AI에 내재화
→ 기술 자산을 기업이 소유해서 인적 의존도를 낮출 수 있게 + 리스크를 관리할 수 있게됨

### 2-3. 컨설팅을 통한 내부 수치 도출 가능성
업계 평균 수치가 아닌, 해당 기업 내부에서 직접 도출(직원 인터뷰, 인프라 분석 등)
→ 현황 진단 시 조직이 가지고 있는 손실을 측정한 의사결정 근거를 제시

지표 설계를 효과 예측에서 → 현재 손실 가시화로
→ 도입하면 얼마나 좋아지는지 vs 지금 얼마나 앓고 있는지 밸런싱

## 3. 의사결정용 지표 고려사항

**도입 전에 이미 존재하는 수치인가?**
파일럿 이후 결과 수치는 "이 솔루션 때문인지"를 증명하기 어렵고
현재 조직 내에 이미 존재하는 문제(온보딩 기간, 재작업 빈도 등)를
컨설팅을 통해 수치화하면 별도 증명 없이도 사실로 받아들여지지 않을까?

**효과보다 현재 손실을 보여주는가?**
"도입하면 이만큼 좋아진다"는 미래 예측은 논쟁 여지가 있어보임.
"지금 이만큼 손실이 발생하고 있다"는 현재 상태 진단은 상대적으로 설득력이 있어보임.
지표를 효과 예측보다 손실 가시화 방향으로 구성하면 어떨지.

**직관적으로 해석이 쉽고 가공이 필요없는가?**
"코드 생산성", "개발 효율" 같은 추상적인 지표는 보는 사람마다 다르게 해석될 수 있음.
무엇을 어떻게 측정했는지, 측정 방식을 단순하고 직관적으로 해서 공통된 이해 만들기
(지표 정의 논쟁 안할 수 있을 만큼)

## 4. 리스크 스캐닝
컨설팅/제안 과정에서 인터뷰·인프라 분석 외에도
우리가 보유한 AI 모듈들을 활용해 코드베이스를 직접 스캔해서 현재 상태를 수치로 제시할 수 있다면?
리스크 스캐닝까지 오퍼에 포함할 수 있다면? → 스캐닝만도 수요가 있을지?
현실적으로 인프라 구축 전에 스캐닝이 가능할지는 모르겠음

**진단 영역 예시 (현실적으로 가능한지 검토 X)**

| 진단 영역               | 질문                               | 스캐닝 예시                                                                  | 비고                                                                                                       |
| ------------------- | -------------------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **기술 부채**           | **손대기 어려운 코드가 얼마나 쌓여 있나**        | EOL 언어·프레임워크 사용 모듈 수 / 알려진 보안 취약점 포함 외부 라이브러리 비율 / 최근 2년간 커밋 없는 모듈 비율   | 부채가 있는지 알고 있었는지, 알고 있었다면 미해결 부채의 규모를 측정·가시화할 수 있도록 진단                                                    |
| **지식 사일로 · 인력 의존도** | **특정 사람만 아는 코드·맥락이 얼마나 되나**      | 모듈별 실질 기여자 1인 집중 비율 (버스 팩터) / 문서 없는 모듈 비율 / 코드↔기획 문서 간 연결 추적 가능 여부      | 비율이 높을수록 특정 인력 이탈 시 영향 범위가 넓어짐. "문서가 없다"는 것과 "연결이 안 된다"는 것은 별개의 문제로 구분해서 볼 필요 있음<br>                     |
| **품질 이슈**           | **반복 장애·재작업이 얼마나 되나**            | 핵심 모듈 테스트 커버리지 / 동일 모듈 반복 버그 발생 패턴 (커밋 메시지 분석) / 운영 배포 후 긴급 패치 비율       | 전체 평균 커버리지보다 핵심 서비스 모듈 기준으로 진단. 커버리지가 낮은 모듈과 장애 빈도가 높은 모듈이 일치한다면, 미검증 구간이 실제 운영 리스크로 이어지고 있는지 판단 가능함<br> |
| **규제 리스크**          | **현재 코드가 규제를 얼마나 준수하고 있나**       | 시큐어코딩 가이드라인 위반 건수 / 규제 항목별 코드 대응 현황 / 규제 변경 시 영향받는 모듈 범위 파악 가능 여부       | 금융·공공은 규제 위반이 과징금·인가 취소로 직결. 위반 건수보다 "파악 가능 여부" 자체가 더 중요한 질문일 수 있음                                       |
| **변경 대응력**          | **요구사항·규제 변경 시 영향 범위를 파악할 수 있나** | 요구사항 변경 후 영향 범위 파악까지 걸리는 시간 (인터뷰) / 코드↔기획 양방향 추적 가능 여부 / 과거 규제 대응 소요 기간 | 금융·공공은 규제 개정 주기가 짧아 코드 변경 요청이 수시로 발생함. 영향 범위 파악에 소요되는 시간이 길수록 대응 지연으로 인한 운영 비용이 발생하고 있다는 의미가 될 수 있음      |
