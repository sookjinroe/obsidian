## 1. 오퍼 구조

고객에게 제공하는 것은 세 가지 레이어로 구성됩니다.

```
┌─────────────────────────────────────────────────────┐
│  Tier 3. 버티컬 에이전트                              │
│  "어떤 일을 하는 에이전트인가"                         │
├─────────────────────────────────────────────────────┤
│  Tier 2. 엔터프라이즈 플랫폼                          │
│  "조직 전체가 안전하게 쓸 수 있는 인프라"               │
├─────────────────────────────────────────────────────┤
│  Tier 1. 구축 서비스                                  │
│  "도입부터 운영까지 에스코어가 책임"                    │
└─────────────────────────────────────────────────────┘
```


### Tier 1. 구축 서비스
고객사에 에이전트를 도입하고 운영 가능한 상태로 만드는 서비스입니다. 현재 이미 판매 중인 코딩 에이전트 구축이 이 형태입니다.

|단계|내용|
|---|---|
|도입 컨설팅|고객사 업무 분석 → 에이전트 적용 범위 정의|
|CF 설계·구축|도메인 전문가 인터뷰 → Rules/Skills/Knowledge 구성|
|시스템 연동|사내 ERP·그룹웨어·DB에 MCP 커넥터 연결|
|배포·교육|플랫폼 설치, 사용자 교육, 운영 이관|
|유지보수|CF 업데이트 (규제 변경, 시스템 변경 반영)|
**수익 형태**: 프로젝트 단가 + 연간 유지보수 계약


### Tier 2. 엔터프라이즈 플랫폼
개인 도구(Claude.ai, ChatGPT)와 결정적으로 다른 점입니다. 에이전트를 조직 인프라로 운영하기 위한 기업용 관리 체계입니다.

|기능|내용|왜 필요한가|
|---|---|---|
|CF 중앙 관리|Skills 버전 관리, 전사 배포|규제 변경 시 전사 동시 반영|
|RBAC|역할별 에이전트 접근 제어|부서별 권한 분리|
|SSO 연동|사내 계정 체계 통합|별도 계정 관리 불필요|
|감사 로그|전체 사용 이력 추적|금융·공공 컴플라이언스 대응|
|비용 관리|LLM API 사용량·비용 모니터링|부서별 비용 배분|
|On-premise 배포|사내 서버 설치|경영 데이터 외부 유출 차단|
**기반**: OpenClaw 오픈소스 위에 위 기능을 에스코어가 자체 구축 **수익 형태**: 플랫폼 라이선스 또는 구축 비용에 포함

### Tier 3. 버티컬 에이전트
현재 보유 및 개발 예정인 에이전트입니다.
#### 현재 보유: 코딩 에이전트
- **범용성이 가장 높음**: 언어·테스트·리팩토링·보안 CF는 업종 무관하게 재사용
- **가변 부분**: 프레임워크(eGovFrame, Spring...), 고객사 개발 표준만 교체
- **포함 자산**: CF + 생산성 대시보드

#### 다음 개발: 데이터 질의 에이전트 (Text-to-SQL)
**목표**: "이번 분기 매출이 왜 떨어졌는지 분석해줘" 같은 자연어 질문을 비전문가도 데이터에서 직접 답을 얻을 수 있도록

**작동 방식**:
```
자연어 질문
    ↓
SQL 변환 (LLM + DB 스키마 CF)
    ↓
쿼리 실행 (DB MCP)
    ↓
결과 해석·시각화
    ↓
자연어 답변 + 근거 데이터
```

**CF 구성**:
```
Rules
├── "수치는 반드시 출처 테이블·컬럼과 함께 제시"
├── "SQL 실행 전 사용자에게 쿼리 확인"  ← 오답 책임 구조
└── "조회만 허용, DML(INSERT/UPDATE/DELETE) 차단"

Skills
├── "이상치 발견 시 원인 데이터까지 드릴다운 제안"
├── "기간 비교 질문에는 전기 대비 % 변화 자동 포함"
└── "데이터 불충분 시 추가 질문 요청"

Knowledge
├── 고객사 DB 스키마 문서 (RAG)
├── KPI 정의 표준 (매출, 영업이익 계산 기준)
└── 산업별 벤치마크 데이터

Examples (few-shot)
├── 좋은 SQL 변환 예시 (모호한 질문 → 명확한 쿼리)
└── 나쁜 예시 (잘못된 조인, 잘못된 집계)
```

**MCP 커넥터 (신규 개발 필요)**:
```
├── ERP DB 연동 (더존, SAP, Oracle)      ← 경영정보 핵심
├── BI 도구 연동 (Tableau, Power BI)
├── 결과 시각화 출력 (차트, Excel)
└── DW/데이터마트 연동
```

**경쟁력**: 데이터 질의는 DB 구조를 알아야 하므로, 스키마 Knowledge를 CF로 구조화하는 것 자체가 진입 장벽입니다. 글로벌 솔루션은 고객사 내부 DB 스키마를 모릅니다.

**주의사항**:
- 경영 데이터는 가장 민감한 데이터 → On-premise 배포가 선결 조건
- "에이전트가 틀린 분석을 하면?" → SQL 확인 단계를 UI에 명시하는 구조 필수

## 2. 실행 과제
전체 과제를 세 트랙으로 나눕니다. 병렬 진행이 가능하지만 의존 관계가 있습니다.
```
Track A: 플랫폼        Track B: 데이터 질의 에이전트    Track C: 사업
(인프라 선행)          (Track A 일부 선행 필요)          (지금 당장 시작)
```


### Track A. 엔터프라이즈 플랫폼 구축
**목표**: 단품 에이전트를 조직 인프라로 올릴 수 있는 기반 완성

|우선순위|과제|이유|
|---|---|---|
|1순위|On-premise 배포 구조|데이터 질의 에이전트 수주 선결 조건|
|1순위|CF 중앙 저장소 + 배포 파이프라인|재사용성·지속성의 핵심|
|2순위|RBAC + SSO 연동|기업 도입 승인 요건|
|2순위|감사 로그|금융·공공 컴플라이언스|
|3순위|비용 관리 대시보드|운영 단계 필요|
|3순위|사내 전용 웹 UI|OpenClaw WebChat 커스터마이징|
**기반**: OpenClaw 오픈소스 (MIT 라이선스) 
**예상 기간**: AI 활용 개발, 3인 기준 약 2.5개월 MVP 
**MVP 범위**: On-premise 배포 + CF 관리 + RBAC (1·2순위만)


### Track B. 데이터 질의 에이전트 개발
**선결 조건**: Track A의 On-premise 배포 구조 완료

|단계|과제|내용|
|---|---|---|
|1단계|DB MCP 커넥터 개발|PostgreSQL, MySQL, Oracle 우선|
|1단계|Text-to-SQL CF 설계|Rules, Skills, few-shot Examples|
|2단계|고객사 스키마 온보딩 도구|DB 스키마 → RAG Knowledge 자동 구성|
|2단계|SQL 확인 UI|실행 전 쿼리 사용자 확인 단계|
|3단계|ERP MCP 커넥터|더존, SAP 연동|
|3단계|시각화 출력|차트, Excel 다운로드|
**1단계 완료 시점**: PoC 가능한 상태 
**전체 완료 시점**: AI 활용 개발, 3인 기준 약 2개월
**참고**: ERP MCP 커넥터는 이후 HR·재무·운영 에이전트에서 재사용됩니다. 한번 만들면 다음 버티컬 구축 속도가 빨라지는 자산입니다.


### Track C. 사업 과제
플랫폼·에이전트 개발과 무관하게 지금 당장 시작해야 하는 것들입니다.

|과제|내용|시점|
|---|---|---|
|경영정보 수요 구체화|들어온 수요가 어떤 DB·어떤 질문 유형인지 파악|지금|
|PoC 대상 고객사 확정|데이터 접근 허용 의향 있는 고객 선정|지금|
|CF 소유권 계약 조항 준비|납품 CF의 소유권 에스코어 귀속 조건|지금|
|코딩 에이전트 CF 자산화|현재 프로젝트별로 흩어진 CF를 사내 라이브러리로 정리|지금|
|삼성 내부 레퍼런스 기획|첫 번째 데이터 질의 에이전트 내부 PoC 대상 선정|Track B 1단계 완료 후|

## Why Us
심 논리는 하나입니다.

> **고객이 직접 하거나 다른 곳에 맡기면, 에스코어가 이미 가진 것을 처음부터 만들어야 합니다.**

---

### 오퍼 1. 국내 기업 환경에 바로 꽂히는 MCP 커넥터를 이미 가지고 있습니다

글로벌 벤더(AWS, Microsoft)는 SAP, Salesforce MCP는 제공하지만 국내 시스템은 없습니다. 에스코어는 국내 기업 환경에서 실제로 쓰는 시스템들의 MCP 커넥터를 보유하고 있습니다.

```
에스코어 보유 / 개발 중인 MCP 커넥터
├── 더존 ERP
├── 영림원 ERP  
├── 카카오워크 / 네이버웍스
├── eGovFrame (전자정부프레임워크)
└── 금융권 내부 시스템 (납품 경험 기반)
```

고객 입장에서: 다른 곳에 맡기면 MCP 커넥터부터 만들어야 합니다. 에스코어는 연결부터 시작합니다.

---

### 오퍼 2. 국내 규제와 도메인 지식이 이미 CF로 구조화되어 있습니다

AI 에이전트가 도메인 전문가처럼 작동하려면 도메인 지식이 CF로 녹아있어야 합니다. 이것을 만드는 데 가장 많은 시간이 걸립니다. 에스코어는 이미 가지고 있습니다.

```
에스코어 보유 CF 자산
├── 코딩: eGovFrame 표준, Java/Spring 규칙, 보안 취약점 패턴
├── 금융: 금융 내부통제 기준, 컴플라이언스 체크리스트 (납품 경험 기반)
└── 개발 중: Text-to-SQL 패턴, KPI 정의 표준
```

고객 입장에서: 도메인 전문가 인터뷰 → CF 설계 → 검증에 수개월이 걸립니다. 에스코어는 불변 레이어를 이미 보유하고 있어 고객사 고유 기준(가변 레이어)만 추가하면 됩니다.

---

### 오퍼 3. 삼성 보안 기준을 통과한 플랫폼입니다

AI 에이전트 도입의 가장 큰 장벽은 보안 승인입니다. 특히 경영 데이터를 다루는 에이전트는 더욱 그렇습니다.

```
에스코어 플랫폼의 보안 검증 근거
├── 삼성 계열사 내부 구축 경험 (가장 높은 보안 요구 수준)
├── On-premise 배포 기본 제공 (데이터 외부 유출 없음)
├── 감사 로그 기본 내장 (금융·공공 컴플라이언스 대응)
└── 금융사 납품 이력 (규제 산업 검증 완료)
```

고객 입장에서: 보안 검토를 처음부터 받으면 수개월이 걸립니다. 에스코어 플랫폼은 이미 검증된 구조입니다.

---

### 오퍼 4. 에이전트를 개인 도구가 아닌 조직 인프라로 운영할 수 있습니다

Claude.ai, ChatGPT 같은 도구는 개인이 씁니다. 조직 전체가 안전하게 쓰려면 별도 인프라가 필요합니다. 에스코어는 이 인프라를 플랫폼으로 제공합니다.

```
개인 도구와의 차이
├── CF 중앙 관리: 규제 변경 시 전사 에이전트 동시 업데이트
├── RBAC: 부서별 에이전트 접근 권한 분리
├── 감사 로그: 누가 무엇을 물었는지 전체 추적
└── 비용 관리: 부서별 LLM 사용량 모니터링
```

고객 입장에서: 이 인프라를 직접 만들면 AI팀 인력과 수개월이 필요합니다.
