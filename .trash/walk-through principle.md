Walk-through의 목적

```
Walk-through의 목적
"에이전트 구성요소 전체를 커버하는 발화 세트를 찾는 것"
```


Walk-through의 최상위 기준
**구성요소를 발동시키는가**

```
범용 발화 분류 기준
├── 외부 시스템 접근이 필요한가?    MCP 커버
├── LLM 판단이 필요한가?            Skill 커버
├── Hook이 발동되는가?              Hook 커버
├── 서브에이전트 위임이 필요한가?   서브에이전트 커버
└── 플러그인 간 오케스트레이션?     연결 구조 커버
```

이 기준으로 다시 잡으면 어떤 에이전트에도 적용됩니다.

```
경영정보 에이전트 발화 목록 재정의

MCP 커버
└── "3월 사업부별 매출 보여줘"       단일 DB
└── "매출이랑 생산량 같이 보여줘"    복수 DB (MCP 라우팅)

Skill 커버
└── "저번달 숫자 뽑아줘"             text-to-sql + 명확화
└── "C사업부 왜 이렇게 나왔어"       anomaly-detect + drill-down

Hook 커버
└── (위 발화에 이미 포함됨)          DML 차단·쿼리 확인은 모든 DB 접근 시 발동

서브에이전트 커버
└── "분석 결과 정리해줘"             insight-writer + reporter

플러그인 간 오케스트레이션
└── "분석해서 팀장님께 보내줘"       경영정보 → 그룹웨어 플러그인

예외/실패 경로
└── "주가 데이터 보여줘"             없는 데이터 처리
```

---

## 3. 의도 명확도는 어떻게 처리하나

범용 축에서 빠졌지만 경영정보 에이전트에서는 여전히 중요합니다. 별도 축으로 두는 대신 **Skill 커버 발화 안에 포함**시키면 됩니다.

```
Skill 커버
├── "C사업부 3월 매출 보여줘"   → 명확, SQL 바로 생성
└── "저번달 숫자 뽑아줘"        → 모호, 명확화 질문 필요
```

두 가지 모두 text-to-sql Skill을 커버하지만, 모호한 쪽이 Skill의 명확화 분기를 검증합니다. 별도 축이 아니라 **같은 Skill의 두 가지 입력 유형**으로 처리하면 됩니다.

## 정리

```
범용 기준 (어떤 에이전트에도 적용)
→ 구성요소를 발동시키는가
  (MCP / Skill / Hook / 서브에이전트 / 플러그인 연결 / 실패 경로)

에이전트 특화 기준 (필요할 때 추가)
→ 해당 에이전트의 핵심 판단 분기를 커버하는가
  경영정보: 모호한 발화 처리
  코딩: 파일 범위 판단
  법무: 위험도 판단
```
