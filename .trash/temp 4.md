# 품질 포탈 소개서 개선 작업 지시서

리더는 여러 시스템에 흩어져 있는 데이터를 끊임없이 뒤져 개선이 필요한 부분을 파악하고, 해결 방안을 분석한 후 실행하는 데 시간을 허비해야 합니다. 그 사이 팀원들은 어려운 대화를 피하기 위해 상황 보고서를 조작하고, 주관적인 정보로 가득 찬 상황을 만들어 실제 상황을 제대로 파악하지 못하다가 결국 위험이 폭발하여 위기로 번지게 됩니다.
대시보드는 권장 사항에 대한 근거를 제공하고 개입의 효과를 측정하는 확인 도구로서 여전히 가치가 있습니다. 하지만 대시보드는 최종 목적지이지 시작점이 되어서는 안 됩니다. 조사는 대시보드를 보기 전에 완료되어야 합니다.
인텔리전스 엔진은 실제 개발 활동을 필요에 따라 분석하고 구체적이고 즉시 실행 가능한 권장 사항과 함께 올바른 문제를 파악함으로써 **조사-분석-실행이라는** 함정을 제거합니다. 이를 통해 리더는 주관적인 상태 업데이트가 아닌 객관적인 증거를 바탕으로 전략적 우선순위에 집중할 수 있습니다.

## 전체 구조 변경

### 현재 (9페이지)

Page0~8

### 변경 후 (7페이지)

```
Page0: 선언
Page1: How It Works
Page2: 연결된 데이터 (통합 뷰)
Page3: 관계를 읽는다 (AI 해석) ← 현재 Page3+Page4 통합 및 재설계
Page4: 더 깊이 파고싶을 때 (대화) ← 현재 Page6
Page5: 자동으로 받아보려면 (리포트) ← 현재 Page7
Page6: 연동 ← 현재 Page8
```

**삭제**: 현재 Page5 (신호 슬라이드) — 완전 제거 **통합**: 현재 Page3 + Page4 → 새 Page3 하나로

`PresentationUI.tsx`의 `TOTAL_PAGES`를 9→7로, `NAV_SECTIONS`도 아래 기준으로 수정.

## 슬라이드별 콘텐츠 지시

---

### Page0: 선언

**역할**: 첫인상. 제품 방향성 선언. 설명이 아니라 각인.

**핵심 메시지 (큰 타이포그래피로)**:

```
대시보드는 녹색입니다.
현실은 왜 적색인가?
```

**보조 텍스트**:

```
개발 데이터는 여러 도구에 있습니다. 각각 보면 이상이 없어 보입니다.
품질 포탈은 흩어진 데이터를 연결하고 AI로 해석해
실제 무슨 일이 일어나고 있는지 보여줍니다.
```

**주의**: 기능 목록, 도구 이름, 순서형 카드 없음. 텍스트 중심 슬라이드.

---

### Page1: How It Works

**역할**: 이후 슬라이드를 이해하는 프레임. Page2~5의 예고편.

**3단계 구조**:

**① 수집·연동**

- 설명: 개발 활동이 발생하는 도구들을 파이프라인으로 연결합니다. 각 도구의 데이터가 하나의 흐름으로 들어옵니다.
- 원천: 이슈·스프린트 / 커밋·PR·리뷰 / 취약점·품질 지표 / 빌드·배포 결과

**② AI 해석**

- 설명: 데이터가 연결되어 있어 도메인 간 교차 분석이 가능합니다. 개별 도구에서는 보이지 않는 패턴과 인과를 읽어냅니다.
- 강조: "단순 집계가 아닙니다. 관계를 읽습니다."

**③ 전달**

- 대시보드에서 조망
- 자연어로 질의
- 자동 리포트로 수신

**주의**:

- "정규화 엔진", "Rule-based", "신호·알림" 표현 없음
- ③의 세 가지가 이후 Page2, Page4, Page5와 1:1 대응 — 독자가 덱의 지도를 갖게 됨

---

### Page2: 연결된 데이터 (통합 뷰)

**역할**: System of Record. "연동하면 이렇게 보입니다." AI 해석 없음. 의도적으로 "이건 기존 도구와 비슷해 보인다"는 인상을 만드는 슬라이드. 다음 슬라이드에서 뒤집기 위한 설계.

**담을 내용**: 5개 영역 통합 대시보드 목업

```
이슈·요구사항 (Jira)
  WBS 진척률, 스프린트 완료율 추이, 납기 예측, 장기 체류 이슈

코드 활동 (Bitbucket)
  주별 커밋 추이, 기여자 분포, 변경 라인

코드 리뷰 (Bitbucket)
  PR 사이클타임 단계별 분해(코딩→픽업→리뷰→배포), 리뷰 없는 머지 현황

코드 품질 (Sparrow / SonarQube)
  취약점 추이(고위험/중위험), 커버리지 추이, 기술부채 지표

빌드·배포 (CI/CD)
  빌드 성공률, DORA 4대 지표(배포빈도·변경리드타임·변경실패율·MTTR)
```

**주의**:

- AI 해석 블록(violet 박스) 없음
- 수치가 실제 목업처럼 보여야 함 (차트, 숫자 포함)
- 슬라이드 하단 또는 우측에 짧게: "5개 영역 데이터를 한 화면에서 — 하지만 수치가 전부가 아닙니다"

---

### Page3: 관계를 읽는다 (AI 해석)

**역할**: System of Action. 이 슬라이드가 제품의 핵심 차별점. "데이터가 연결되어 있어야 보이는 것들"을 보여준다.

**상단 메시지**:

```
수치는 있었습니다. 연결이 없었을 뿐입니다.
```

**AI 해석 사례 3개** (각각 어떤 데이터를 교차했는지 명시):

**사례 1: 속도 저하의 진짜 원인**

- 교차: Jira 이슈 상태 × Bitbucket 커밋 활동
- 발견: 완료 처리된 이슈에 연결된 커밋이 없다 / 특정 이슈 장기 체류 구간에 팀 전체 커밋이 감소한다
- 의미: 단순 속도 저하가 아니라 특정 이슈가 팀 전체 흐름을 막고 있는 패턴
- Jira만 봐서는 완료율만 보인다. Bitbucket과 교차해야 실제 작업 흐름이 보인다.

**사례 2: 코드 품질 저하의 근본 원인**

- 교차: Bitbucket PR 리뷰 패턴 × Sparrow 취약점 발생 시점
- 발견: 리뷰 없는 머지가 집중된 구간과 취약점 증가 구간이 일치한다
- 의미: 코드 품질 저하의 원인이 코딩 실수가 아니라 프로세스 위반
- Sparrow만 봐서는 취약점 수만 안다. Bitbucket과 교차해야 왜 생겼는지 보인다.

**사례 3: 빌드 불안정의 원인 추적**

- 교차: Bitbucket 특정 모듈 변경 이력 × CI/CD 빌드 결과
- 발견: 리뷰 없는 머지 시점과 빌드 실패 급증이 반복적으로 겹친다
- 의미: 배포 불안정이 코드 변경 프로세스와 연결된 구조적 패턴
- 빌드 실패만 보면 단발성 이슈로 보인다. 교차하면 반복 패턴이 보인다.

**각 사례 표현 방식**:

```
[데이터 A] + [데이터 B] → 발견 → 의미
```

CrossTag 컴포넌트(기존 것 재사용)로 소스 명시.

**주의**:

- 현재 Page3의 차트+테이블 대시보드 구조 사용 안 함
- 차트가 필요하다면 사례를 뒷받침하는 보조 수단으로만
- 메인은 AI 해석 텍스트와 교차 구조

---

### Page4: 더 깊이 파고싶을 때 (대화)

**역할**: 대시보드로 상황을 봤다. 더 알고 싶다. 자연어로 묻는다.

**상단 메시지**:

```
대시보드가 답하지 못하는 질문들
```

**대화 시나리오 3개** (수행사/외주 컨텍스트 없음):

**시나리오 1: 원인 탐색**

- 질의: "이번 스프린트 속도가 왜 이렇게 떨어졌어?"
- 답변: Jira + Bitbucket 교차 분석 → 병목 이슈와 실제 커밋 활동 연결해서 원인 설명
- 소스 뱃지: Jira × Bitbucket

**시나리오 2: 릴리스 판단**

- 질의: "다음 주 릴리스 가능한 상태야?"
- 답변: Sparrow(취약점) + CI/CD(빌드 안정성) + Jira(미완료 이슈) 종합 → 릴리스 가능 여부와 조건 제시
- 소스 뱃지: Sparrow × CI/CD × Jira

**시나리오 3: 스프린트 리뷰 준비**

- 질의: "이번 스프린트 결과 정리해줘"
- 답변: 완료 이슈 집계 + 코드 변경 범위 + 품질 지표 변화 통합 → 데이터 기반 스프린트 리뷰 자료 초안
- 소스 뱃지: Jira × Bitbucket × Sparrow
- 부연: 수치 검증(보고 vs 실측)이 자연스럽게 포함됨

**주의**:

- "수행사", "보고서 검증", "수행사 확인 요청" 표현 없음
- 답변 각 항목 옆에 데이터 소스 뱃지 필수 (어떤 데이터 교차했는지 보여야 함)
- 기존 Page6 채팅 UI 디자인 재사용 가능, 시나리오만 교체

---

### Page5: 자동으로 받아보려면 (리포트)

**역할**: 포탈에 들어오지 않아도 현황을 받아볼 수 있다.

**상단 메시지**:

```
매번 데이터를 취합하지 않아도 됩니다
```

**트리거 (간결하게, 좌측 절반 이하)**:

```
스프린트 종료 시     → 이번 스프린트 결과와 다음 스프린트 리스크 요약
납기 D-7, D-3      → 현재 진척 기준 납기 달성 가능성과 주요 리스크
배포 전             → 릴리스 준비 상태 점검
```

**리포트 예시 (우측, 메인)**:

```
헤더: ERP 고도화 · 납기 사전 점검 리포트
      2025.01.14 17:00 · 납기 D-7 자동 생성

종합 현황
  코드 품질: 고위험 취약점 3건 미해소 — 릴리스 조건 미충족
  진척: 현재 속도 기준 D+8 초과 예측

AI 해석
  리뷰 없는 머지 구간과 취약점 증가 구간이 일치.
  프로세스 정상화가 품질 개선의 핵심 레버입니다.

권고 사항
  1. 고위험 취약점 3건 — 즉시 해소 일정 수립
  2. DEV-112 — 진행 현황 확인
  3. PR 리뷰 프로세스 — 리뷰어 지정 강제 적용
```

**주의**:

- "수행사", "수행사 확인 요청" 없음
- 트리거 설명을 절반 이상 차지하지 않도록
- 기존 Page7 레이아웃 재사용 가능, 내용만 교체

---

### Page6: 연동

**역할**: 어떤 도구가 연결되는가. 도입 장벽을 낮추는 슬라이드.

**상단 메시지**:

```
앞서 본 기능들은 모두 이 연결 위에서 작동합니다
```

**5개 영역 × 도구 목록**:

```
이슈·요구사항   Jira, GitHub Issues, GitLab Issues
코드 활동       Bitbucket, GitHub, GitLab
코드 리뷰       Bitbucket PR, GitHub PR, GitLab MR
코드 품질       Sparrow, SonarQube
빌드·배포       Jenkins, GitHub Actions, Bamboo, GitLab CI
```

**추가 메시지**:

- 목록에 없는 도구도 연동 협의 가능
- 보고 싶은 지표와 뷰 맞춤 구성 가능

**주의**:

- "도메인 모델", "정규화", "Cross-domain" 표현 없음
- "고객 환경 ✓" 뱃지 일부 도구에만 붙이지 말 것 — 오해 유발
- 기존 Page8 레이아웃 재사용 가능, 내용 일부 수정

## 네비게이션 수정

`PresentationUI.tsx`의 TOTAL_PAGES를 9→7로 변경. NAV_SECTIONS는 소개 / 작동 방식 / 통합 뷰 / AI 해석 / 대화 / 리포트 / 연동 — 7개로 재구성. `App.tsx`의 페이지 라우팅도 0~6에 맞춰 정리.

## 금지 표현 목록

아래 표현이 덱 어디에도 없어야 한다:

- 수행사, 발주사
- 산출물 검수, 게이트 리뷰, 게이트 조건
- Rule-based, 정규화 엔진, 도메인 모델
- 신호·알림 (별도 기능으로)
- 고객 환경 ✓ (특정 도구에만 붙이는 뱃지)
- "수행사 확인 요청", "수행사 보고서"
