# 품질 포탈 — 가치 제안 구조 (수정본)

---

## 배경

### 문제 서술

1. **체계 없이 진행되던 개발 프로젝트** SK케미칼의 개발 프로젝트는 수행사가 담당하고 PM이 관리하는 구조. 이전까지는 품질 기준도, 산출물 기준도, 평가 방식도 명확하지 않아 일관된 관리가 어려운 상황이었다.
    
2. **1차: SDLC 표준 수립** 단계별 체크리스트, 산출물 패키지, 품질 임계치를 정의했다. 착수 → 설계 → 개발 → 테스트/배포 → 운영 각 단계에 게이트가 생겼다.
    
3. **표준 운영의 부담** PM 1명이 여러 프로젝트를 병행하는 구조에서, 체크리스트 확인·산출물 검수·도구별 현황 파악을 수동으로 챙기는 것은 상당한 부담이다. 표준이 실제로 운영되려면 이를 지원하는 체계가 함께 필요하다.
    

> **2차 과제 목표:** 1차에서 정의한 표준을 시스템 안에 내재화한다. PM의 부담을 줄이면서 표준이 실제로 작동할 수 있게 한다.

---

## 제안 요약

**품질 포탈** 개발 프로젝트에서 발생하는 데이터는 여러 도구에 분산되어 있다. 각 도구는 데이터를 가지고 있지만, SK케미칼의 기준으로 해석해주지는 않는다. 품질 포탈은 그 사이를 채운다.

```
[ 개발 도구들 ]          [ 공백 ]            [ PM ]
Jira, Bitbucket,   →   기준 없는 데이터   →   판단 불가
Sparrow, 기타...

[ 개발 도구들 ]          [ 품질 포탈 ]         [ PM ]
Jira, Bitbucket,   →   SK케미칼 표준으로  →   현황 파악
Sparrow, 기타...        해석된 지표            의사결정
```

---

### PULL — 대시보드

PM이 필요할 때 — 게이트 리뷰, 주간 보고, 이슈 판단 시 — 도구를 오가지 않고 한 화면에서 전체 현황을 확인한다. 숫자를 보여주는 것을 넘어, SK케미칼 표준 기준으로 현재 상태가 어떤 의미인지까지 해석해서 제공한다.

포함 영역:

- 산출물 품질
- 코드 품질
- 프로세스 품질
- 일정 품질

---

### PUSH — 에이전트

시스템이 먼저 — 임계치 초과, 산출물 미제출, 납기 임박 등 특정 이벤트가 발생하면 포탈이 자동으로 감지하고 PM에게 전달한다. 대시보드를 열지 않아도 중요한 상황을 놓치지 않는다.

포함 기능:

- 이벤트 트리거 (임계치 초과, 표준 위반 등)
- 일정 트리거 (게이트 D-7, 납품 D-7 등)
- 산출물 AI 검수 결과 전달

---

## 통합: 데이터에 맥락을 더한다

각 개발 도구는 이미 데이터를 가지고 있다. Jira에는 이슈 진척이, Bitbucket에는 코드 리뷰 기록이, Sparrow에는 취약점 분석 결과가 있다. 그러나 이 데이터들은 서로 연결되지 않고, SK케미칼의 기준으로 해석되지 않는다.

품질 포탈은 이 데이터들을 한 곳에 모아 SK케미칼 표준 임계치와 대조한다. 도구를 추가하거나 교체하더라도 기준은 포탈 안에 있기 때문에, 같은 방식으로 해석된다.

> 도구가 늘어날수록 관리 부담도 늘어나는 구조에서 → 도구가 늘어도 PM이 보는 화면은 하나인 구조로.

수집 대상은 현재 확인된 도구(Jira, Bitbucket, Sparrow, Confluence 등)를 포함하며, 프로젝트 환경에 따라 CI/CD, OSS 모니터링 등으로 확장 가능하다.

## 대시보드: 현황, 판단, 액션

대시보드는 단순히 수치를 모아 보여주는 화면이 아니다. 수집된 데이터를 세 가지 레이어로 가공해서 보여준다.

- **현황**: 지금 프로젝트가 어디까지 왔는가 (진척률, 제출 상태, 지표 추이)
- **판단**: 그게 기준 대비 괜찮은 상태인가 (임계치 대비 pass/위험/주의)
- **액션**: 뭘 해야 하는가 (이슈 목록, 조치 필요 항목)

각 품질 영역은 이 세 레이어로 구성되며, 통합 뷰에서는 4개 영역의 현재 상태를 한눈에 확인하고 AI 요약을 통해 전체 맥락을 파악할 수 있다.

## AI 보조: 문서를 읽는다

숫자로 수집·비교 가능한 지표와 달리, 산출물 내용이 기준을 충족하는지는 사람이 직접 읽어야만 판단할 수 있었다. AI가 이 역할을 보조한다.

**판단 기준은 두 가지다.**

첫째, **단계별 체크리스트** — 1차에서 정의한 각 단계(설계, 개발 등)의 체크리스트에는 해당 단계에서 제출되어야 할 산출물 목록과 확인 항목이 포함되어 있다. 시스템은 이를 기준으로 제출 여부와 체크리스트 이행 상태를 자동으로 확인한다.

둘째, **산출물 템플릿** — 1차에서 산출물 유형별 표준 템플릿이 정의되어 있다. AI는 제출된 문서가 해당 템플릿의 구성 항목을 갖추고 있는지를 대조하고, 항목별 충족 여부와 판단 근거를 제공한다.

PM은 AI가 만들어놓은 초안 판단을 검토하고 컨펌 또는 반려한다.

---

### 처리 흐름

1. **제출 감지** 수행사가 Confluence에 산출물 업로드 → 비기능 요구사항 정의서 — 방금 업로드됨
    
2. **단계 체크리스트 대조** 현재 단계(설계)의 체크리스트 기준으로 제출 여부 및 항목 이행 확인
    
3. **템플릿 기준 내용 검수** 비기능 요구사항 정의서 표준 템플릿 항목과 문서 내용 자동 대조 → 7개 항목 검토 중…
    
4. **결과 생성** 항목별 충족/미충족/불명확 판단 + 근거 문장 첨부 → 충족 4 · 미충족 2 · 불명확 1
    
5. **PM 컨펌** AI 판단 초안을 PM이 검토 후 승인 또는 반려 → 2분 소요 (직접 읽을 경우 평균 25분)

## 능동적 전달: 먼저 알린다

대시보드는 PM이 능동적으로 열어야 한다는 한계가 있다. 에이전트는 이 한계를 보완한다. 이벤트가 발생하면 시스템이 먼저 감지하고 PM에게 전달한다. PM의 역할은 순찰이 아니라 예외 처리가 된다.
