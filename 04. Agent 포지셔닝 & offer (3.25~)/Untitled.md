## 전체 틀

---

### 1단계 — 시장 인식 교정

**역할:** 고객의 "남들 다 하는 에이전트잖아요"라는 인식을 해체한다. 에이전트 도입이 실패하는 게 예외가 아니라 기본값임을 보여준다.

```
[현실 진단]
에이전트 도입은 이미 대세다.
그런데 실제로 작동하는 경우는 극소수다.

[증거]
실패율 데이터

[원인 분석]
왜 실패하는가?
→ 모델이 나빠서가 아니다
→ 설계가 없어서다

[범용 에이전트의 현실]
가장 많이 팔린 에이전트도 실제로는 안 쓰인다
```

---

### 2단계 — 격차 정의

**역할:** "에이전트는 다 똑같다"는 인식을 깬다. SOTA와 범용의 격차가 실재하며, 그 격차가 업무 결과에서 어떻게 드러나는지 보여준다.

```
[능력의 진화]
지난 2년간 에이전트 능력은 폭발적으로 향상됐다

[격차의 실체]
SOTA 에이전트가 실제 엔터프라이즈에서 낸 결과
(Harvey, Devin, Abridge 사례)

[격차의 성격]
속도 차이가 아니라 가능성의 차이다
이전에 불가능했던 것이 가능해진다

[반전]
그런데 SOTA도 그냥 가져다 쓰면 안 된다
→ 설계 없이는 효과가 나지 않는다

[빈 곳 A]
역할: "설계 없이는 SOTA도 안 된다"는 주장의
Escor 맥락 근거
→ 지금은 외부 데이터로만 뒷받침됨
→ 향후 Escor 직접 사례로 교체할 자리
→ 임시: Claude Code 헤비/라이트유저 격차,
   Devin 플레이북 유무 차이
```

---

### 3단계 — 진입 장벽

**역할:** "그럼 우리도 SOTA 쓰면 되는 거 아닌가요?"라는 질문에 답한다. 왜 아무나 할 수 없는지를 설명하고, 자연스럽게 Escor로 연결되는 조건을 만든다.

```
[세 가지 조건]
SOTA 에이전트가 실제로 작동하려면
세 가지가 동시에 필요하다:

  도메인 지식
  — 어떤 업무 문제를 에이전트로 풀어야 하는지 안다

  설계 역량
  — 에이전트가 어떻게 작동해야 하는지 구조화할 수 있다

  기술 실행력
  — 그것을 실제로 고객 환경에 배포할 수 있다

[시장의 현실]
빌더 플랫폼 → 세 번째만 준다
범용 에이전트 → 세 가지 모두 없다
일반 컨설팅 → 첫 번째만 있다

[빈 곳 B]
역할: 3단계에서 4단계로 넘어가는 연결 논리
"세 가지가 필요하다"에서
"Escor가 세 가지를 갖췄다"로 이어지는 문장
→ 지금은 이 연결이 없어서 논리가 끊긴다
→ 채울 내용:
   도메인 지식 ← 컨설팅 이력 (어떤 업무 영역인지 구체성 필요)
   설계 역량   ← S-Core Context Framework
   기술 실행력 ← 개발 조직
```

---

### 4단계 — Escor 포지셔닝

**역할:** Escor가 무엇을 하는 회사인지, 다른 회사와 무엇이 다른지를 고객이 이해할 수 있는 언어로 설명한다.

```
[우리가 하는 것]
컨설팅으로 문제 정의
→ S-Core Framework로 설계
→ 고객 환경에 배포

[빈 곳 C]
역할: S-Core Context Framework를
고객 언어로 설명하는 한 단락
→ 기술 언어(커맨드, 훅, MCP)가 아니라
   고객이 체감하는 언어로
→ 채울 방향 선택 필요:
   A) 결과 중심: "어떤 상황에서 어떤 에이전트가
      어떻게 움직일지를 설계하는 방법론"
   B) 비유 중심: "설계도면 — 에이전트 하나가 아니라
      그 에이전트가 살아있는 구조 전체를 납품"

[고객이 받는 것]
에이전트 하나가 아니라
업무에 맞게 설계된 에이전트 시스템

[빈 곳 D]
역할: 구조째 납품이 고객에게 왜 좋은지
"So what?" 에 답하는 단락
→ 채울 내용 후보:
   블랙박스가 아니다 (들여다볼 수 있다)
   고장나도 고칠 수 있다 (어디서 문제인지 찾을 수 있다)
   나중에 확장할 수 있다 (부분만 바꾸면 된다)
→ 실제로 Escor가 제공하는 것과 맞는지 확인 필요
```

---

### 5단계 — 효과

**역할:** 고객이 실제로 경험하게 되는 변화를 보여준다. 단순 효율이 아니라 가능성과 역할의 변화로 표현한다.

```
[효과의 성격]
시간 절약이 아니다
이전에 불가능했던 것이 가능해진다
판단의 질이 바뀐다
사람의 역할이 바뀐다

[외부 사례]
Harvey / Devin / Abridge
(설계 원리가 같다는 것을 보여주는 방식으로 연결)

[빈 곳 E]
역할: Escor 맥락의 효과 표현
→ 지금 당장 채울 수 없는 자리
→ 프로젝트 레퍼런스나 시연 결과가 나와야 채워진다
→ 임시 전략 선택 필요:
   A) 외부 사례를 S-Core 설계 논리와 연결
      "이런 구조로 설계했기 때문에 이런 결과가 가능했다"
   B) Palantir 부트캠프 방식
      효과를 주장하는 대신
      고객 문제 하나를 같이 풀어보는 세션 제안
```

---

### 6단계 — 다음 단계

**역할:** 고객이 "그래서 어떻게 시작하나요?"라고 물었을 때의 답. 전체 도입이 아니라 작게 시작할 수 있다는 것을 보여준다.

```
[진입 방식]
진단 → 설계 → 배포 → 확장

[빈 곳 F]
역할: Escor의 실제 영업 프로세스와 이 흐름이
일치하는지 확인하는 자리
→ "빠른 가치 증명" 방식이 있는가
   (Palantir 부트캠프처럼)
→ 첫 번째 접점이 진단인가, 제안서인가
→ 이 흐름이 실제로 가능한지 내부 확인 필요
```
